# 2026-03-15 - Agent Route Company Scope Hardening

## Repo Context Inspected

- product and workflows: Paperclip remains a company-scoped control plane, not a general chat app; work stays attached to issues, comments, approvals, goals, and agent runs
- actors, auth, and permissions: `server/src/middleware/auth.ts` resolves implicit local board, authenticated board sessions, agent API keys, and local agent JWTs; `server/src/middleware/board-mutation-guard.ts` protects authenticated board mutations by trusted origin
- APIs, routes, and orchestration: `server/src/app.ts` mounts `/api` route groups, while `server/src/routes/agents.ts` owns most board-managed agent lifecycle and key workflows plus mutation activity logging
- types, validators, and schema: agent API key records stay company-scoped in `packages/db/src/schema/agent_api_keys.ts`; shared activity rendering is string-action based, with no contract change required for a new `agent.key_revoked` event
- UI workflows and persistence: issue document autosave, comment drafts, and activity rendering were inspected to find a second candidate, but the stronger bug was a company-scope gap in board-managed agent routes

## Safe Action

- goal: close a real cross-company authorization gap for authenticated board users on agent lifecycle and API-key routes
- shipped change: board-managed agent pause, resume, terminate, delete, key list, key create, and key revoke routes now resolve the target agent first and enforce `assertCompanyAccess(req, agent.companyId)` before performing the action
- why it was safe: behavior is unchanged for local-trusted and properly scoped board users, while authenticated session users now correctly stop at the company boundary already required by the V1 contract

## Moonshot

- goal: make future board-managed agent mutations harder to mis-scope and improve auditability of key governance
- shipped change or foundation slice: introduced a shared `loadBoardManagedAgent` guard path in `server/src/routes/agents.ts` and added activity logging for API-key revocation, plus a UI verb for the new `agent.key_revoked` event
- long-term payoff: this reduces drift across agent-management routes and keeps sensitive credential lifecycle mutations visible in the activity stream
- why this still stayed in scope: it strengthens existing control-plane governance without adding dependencies, changing schemas, or expanding the product surface
