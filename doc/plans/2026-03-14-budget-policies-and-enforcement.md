# 预算策略与执行

## 背景

Paperclip 已经将预算视为核心控制面的职责：

- `doc/SPEC.md` 赋予看板设置预算、暂停智能体、暂停工作和覆盖任何预算的权限。
- `doc/SPEC-implementation.md` 规定 V1 必须支持 UTC 月度预算窗口、软告警和硬自动暂停。
- 当前代码仅部分实现了该意图。

目前系统具有狭窄的资金预算行为：

- 公司跟踪 `budgetMonthlyCents` 和 `spentMonthlyCents`
- 智能体跟踪 `budgetMonthlyCents` 和 `spentMonthlyCents`
- `cost_events` 摄入时递增这些计数器
- 当智能体超出其月度预算时，智能体被暂停

这留下了重大产品缺口：

- 没有项目预算模型
- 预算达到时不会生成审批
- 没有通用预算策略系统
- 没有与预算关联的项目暂停语义
- 没有持久的事件跟踪来防止重复告警
- 没有区分可执行的支出预算和建议性用量配额

本计划定义了 Paperclip 下一步应实现的具体预算模型。

## 产品目标

Paperclip 应让运营人员能够：

1. 在智能体和项目上设置预算。
2. 了解预算是基于资金还是用量。
3. 在预算耗尽前收到警告。
4. 当硬预算触发时自动暂停工作。
5. 通过明显的 UI 批准、提高或恢复预算停止。
6. 在仪表盘、`/costs` 和范围详情页面查看预算状态。

系统应非常清楚地表明：

- 预算是策略控制
- 配额是用量可见性

它们相关，但不是同一概念。

## 产品决策

### V1 预算默认值

对于下一轮实现，Paperclip 应执行以下默认值：

- 智能体预算是循环月度预算
- 项目预算是生命周期总额预算
- 硬停止执行使用计费美元而非 token
- 月度窗口使用 UTC 日历月
- 项目总额预算不会自动重置

这提供了一个清晰的心智模型：

- 智能体是持续运行的工作者，所以月度循环预算是自然的
- 项目是有限的工作流，所以生命周期上限是自然的

### 首先执行的指标

首个可执行指标应为 `billed_cents`。

原因：

- 跨提供商、计费方和模型都能工作
- 直接映射到真实的财务风险
- 一致地处理超额和计量用量
- 避免跨提供商的 token 规范化问题
- 即使未来的财务事件不是基于 token 的也能干净地应用

Token 预算不应是首个硬停止策略。
它们应在资金预算系统稳固后，作为建议性用量控制加入。

### 订阅用量决策

Paperclip 应将订阅包含用量与计费支出分开：

- `subscription_included`
  - 在报表中可见
  - 在用量摘要中可见
  - 不计入资金预算
- `subscription_overage`
  - 在报表中可见
  - 计入资金预算
- `metered_api`
  - 在报表中可见
  - 计入资金预算

这使预算系统保持诚实：

- 用户不应看到"支出"因未产生边际计费成本的用量而上升
- 用户仍应看到 token 用量和提供商配额状态

### 软告警与硬停止

Paperclip 应有两类阈值：

- 软告警
  - 创建可见的通知状态
  - 不创建审批
  - 不暂停工作
- 硬停止
  - 自动暂停受影响的范围
  - 创建需要人工解决的审批
  - 阻止该范围内额外的心跳或任务拾取

默认阈值：

- 软告警在 `80%`
- 硬停止在 `100%`

这些后续应可按策略配置，但目前是好的默认值。

## 范围模型

### 支持的范围类型

预算策略应支持：

- `company`
- `agent`
- `project`

本计划重点是先完成 `agent` 和 `project`，同时保留现有的公司预算行为。

### 推荐的 V1.5 策略预设

- 公司
  - 指标：`billed_cents`
  - 窗口：`calendar_month_utc`
- 智能体
  - 指标：`billed_cents`
  - 窗口：`calendar_month_utc`
- 项目
  - 指标：`billed_cents`
  - 窗口：`lifetime`

未来扩展可以添加：

- token 建议性策略
- 每日或每周支出窗口
- 按提供商或计费方范围的预算
- 沿组织树向下继承的委托预算

## 当前实现基线

当前代码库并非从零开始，但现有形态太临时化，无法安全扩展。

### 今天已有的

- 公司和智能体月度美分计数器
- 更新这些计数器的成本摄入
- 智能体月度预算超支时的硬停止暂停

### 缺失的

- 项目预算
- 通用预算策略持久化
- 通用阈值越界检测
- 按范围/窗口的事件去重
- 硬停止时创建审批
- 项目执行阻断
- 预算时间线和事件 UI
- 区分建议性配额和可执行预算

## 建议的数据模型

### 1. `budget_policies`

为规范预算定义创建新表。

建议字段：

- `id`
- `company_id`
- `scope_type`
- `scope_id`
- `metric`
- `window_kind`
- `amount`
- `warn_percent`
- `hard_stop_enabled`
- `notify_enabled`
- `is_active`
- `created_by_user_id`
- `updated_by_user_id`
- `created_at`
- `updated_at`

说明：

- `scope_type` 是 `company | agent | project` 之一
- `scope_id` 仅当公司级策略隐含公司时可为空；否则保持显式
- `metric` 应从 `billed_cents` 开始
- `window_kind` 从 `calendar_month_utc | lifetime` 开始
- `amount` 以指标的自然单位存储

### 2. `budget_incidents`

创建阈值越界的持久记录。

建议字段：

- `id`
- `company_id`
- `policy_id`
- `scope_type`
- `scope_id`
- `metric`
- `window_kind`
- `window_start`
- `window_end`
- `threshold_type`
- `amount_limit`
- `amount_observed`
- `status`
- `approval_id` nullable
- `activity_id` nullable
- `resolved_at` nullable
- `created_at`
- `updated_at`

说明：

- `threshold_type`：`soft | hard`
- `status`：`open | acknowledged | resolved | dismissed`
- 每个策略每个阈值每个窗口一个开放事件，防止重复审批和告警轰炸

### 3. 项目暂停状态

项目需要显式的暂停语义。

推荐方案：

- 扩展项目状态或添加暂停字段，使项目可以被预算阻断
- 保留项目因预算暂停与手动暂停的区分

推荐形态：

- 保持项目工作流状态不变
- 添加执行状态字段：
  - `execution_status`：`active | paused | archived`
  - `pause_reason`：`manual | budget | system | null`

如果对即时实现来说范围过大，较小版本是：

- 添加 `paused_at`
- 添加 `pause_reason`

关键要求是行为性的，而非外观性的：
Paperclip 必须知道项目因预算被暂停，并加以执行。

### 4. 与现有预算列的兼容性

现有的公司和智能体月度预算列应暂时保留以保持兼容。

迁移计划：

1. 在过渡期间继续读取现有列
2. 创建等价的 `budget_policies` 行
3. 将执行和 UI 切换到策略
4. 后续移除或弃用遗留列

## 预算引擎

预算执行应移入专用服务。

当前逻辑埋在成本摄入内部。
这太狭窄了，因为预算检查必须在多个执行边界处应用。

### 职责

新服务：`budgetService`

职责：

- 为成本事件解析适用的策略
- 计算当前窗口总额
- 检测阈值越界
- 创建事件、活动和审批
- 在硬停止时暂停受影响的范围
- 为执行入口点提供预检执行检查

### 规范评估流程

当写入新的 `cost_event` 时：

1. 持久化 `cost_event`
2. 识别受影响的范围
   - 公司
   - 智能体
   - 项目
3. 获取这些范围的活跃策略
4. 计算每个策略窗口的当前观察金额
5. 与阈值比较
6. 如果在窗口内首次越过软阈值，创建软事件
7. 如果在窗口内首次越过硬阈值，创建硬事件
8. 如果是硬事件：
   - 暂停该范围
   - 创建审批
   - 创建活动事件
   - 发出通知状态

### 预检执行检查

预算执行不能仅依赖事后的成本摄入。

Paperclip 还必须在新工作开始前阻断执行。

在以下位置添加预算检查：

- 调度器心跳分发
- 手动调用端点
- 分配驱动的唤醒
- 排队运行的提升
- 任务签出或拾取路径（如适用）

如果范围因预算被暂停：

- 不启动新的心跳
- 不让智能体拾取额外工作
- 在 API 和 UI 中呈现清晰的原因

### 活跃运行行为

当硬停止在运行已经活跃时触发：

- 立即标记范围为暂停以阻止未来工作
- 请求当前运行优雅取消
- 允许正常的取消超时行为
- 写入活动说明暂停来自预算执行

这与产品已预期的通用暂停语义一致。

## 审批模型

预算硬停止应创建一等审批。

### 新审批类型

添加审批类型：

- `budget_override_required`

载荷应包含：

- `scopeType`
- `scopeId`
- `scopeName`
- `metric`
- `windowKind`
- `thresholdType`
- `budgetAmount`
- `observedAmount`
- `windowStart`
- `windowEnd`
- `topDrivers`
- `paused`

### 解决操作

审批 UI 应支持：

- 提高预算并恢复
- 恢复一次但不更改策略
- 保持暂停

可选的后续操作：

- 禁用预算策略

### 软告警不需要审批

软告警应创建：

- 活动事件
- 仪表盘告警
- 收件箱通知或类似的看板可见信号

默认情况下不应创建审批。

## 通知和活动模型

预算事件需要明显的运营人员可见性。

必需的输出：

- 阈值越界时的活动日志条目
- 活跃预算事件的仪表盘展示
- 被暂停智能体或项目详情页面的横幅
- `/costs` 上活跃事件和策略健康状况的摘要

后续渠道：

- 邮件
- webhook
- Slack 或其他集成

## API 计划

### 策略管理

添加以下路由：

- 列出公司的预算策略
- 创建预算策略
- 更新预算策略
- 归档或禁用预算策略

### 事件展示

添加以下路由：

- 列出活跃的预算事件
- 列出事件历史
- 获取范围的事件详情

### 审批解决

预算审批应在添加新审批类型后使用现有的审批系统。

预期流程：

- 硬停止时创建审批
- 通过更改策略并恢复来解决审批
- 通过恢复一次来解决审批

### 执行错误

当工作被预算阻断时，API 应返回明确的错误。

示例：

- 智能体调用被阻断，因为智能体预算已暂停
- 任务执行被阻断，因为项目预算已暂停

不要静默无操作。

## UI 计划

预算应在运营人员做决策的地方可见。

### `/costs`

添加预算部分，包含：

- 活跃的预算事件
- 带范围、窗口、指标和阈值状态的策略列表
- 当前周期或总额的进度条
- 清晰区分：
  - 支出预算
  - 订阅配额
- 快捷操作：
  - 提高预算
  - 打开审批
  - 恢复范围（如果允许）

页面应使以下视觉区分显而易见：

- 预算
  - 可执行的支出策略
- 配额
  - 提供商或订阅用量窗口

### 智能体详情

添加智能体预算卡片：

- 月度预算金额
- 当月支出
- 剩余支出
- 状态
- 警告或暂停横幅
- 如果被阻断则链接到审批

### 项目详情

添加项目预算卡片：

- 总预算金额
- 截至目前的总支出
- 剩余支出
- 暂停状态
- 审批链接

项目详情还应显示任务执行是否因项目预算暂停而被阻断。

### 仪表盘

添加高信号的预算部分：

- 活跃的预算违规
- 即将到来的软告警
- 因预算而暂停的智能体和项目数量

运营人员不应必须访问 `/costs` 才能得知工作已停止。

## 预算计算

### 什么计入预算

对于 V1.5 执行，包含：

- `metered_api` 成本事件
- `subscription_overage` 成本事件
- 任何未来有非零计费美分的请求范围成本事件

不包含：

- 零计费美分的 `subscription_included` 成本事件
- 建议性配额行
- 账户级财务事件，除非且直到明确添加公司级财务预算

### 为什么不先用 Token

Token 预算不应是首个硬停止，因为：

- 提供商计算 token 的方式不同
- 缓存 token 使简单合计变得复杂
- 某些未来费用不是基于 token 的
- 订阅 token 不一定意味着支出
- 资金仍然是最干净的跨提供商执行指标

### 未来的预算指标

未来的策略指标可以包含：

- `total_tokens`
- `input_tokens`
- `output_tokens`
- `requests`
- `finance_amount_cents`

但它们应在资金预算路径稳定后才加入。

## 迁移计划

### 阶段 1：基础

- 添加 `budget_policies`
- 添加 `budget_incidents`
- 添加新的审批类型
- 添加项目暂停元数据

### 阶段 2：兼容性

- 从现有公司和智能体月度预算列回填策略
- 在迁移期间保持遗留列可读

### 阶段 3：执行

- 将预算逻辑移入专用服务
- 添加硬停止事件创建
- 添加活动和审批创建
- 在心跳和调用路径上添加执行守卫

### 阶段 4：UI

- `/costs` 预算部分
- 智能体详情预算卡片
- 项目详情预算卡片
- 仪表盘事件摘要

### 阶段 5：清理

- 将所有读写迁移到 `budget_policies`
- 减少对遗留列的依赖
- 决定是否移除旧预算列

## 测试

必需覆盖：

- 智能体月度预算在 80% 时软告警
- 智能体月度预算在 100% 时硬停止
- 项目生命周期预算软告警
- 项目生命周期预算硬停止
- `subscription_included` 用量不消耗资金预算
- `subscription_overage` 消耗资金预算
- 硬停止每个阈值每个窗口创建一个事件
- 硬停止创建审批并暂停正确的范围
- 被暂停的项目阻断新的任务执行
- 被暂停的智能体阻断新的心跳分发
- 策略更新和恢复正确清除或解决活跃事件
- 仪表盘和 `/costs` 展示活跃事件

## 开放问题

这些应明确延后，除非它们阻断实现：

- 项目预算是否也应支持月度模式，还是首次发布只需要生命周期就够了？
- 公司级预算最终是否应包含 `finance_events`，如 OpenRouter 充值费和 Bedrock 预留费用？
- 委托预算编辑在 V1 中是否应受组织层级限制，还是在 UI 中保持仅看板可用，即使数据模型后续可以支持委托？
- 我们是否需要立即支持"恢复一次"，还是首次审批解决可以是"提高预算并恢复"加"保持暂停"？

## 建议

实现首个连贯的预算系统，遵循以下规则：

- 智能体预算 = 月度计费美元
- 项目预算 = 生命周期计费美元
- 硬停止 = 自动暂停 + 审批
- 软告警 = 可见警告，无审批
- 订阅用量 = 可见配额和 token 报表，非资金预算执行

这解决了真正的运营人员问题，而不会将支出控制、提供商配额窗口和 token 核算混在一起。
