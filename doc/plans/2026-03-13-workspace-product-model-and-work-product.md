# 工作区产品模型、工作成果与 PR 流程

## 背景

Paperclip 需要支持两种非常不同但同样有效的工作方式：

- 单独开发者直接在 `master` 上工作，或在一个甚至不是 git 仓库的文件夹中工作
- 更大的工程工作流，带有隔离的分支、预览、Pull Request 和清理自动化

今天，Paperclip 已经有了这个模型的雏形：

- `projects` 可以携带执行工作区策略
- `project_workspaces` 已作为持久的项目范围对象存在
- 任务可以携带执行工作区设置
- 运行时服务可以附加到工作区或任务

缺失的是一个清晰的产品模型和 UI，使这些能力可理解和可操作。

主要产品风险是一个概念承载过多：

- 让子任务承担分支或 PR 的职责
- 让项目变得过于基础设施化
- 让工作区隐藏到用户无法形成心智模型
- 让 Paperclip 感觉像代码审查工具而不是控制平面

## 目标

1. 保持 `project` 足够轻量，使其仍然是规划容器。
2. 使工作区行为对 git 和非 git 项目都可理解。
3. 支持三种真实工作流而不强制其中一种：
   - 共享工作区 / 直接编辑工作流
   - 隔离的任务工作区工作流
   - 长期分支或操作者集成工作流
4. 提供一等的位置来查看工作产出：
   - 预览
   - PR
   - 分支
   - 提交
   - 文档和工件
5. 保持主导航和任务板简洁。
6. 无缝升级现有 Paperclip 用户到新模型，而不强制破坏性的重新配置。
7. 支持云托管的 Paperclip 部署，其中执行发生在远程或适配器管理的环境中而不是本地 worker。

## 非目标

- 将 Paperclip 变成完整的代码审查产品
- 要求每个任务都有自己的分支或 PR
- 要求每个项目都配置代码/工作区自动化
- 在 V1 中将工作区作为顶层全局导航原语
- 要求本地文件系统路径或本地 git checkout 才能使用工作区感知执行

## 核心产品决策

### 1. 项目保持规划对象

`项目` 仍然是围绕交付物或倡议分组工作的东西。

它可能有：

- 完全没有代码
- 一个默认代码库/工作区
- 多个代码库/工作区

项目不需要变得重量级。

### 2. 项目工作区是一等对象，但范围限定在项目下

`项目工作区` 是项目的持久代码库或根环境。

示例：

- 磁盘上的本地文件夹
- git 仓库 checkout
- monorepo 包根
- 非 git 的设计/文档文件夹
- 远程适配器管理的代码库引用

这是操作者配置一次的稳定锚点。

它不应成为主应用中的顶层侧边栏项。它应存在于项目体验中。

### 3. 执行工作区是一等运行时对象

`执行工作区` 是特定运行或任务实际执行的地方。

示例：

- 共享项目工作区本身
- 隔离的 git worktree
- 长期的操作者分支 checkout
- 适配器管理的远程沙箱
- 云智能体提供者的隔离分支/会话环境

此对象必须被显式记录，以便 Paperclip 能够：

- 显示工作发生在哪里
- 附加预览和运行时服务
- 链接 PR 和分支
- 决定清理行为
- 支持跨多个相关任务的复用

### 4. PR 是工作成果，不是核心任务模型

PR 是工作的输出，不是规划单元。

Paperclip 应将 PR 视为一种工作成果类型，链接回：

- 任务
- 执行工作区
- 可选的项目工作区

Git 特定的自动化应存在于工作区策略下，而不是核心任务抽象下。

### 5. 现有用户必须自动升级

Paperclip 已有用户和现有的项目/任务数据。任何新模型必须保持连续性。

产品应将现有安装默认设为合理的兼容模式：

- 没有工作区配置的现有项目继续正常工作
- 现有 `project_workspaces` 成为持久的 `项目工作区` 对象
- 现有项目执行工作区策略被向前映射而不是丢弃
- 没有显式工作区字段的任务继续继承当前行为

此迁移应感觉像是增量添加，而不是强制的重新入门流程。

### 6. 云托管的 Paperclip 必须是一等部署模式

Paperclip 不能假设它运行在与代码相同的机器上。

在云部署中，Paperclip 可能：

- 运行在 Vercel 或另一个 serverless 主机上
- 没有长期运行的本地 worker 进程
- 将执行委托给远程编码智能体或提供者管理的沙箱
- 从该远程环境接收回分支、PR、预览 URL 或工件

因此模型必须是可移植的：

- `项目工作区` 可能是远程管理的，而非本地
- `执行工作区` 可能没有本地 `cwd`
- `运行时服务` 可能通过提供者引用和 URL 跟踪，而不是宿主进程
- 工作成果采集必须处理外部拥有的预览和 PR

### 7. 子任务仍然是规划和所有权结构

子任务用于分解和并行所有权。

它们不等同于：

- 分支
- worktree
- PR
- 预览

它们可能与这些相关联，但不应被重载为这些含义。

## 术语

在产品文案中一致使用这些术语：

- `项目`：规划容器
- `项目工作区`：持久配置的代码库/根
- `执行工作区`：用于任务执行的实际运行时工作区
- `隔离的任务工作区`：面向用户的术语，指任务特定的派生工作区
- `工作成果`：预览、PR、分支、提交、工件、文档
- `运行时服务`：Paperclip 拥有或跟踪的工作区进程或服务

在迁移和部署消息中一致使用这些术语：

- `兼容模式`：无新工作区自动化情况下保留现有行为
- `适配器管理的工作区`：由远程或云执行提供者实现的工作区

避免教用户"工作区"总是意味着"我机器上的 git worktree"。

## 产品对象模型

## 1. 项目

现有对象。角色无根本变化。

### 必需行为

- 可以在没有代码/工作区配置的情况下存在
- 可以有零个或多个项目工作区
- 可以定义新任务继承的执行默认值

### 提议的字段

- `id`
- `companyId`
- `name`
- `description`
- `status`
- `goalIds`
- `leadAgentId`
- `targetDate`
- `executionWorkspacePolicy`
- `workspaces[]`
- `primaryWorkspace`

## 2. 项目工作区

持久的、已配置的、项目范围的代码库/根对象。

这应从当前 `project_workspaces` 表演进为更明确的产品对象。

### 动机

这分离了：

- "这个项目使用什么代码库/根？"

与：

- "这个任务运行在什么临时执行环境中？"

这使模型对单独用户保持简单，同时仍支持高级自动化。
它也让云托管的 Paperclip 部署可以指向代码库和远程而不假装 Paperclip 主机有直接的文件系统访问。

### 提议的字段

- `id`
- `companyId`
- `projectId`
- `name`
- `sourceType`
  - `local_path`
  - `git_repo`
  - `remote_managed`
  - `non_git_path`
- `cwd`
- `repoUrl`
- `defaultRef`
- `isPrimary`
- `visibility`
  - `default`
  - `advanced`
- `setupCommand`
- `cleanupCommand`
- `metadata`
- `createdAt`
- `updatedAt`

### 说明

- `sourceType=non_git_path` 很重要，使非 git 项目成为一等公民。
- 即使不使用隔离执行，这里也应允许 `setupCommand` 和 `cleanupCommand` 用于工作区根引导。
- 对于 monorepo，多个项目工作区可能指向一个仓库下的不同根或包。
- `sourceType=remote_managed` 对于云部署很重要，其中持久代码库由提供者/仓库元数据定义而不是本地 checkout 路径。

## 3. 项目执行工作区策略

项目级别的任务执行默认值。

这是主要的面向操作者的配置表面。

### 动机

这让 Paperclip 支持：

- 在共享工作区中直接编辑
- 用于任务并行的隔离工作区
- 长期集成分支工作流
- 返回分支或 PR 的远程云智能体执行

而不强迫每个任务或智能体暴露低级运行时配置。

- `defaultProjectWorkspaceId: uuid | null`
- `workspaceStrategy`
  - `type`
    - `project_primary`
    - `git_worktree`
    - `adapter_managed`
  - `baseRef`
  - `branchTemplate`
  - `worktreeParentDir`
  - `provisionCommand`
  - `teardownCommand`
- `branchPolicy`
  - `namingTemplate`
  - `allowReuseExisting`
  - `preferredOperatorBranch`
- `pullRequestPolicy`
  - `mode`
    - `disabled`
    - `manual`
    - `agent_may_open_draft`
    - `approval_required_to_open`
    - `approval_required_to_mark_ready`
  - `baseBranch`
  - `titleTemplate`
  - `bodyTemplate`
- `runtimePolicy`
  - `allowWorkspaceServices`

  - `autoHarvestOwnedUrls`
- `cleanupPolicy`
  - `mode`

## 4. 任务工作区绑定

任务级别的执行行为选择。

在正常情况下应保持轻量，仅在相关时展示更丰富的控制。

### 动机

不是代码项目中的每个任务都应创建新的派生工作区。

示例：

- 一个小修复可以在共享工作区中运行
- 三个相关任务可能有意共享一个集成分支
- 单独操作者可能直接在 `master` 上工作

### 提议的 `issues` 字段

- `projectWorkspaceId: uuid | null`
- `executionWorkspacePreference`
  - `inherit`
  - `shared_workspace`
  - `isolated_workspace`
  - `operator_branch`
  - `reuse_existing`
- `preferredExecutionWorkspaceId: uuid | null`
- `executionWorkspaceSettings`
  - 高级的每任务覆盖字段保留在这里

### 规则

- 如果项目没有工作区自动化，这些字段可能全部为 null
- 如果项目有一个主工作区，任务创建应静默默认使用它
- `reuse_existing` 仅限高级用户，应针对活跃的执行工作区而不是整个工作区范围
- 迁移期间没有这些字段的现有任务应表现为 `inherit`

## 5. 执行工作区

共享或派生运行时工作区的持久记录。

这是使清理、预览、PR 和分支复用可处理的缺失对象。

### 动机

没有显式的 `执行工作区` 记录，Paperclip 没有稳定的位置来附加：

- 派生的分支/worktree 身份
- 活跃的预览所有权
- PR 链接
- 清理状态
- "复用此现有集成分支"行为
- 远程提供者会话身份

### 提议的新对象

`execution_workspaces`

  - `project_primary`
  - `git_worktree`
  - `adapter_managed`
- `name`
- `status`
  - `active`
  - `idle`
  - `in_review`
  - `archived`
  - `cleanup_failed`
- `cwd`
- `repoUrl`
- `baseRef`
- `branchName`
- `providerRef`
- `providerType`
  - `local_fs`
  - `git_worktree`
  - `adapter_managed`
  - `cloud_sandbox`

### 说明

- `sourceIssueId` 是最初导致工作区创建的任务，不一定是后来链接到它的唯一任务。
- 多个任务可能在长期分支工作流中链接到同一个执行工作区。
- 对于远程执行工作区，`cwd` 可能为 null；提供者身份和工作成果链接仍使该对象有用。

## 6. 任务到执行工作区的链接

一个任务可能需要随时间链接到一个或多个执行工作区。

示例：

- 一个任务在共享工作区中开始，后来移到隔离的工作区
- 一次失败的尝试被归档，创建了新的工作区
- 多个任务有意共享一个操作者分支工作区

### 提议的对象

`issue_execution_workspaces`

### 提议的字段

- `issueId`
- `executionWorkspaceId`
- `relationType`
  - `current`
  - `historical`
  - `preferred`
- `createdAt`
- `updatedAt`

### UI 简化

大多数任务在主 UI 中应只显示一个当前工作区。历史链接属于高级/历史视图。

## 页面和 UI 模型

## 1. 全局导航

V1 中不要将 `工作区` 添加为顶层侧边栏项。

### 动机

这会使整个产品感觉基础设施化，即使对于不使用代码自动化的公司也是如此。

### 全局导航保持

- 仪表盘
- 收件箱
- 公司
- 智能体
- 目标
- 项目
- 任务
- 审批

工作区和工作成果应通过项目和任务详情视图展示。

## 2. 项目详情

添加项目子导航，保持规划优先、代码在后。

### 标签页

- `概览`
- `任务`
- `代码`
- `活动`

可选未来：

- `产出`

### `概览` 标签页

规划优先的摘要：

- 项目状态
- 目标
- 负责人
- 任务计数
- 顶层进度
- 最新重要工作成果摘要

### `任务` 标签页

- 默认仅显示顶层任务
- 显示父任务汇总：
  - 子任务数量
  - `x/y` 已完成
  - 活跃的预览/PR 徽章
- 可选切换：`显示子任务`

### `代码` 标签页

这是主要的工作区配置和可见性界面。

#### 部分：`项目工作区`

列出项目的持久项目工作区。

卡片/列表列：

- 工作区名称
- 源类型
- 路径或仓库
- 默认引用
- 主/默认徽章
- 活跃执行工作区数量
- 活跃任务数量
- 活跃预览数量
- 远程管理时的托管类型 / 提供者

操作：

- `添加工作区`
- `编辑`
- `设为默认`
- `归档`

#### 部分：`执行默认值`

字段：

- `启用工作区自动化`
- `默认任务执行模式`
  - `共享工作区`
  - `隔离工作区`
  - `操作者分支`
  - `适配器默认`
- `默认代码库`
- `允许任务覆盖`

#### 部分：`配置`

字段：

- `设置命令`
- `清理命令`
- `实现方式`
  - `共享工作区`
  - `Git worktree`
  - `适配器管理`
- `基础引用`
- `分支命名模板`
- `派生工作区父目录`

当所选工作区不是 git 支持的时，隐藏 git 特定字段。
当所选工作区是远程管理的时，隐藏本地路径特定字段。

#### 部分：`Pull Request`

字段：

- `PR 工作流`
  - `禁用`
  - `手动`
  - `智能体可以打开草稿 PR`
  - `打开 PR 需要审批`
  - `标记就绪需要审批`
- `默认基础分支`
- `PR 标题模板`
- `PR 正文模板`

#### 部分：`预览和运行时`

字段：

- `允许工作区运行时服务`
- `默认服务配置`
- `采集拥有的预览 URL`
- `跟踪外部预览 URL`

#### 部分：`清理`

字段：

- `清理模式`
  - `手动`
  - `当任务终态时`
  - `当 PR 关闭时`
  - `保留窗口后`
- `保留窗口`
- `预览活跃时保留`
- `PR 开放时保留`

## 3. 添加项目工作区流程

入口点：`项目 > 代码 > 添加工作区`

### 表单字段

- `名称`
- `源类型`
  - `本地文件夹`
  - `Git 仓库`
  - `非 git 文件夹`
  - `远程管理`
- `本地路径`
- `仓库 URL`
- `远程提供者`
- `远程工作区引用`
- `默认引用`
- `设为默认工作区`
- `设置命令`
- `清理命令`

### 行为

- 如果源类型是非 git，隐藏分支/PR 特定设置
- 如果源类型是 git，显示引用和可选的高级分支字段
- 如果源类型是远程管理，显示提供者/引用字段并隐藏仅本地路径配置
- 对于简单的单独用户，这可以是一个路径字段和一个保存按钮

## 4. 任务创建流程

任务创建默认应保持简单。

### 默认行为

如果所选项目：

- 没有工作区自动化：不显示工作区 UI
- 有一个默认项目工作区和默认执行模式：静默继承

### 仅在相关时显示 `工作区` 部分

#### 基本字段

- `代码库`
  - 默认选中的项目工作区
- `执行模式`
  - `项目默认`
  - `共享工作区`
  - `隔离工作区`
  - `操作者分支`

#### 仅高级字段

- `复用现有执行工作区`

此下拉应仅显示所选项目工作区的活跃执行工作区，标签如：

- `dotta/integration-branch`
- `PAP-447-add-worktree-support`
- `共享主工作区`

### 重要规则

不要默认显示包含每个可能工作区对象的选择器。

正常流程应感觉像：

- 选择项目
- 可选选择代码库
- 可选选择执行模式

而不是：

- 从根、派生 worktree、预览和分支名称的长混合列表中选择

### 迁移规则

对于现有用户，任务创建应继续看起来相同，直到项目显式启用更丰富的工作区行为。

## 5. 任务详情

任务详情应清晰地展示工作区和工作成果，但不变成代码托管 UI。

### 标题芯片

在标题/状态区域附近显示紧凑的摘要芯片：

- `代码库: Web App`
- `工作区: 共享`
- `工作区: PAP-447-add-worktree-support`
- `PR: 开放`
- `预览: 健康`

### 标签页

- `评论`
- `子任务`
- `工作成果`
- `活动`

### `工作成果` 标签页

部分：

- `当前工作区`
- `预览`
- `Pull Request`
- `分支和提交`
- `工件和文档`

#### 当前工作区面板

字段：

- 工作区名称
- 模式
- 分支
- 基础引用
- 上次使用
- 关联任务数量
- 清理状态

操作：

- `打开工作区详情`
- `标记为审查中`
- `请求清理`

#### Pull Request 卡片

字段：

- 标题
- 提供者
- 状态
- 审查状态
- 关联分支
- 打开/就绪/合并时间戳

操作：

- `打开 PR`
- `刷新状态`
- `请求 board 审查`

#### 预览卡片

字段：

- 标题
- URL
- 提供者
- 健康状况
- 所有权
- 更新时间

操作：

- `打开预览`
- `刷新`
- `归档`

## 6. 执行工作区详情

可从项目代码标签页或任务工作成果标签页到达。

不需要在主侧边栏中。

### 部分

- 身份
- 源任务
- 关联任务
- 分支/引用
- 提供者/会话身份
- 活跃运行时服务
- 预览
- PR
- 清理状态
- 事件/活动历史

### 动机

这是高级用户需要检查机制时去的地方。大多数用户在正常流程中不应需要它。

## 7. 收件箱行为

收件箱应展示可操作的工作成果事件，而不是每个实现细节。

### 显示收件箱项目

- 任务被分配或更新
- PR 需要 board 审查
- PR 被打开或标记为就绪
- 预览不健康
- 工作区清理失败
- 运行时服务失败
- 远程云智能体运行返回了需要审查的 PR 或预览

### 默认不显示

- 每次工作区心跳
- 每次分支更新
- 每次派生工作区创建

### 显示风格

如果收件箱项目关于预览或 PR，显示任务上下文：

- 任务标识符和标题
- 如果是子任务则显示父任务
- 相关时显示工作区名称

## 8. 任务列表和看板

保持列表和看板以规划为先。

### 默认行为

- 默认显示顶层任务
- 显示子任务的父任务汇总
- 不将每个子级执行细节扁平化到主看板

### 行/卡片装饰

对于有关联工作成果的任务，显示紧凑徽章：

- `1 PR`
- `2 预览`
- `共享工作区`
- `隔离工作区`

### 可选高级过滤器

- `有 PR`
- `有预览`
- `工作区模式`
- `代码库`

## 升级和迁移计划

## 1. 产品级迁移立场

迁移必须默认静默且保持兼容性。

现有用户不应被强迫：

- 在继续工作前手动创建新的工作区对象
- 重新标记旧任务
- 在基本任务流程继续运行之前学习新的工作区概念

## 2. 现有项目迁移

升级时：

- 现有 `project_workspaces` 记录被保留并显示为 `项目工作区`
- 当前主工作区保持为默认代码库
- 现有项目执行工作区策略被映射到新的 `项目执行工作区策略` 界面
- 没有执行工作区策略的项目保持兼容/共享模式

## 3. 现有任务迁移

升级时：

- 现有任务默认为 `executionWorkspacePreference=inherit`
- 如果任务已有执行工作区设置，直接向前映射
- 如果任务没有显式工作区数据，保留现有行为且不强制用户可见的选择

## 4. 现有运行/运行时迁移

升级时：

- 活跃或最近的运行时服务可以在可行的情况下回填到执行工作区历史中
- 缺失的历史不应阻塞推出；前向正确性比完美的历史重建更重要

## 5. 推出 UX

在 UI 中使用增量语言：

- `代码`
- `工作区自动化`
- `可选`
- `高级`

避免暗示用户以前"错误地"使用产品的迁移文案。

## 云部署要求

## 1. Paperclip 主机和执行主机必须解耦

Paperclip 可能运行在：

- 本地，有直接文件系统访问
- 云应用主机如 Vercel
- 混合设置，有外部任务运行器

工作区模型必须在所有三种情况下工作。

## 2. 远程执行必须支持一等工作成果报告

云智能体应能够：

- 解析项目工作区
- 远程实现适配器管理的执行工作区
- 产生分支
- 打开或更新 PR
- 发布预览 URL
- 注册工件

而 Paperclip 主机本身不运行本地 git 或本地预览进程。

## 3. 仅本地假设必须是可选的

以下必须是可选的，而非必需：

- 本地 `cwd`
- 本地 git CLI
- 宿主管理的 worktree 目录
- 宿主拥有的长期运行预览进程

## 4. 相同产品界面，不同提供者行为

UI 不应分裂为"本地模式"和"云模式"产品。

取而代之：

- 本地项目显示路径/git 实现细节
- 云项目显示提供者/引用细节
- 两者都展示相同的高层对象：
  - 项目工作区
  - 执行工作区
  - 工作成果
  - 运行时服务或预览

## 行为规则

## 1. 清理不得依赖智能体记住 `in_review`

智能体仍可使用 `in_review`，但清理行为必须由策略和观察到的状态治理。

### 在以下任一条件为真时保持执行工作区存活

- 关联任务非终态
- 关联 PR 处于开放状态
- 关联预览/运行时服务处于活跃状态
- 工作区仍在保留窗口内

### 隐藏而非积极删除

归档或空闲的工作区应在硬清理之前从默认列表中隐藏。

## 2. 多个任务可以有意共享一个执行工作区

这是 Paperclip 支持以下场景的方式：

- 在共享分支上的单独开发
- 操作者集成分支
- 相关功能批量打包到一个 PR 中

这是不强制 1 个任务 = 1 个工作区 = 1 个 PR 的关键原因。

## 3. 隔离的任务工作区保持可选

即使在 git 密集型项目中，隔离工作区也应是可选的。

共享模式有效的示例：

- 小型 bug 修复
- 无分支的原型开发
- 非 git 项目
- 单用户本地工作流

## 4. PR 策略属于 git 支持的工作区策略

PR 自动化决策应在项目/工作区策略层做出。

任务应仅：

- 展示结果 PR
- 路由审批/审查请求
- 显示状态和审查状态

## 5. 工作成果是面向用户的统一器

预览、PR、提交和工件都应通过一个一致的任务级别界面可发现。

这使 Paperclip 专注于协调和可见性，而不是将产出分散在许多隐藏的子系统中。

## 推荐的实现顺序

## 第一阶段：在 UI 中明确当前对象

1. 展示 `项目 > 代码` 标签页
2. 在那里显示现有项目工作区
3. 用修订的文案重新启用项目级执行工作区策略
4. 保持任务创建简单，使用继承的默认值

## 第二阶段：添加显式执行工作区记录

1. 添加 `execution_workspaces`
2. 将运行、任务、预览和 PR 链接到它
3. 添加简单的执行工作区详情页面
4. 使 `cwd` 可选并确保从第一天起支持提供者管理的远程工作区

## 第三阶段：添加工作成果模型

1. 添加 `issue_work_products`
2. 摄入 PR、预览、分支、提交
3. 添加任务 `工作成果` 标签页
4. 为可操作的工作成果状态变更添加收件箱项目
5. 支持远程智能体创建的 PR/预览报告而无需本地所有权

## 第四阶段：添加高级复用和清理工作流

1. 添加 `复用现有执行工作区`
2. 添加清理生命周期 UI
3. 添加操作者分支工作流快捷方式
4. 添加更丰富的外部预览采集
5. 在改善现有用户连续性的地方添加迁移工具/回填

## 为什么这个模型是正确的

这个模型保持了产品的平衡：

- 对单独用户足够简单
- 对真实工程团队足够强大
- 对非 git 项目足够灵活
- 对治理 PR 和预览足够明确

最重要的是，它保持了抽象的干净：

- 项目规划工作
- 项目工作区定义持久代码库
- 执行工作区定义工作运行的位置
- 工作成果定义工作产出了什么
- PR 保持为输出，不是核心任务模型

它还使推出切实可行：

- 现有用户可以升级而不破坏工作流
- 本地优先安装保持简单
- 云托管的 Paperclip 部署保持一等

这比任何极端都更适合 Paperclip：

- 隐藏工作区行为直到没人理解
- 或让整个应用围绕代码托管机制旋转
