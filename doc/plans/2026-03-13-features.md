# 功能规格

## 1）引导式入门 + 首次任务魔法

仓库已有 `onboard`、`doctor`、`run`、部署模式，甚至面向智能体的入门文本/技能端点，但也存在当前的入门/认证验证问题和一个开放的"入门失败"报告。这意味着这不仅是打磨；而是产品关键问题。([GitHub][1])

### 产品决策

用**访谈式入门**替代"配置优先的入门"。

### 我们想要的

- 先问 3-4 个问题，而不是 20 个设置。
- 自动生成正确的路径：本地单机、共享私有或公有云。
- 检测已存在的智能体/运行时环境。
- 让 Claude/OpenClaw/Codex 帮助完成设置成为常态。
- 入门结束时有一个**真实的首个任务**，而不是空白仪表盘。

### 我们不想要的

- 在展示价值之前出现提供商术语。
- "去找一个 API 密钥"作为默认的第一条指令。
- 安装成功但用户仍不确定接下来该做什么。

### 提议的 UX

首次运行时，展示一个访谈：

```ts
type OnboardingProfile = {
  useCase: "startup" | "agency" | "internal_team";
  companySource: "new" | "existing";
  deployMode: "local_solo" | "shared_private" | "shared_public";
  autonomyMode: "hands_on" | "hybrid" | "full_auto";
  primaryRuntime: "claude_code" | "codex" | "openclaw" | "other";
};
```

问题：

1. 你在构建什么？
2. 这是一个新公司、现有公司，还是服务/代理团队？
3. 你是在单台机器上单独工作、与团队私有共享，还是公开部署？
4. 你想要全自动、混合，还是严格手动控制？

然后 Paperclip 应该：

- 检测已安装的 CLI/提供商/订阅
- 推荐匹配的部署/认证模式
- 生成本地 `onboarding.txt` / LLM 交接 prompt
- 提供按钮：**"在 Claude 中打开 / 复制设置 prompt"**
- 创建启动对象：

  - 公司
  - 公司目标
  - CEO
  - 创始工程师或等效的第一个下属
  - 首个建议任务

### 后端 / API

- 添加 `GET /api/onboarding/recommendation`
- 添加 `GET /api/onboarding/llm-handoff.txt`
- 复用现有的邀请/入门/技能模式进行本地优先引导
- 将入门答案持久化到实例配置中用于后续默认值

### 验收标准

- 使用支持的本地运行时的全新安装无需手动 JSON/env 编辑即可完成。
- 用户在离开入门前看到第一个智能体实际动作。
- 空白仪表盘不再是安装后的默认状态。
- 如果缺少必需的依赖项，错误信息是规范的且可从 UI/CLI 修复。

### 非目标

- 账户创建
- 企业 SSO
- 对每个运行时的完美提供商自动检测

## 2）Board 命令界面，而非通用聊天

这里有一个真实的张力：文字记录说用户想要"与我的 CEO 聊天"，而公开的产品定义说 Paperclip **不是聊天机器人**，V1 通信是**仅任务 + 评论**。同时，仓库已经在探索插件基础设施，甚至通过插件 SSE 流的聊天插件。干净的解决方案是：**让核心界面具有对话感，但保持数据模型以任务/线程为中心；将完整聊天保留为可选插件**。([GitHub][2])

### 产品决策

构建一个**命令编辑器**，由任务/评论/审批支撑，而不是独立的聊天子系统。

### 我们想要的

- 用户有"与 CEO 对话"的感觉。
- 每次对话最终都附加到一个真实的公司对象上。
- 战略讨论可以产生任务、工件和审批。

### 我们不想要的

- 与组织脱节的空白"与 AI 聊天"主屏幕。
- 又一个智能体聊天产品。

### 提议的 UX

添加带模式的全局编辑器：

```ts
type ComposerMode = "ask" | "task" | "decision";
type ThreadScope = "company" | "project" | "issue" | "agent";
```

示例：

- 在仪表盘上："让 CEO 制定一个招聘计划" → 创建一个范围为公司的 `strategy` 任务/线程。
- 在智能体页面："告诉设计师把这个做得更干净" → 向任务追加指令评论或生成新的委派任务。
- 在审批页面："你为什么要求招聘？" → 向审批上下文追加 board 评论。

添加任务类型：

```ts
type IssueKind = "task" | "strategy" | "question" | "decision";
```

### 后端 / 数据模型

优先扩展现有的 `issues` 而不是创建 `chats`：

- `issues.kind`
- `issues.scope`
- 可选的 `issues.target_agent_id`
- 评论元数据：`comment.intent = hint | correction | board_question | board_decision`

### 验收标准

- 用户可以从仪表盘"询问 CEO"并在公司范围的线程中收到回复。
- 从该线程中，用户可以一键创建/批准任务。
- 此功能的 v1 不需要单独的聊天数据库。

### 非目标

- 消费者聊天 UX
- 模型市场
- 与公司上下文无关的通用助手

## 3）实时组织可见性 + 可解释性层

核心产品承诺已经是可见性和治理，但目前文字记录清楚表明 UI 仍然太接近原始智能体执行。仓库已有组织图、活动、心跳运行、成本和智能体详情界面；缺失的是上层的解释层。([GitHub][1])

### 产品决策

默认 UI 为**人类可读的运营摘要**，原始日志在下一层。

### 我们想要的

- 在公司层面："谁在活跃，他们在做什么，团队间有什么工作在流转"
- 在智能体层面："计划是什么，完成了哪一步，产出了什么"
- 在运行层面："先看摘要，再看文字记录"

### 提议的 UX

公司页面：

- 带实时活跃状态指示器的组织图
- 工作流转时节点间的委派动画
- 当前开放的优先事项
- 待审批
- 消耗 / 预算警告条

智能体页面：

- 状态卡片
- 当前任务
- 计划检查清单
- 最新工件
- 上次运行摘要
- 可展开的原始追踪/日志

运行页面：

- **摘要**
- **步骤**
- **原始文字记录 / 工具调用**

### 后端 / API

从当前运行/活动数据生成运行视图模型：

```ts
type RunSummary = {
  runId: string;
  headline: string;
  objective: string | null;
  currentStep: string | null;
  completedSteps: string[];
  delegatedTo: { agentId: string; issueId?: string }[];
  artifactIds: string[];
  warnings: string[];
};
```

第一阶段可以在服务器端从现有运行日志/评论中派生。仅在需要时才持久化。

### 验收标准

- Board 无需阅读 shell 命令即可了解正在发生的事情。
- 原始日志仍可访问，但不是默认界面。
- 首次任务 / 首次招聘 / 首次完成等里程碑时刻有明显庆祝。

### 非目标

- 过度设计的动画系统
- 在核心数据质量存在之前追求完美的语义摘要

## 4）工件系统：附件、文件浏览器、预览

这个缺口已经在仓库中显现。存储已存在，附件端点已有，但当前问题表明附件仍然主要是图片为中心的，评论附件渲染不完整。同时，文字记录希望计划、文档、文件和生成的网页被干净地呈现。([GitHub][4])

### 产品决策

引入一等的**工件**模型，统一：

- 上传/生成的文件
- 感兴趣的工作区文件
- 预览 URL
- 生成的文档/报告

### 我们想要的

- 计划、规格、CSV、markdown、PDF、日志、JSON、HTML 输出
- 从任务/运行/公司页面轻松发现
- 项目工作区的轻量文件浏览器
- 生成网站/应用的预览链接

### 我们不想要的

- 强迫智能体将所有内容内联粘贴到评论中
- 将 HTML 塞入评论正文作为变通方案
- 完整的 Web IDE

### 第一阶段：修复明显的缺口

- 接受任务附件的非图片 MIME 类型
- 正确地将文件附加到评论
- 在任务页面显示文件元数据 + 下载/打开

### 第二阶段：引入工件

```ts
type ArtifactKind = "attachment" | "workspace_file" | "preview" | "report_link";

interface Artifact {
  id: string;
  companyId: string;
  issueId?: string;
  runId?: string;
  agentId?: string;
  kind: ArtifactKind;
  title: string;
  mimeType?: string;
  filename?: string;
  sizeBytes?: number;
  storageKind: "local_disk" | "s3" | "external_url";
  contentPath?: string;
  previewUrl?: string;
  metadata: Record<string, unknown>;
}
```

### UX

任务页面添加**交付物**部分：

- 文件
- 报告
- 预览链接
- 最新生成的工件突出显示在顶部

项目页面添加**文件**标签页：

- 文件夹树
- 最近变更
- "打开产出文件"快捷方式

### 预览处理

对于 HTML/静态输出：

- 本地部署 → 打开本地预览 URL
- 共享/公有部署 → 通过配置的预览服务或静态存储托管
- 预览 URL 作为工件注册回任务

### 验收标准

- 智能体可以附加 `.md`、`.txt`、`.json`、`.csv`、`.pdf` 和 `.html`。
- 用户可以从任务页面打开/下载它们。
- 生成的静态站点可以从任务页面打开，无需在文件系统中查找。

### 非目标

- 浏览器 IDE
- 协作文档编辑器
- 完整的对象存储管理 UI

## 5）共享/云部署 + 云运行时

仓库在文档中已有清晰的部署方案：`local_trusted`、`authenticated/private` 和 `authenticated/public`，加上 Tailscale 指导。路线图明确提到了云智能体如 Cursor / e2b。这意味着下一步不是发明部署模型；而是让共享/云路径成为规范的且可用于生产的。([GitHub][5])

### 产品决策

让**共享/私有部署**和**公有/云部署**成为一等支持的模式，并为云执行智能体添加**远程运行时驱动**。

### 我们想要的

- 一个团队可以实际共享的实例
- 本地优先路径可升级到私有/公有而不改变心智模型
- 非本地运行时的远程智能体执行

### 提议的架构

更明确地分离**控制平面**和**执行运行时**：

```ts
type RuntimeDriver = "local_process" | "remote_sandbox" | "webhook";

interface ExecutionHandle {
  externalRunId: string;
  status: "queued" | "running" | "completed" | "failed" | "cancelled";
  previewUrl?: string;
  logsUrl?: string;
}
```

第一个远程驱动：`remote_sandbox` 用于 e2b 风格的执行。

### 交付物

- 规范部署方案：

  - 本地单机
  - 共享私有（Tailscale/私有认证）
  - 公有云（托管 PostgreSQL + 对象存储 + 公共 URL）

- 运行时健康页面
- 适配器/运行时能力矩阵
- 一个官方参考部署路径

### UX

新的"部署"设置页面：

- 实例模式
- 认证/暴露
- 存储/数据库状态
- 已配置的运行时驱动
- 健康和可达性检查

### 验收标准

- 两个人可以登录同一个认证/私有实例并同时使用。
- 公有部署可以通过至少一个远程运行时运行智能体。
- `doctor` 能捕获缺失的公有/私有配置并给出具体修复方案。

### 非目标

- 完全托管的 Paperclip SaaS
- v1 中支持每个可能的云提供商

## 6）多用户协作（最小化，非企业 RBAC）

这是与当前 V1 规格最大的刻意偏离。公开地，V1 仍然说"单一人类 board 操作者"并将基于角色的人类粒度放在范围之外。但文字记录是对的，如果 Paperclip 要对团队真正有用，共享使用是必要的。关键是做一个**最小的协作模型**，而不是庞大的权限系统。([GitHub][2])

### 产品决策

发布**粗粒度的多用户公司成员资格**，而不是细粒度的企业 RBAC。

### 提议的角色

```ts
type CompanyRole = "owner" | "admin" | "operator" | "viewer";
```

- **owner**：实例/公司所有权，用户邀请，配置
- **admin**：管理组织、智能体、预算、审批
- **operator**：创建/更新任务，与智能体交互，查看工件
- **viewer**：只读

### 数据模型

```ts
interface CompanyMembership {
  userId: string;
  companyId: string;
  role: CompanyRole;
  invitedByUserId: string;
  createdAt: string;
}
```

延伸目标（后续）：

- 可选的项目/团队范围控制

### 我们想要的

- 真实团队的共享仪表盘
- 活动日志中的用户归属
- 简单的邀请流程
- 保持公司级别的隔离

### 我们不想要的

- 字段级 ACL
- SCIM/SSO/企业管理控制台
- 每页十个权限开关

### 验收标准

- 3 人团队可以使用一个共享的 Paperclip 实例。
- 每个用户操作在活动中正确归属。
- 公司成员资格边界被强制执行。
- 查看者不能变更；操作者/管理员可以。

### 非目标

- 企业 RBAC
- 跨公司矩阵权限
- 首个版本中的多 board 治理逻辑

## 7）自动模式 + 中断/恢复

这是一个产品行为问题，而不是 UI 锦上添花。如果智能体无法继续工作或接受课程修正而不重启，自治模型感觉是假的。

### 产品决策

让自动模式和运行中中断成为一等运行时语义。

### 我们想要的

- 自动模式持续运行直到被审批、预算或显式暂停阻止。
- 运行中的"你漏了这个"修正不丢失会话连续性。
- 智能体等待、阻塞或暂停时的清晰状态。

### 提议的状态模型

```ts
type RunState =
  | "queued"
  | "running"
  | "waiting_approval"
  | "waiting_input"
  | "paused"
  | "completed"
  | "failed"
  | "cancelled";
```

添加 board 插入作为可恢复的输入事件：

```ts
interface RunMessage {
  runId: string;
  authorUserId: string;
  mode: "hint" | "correction" | "hard_override";
  body: string;
  resumeCurrentSession: boolean;
}
```

### UX

活跃运行上的按钮：

- 暂停
- 恢复
- 中断
- 终止
- 从头重启

中断打开一个小编辑器，明确说明：

- 继续当前会话
- 或重启运行

### 验收标准

- board 评论可以恢复活跃会话而不是生成新会话。
- "继续"路径的会话 ID 保持稳定。
- UI 清楚区分阻塞、等待和暂停状态。

### 非目标

- 同一运行文字记录的多用户同时实时编辑
- 在运行时语义修复之前追求完美的对话 UX

## 8）成本安全 + 心跳/运行时加固

这可能是最重要的即时工作流。文字记录说 token 消耗是最高的痛点，仓库当前有关于预算执行证据、入门/认证验证和断路器式浪费防止的活跃问题。公开文档已承诺硬性预算，任务跟踪器正指向缺失的运营保护。([GitHub][6])

### 产品决策

将此视为 **P0 运行时合约**，而不是锦上添花。

### A 部分：确定性唤醒门控

在调用 LLM 之前进行低成本、显式的工作检测。

```ts
type WakeReason =
  | "new_assignment"
  | "new_comment"
  | "mention"
  | "approval_resolved"
  | "scheduled_scan"
  | "manual";
```

规则：

- 如果没有新的可操作输入，不调用模型
- 计划扫描应首先是低成本的策略检查，而不是完整的推理过程

### B 部分：预算合约

保持现有的公开承诺，但使其不可否认：

- 80% 时警告
- 100% 时自动暂停
- 可见的审计追踪
- 需要显式的 board 覆盖才能继续

### C 部分：断路器

添加每智能体的运行时防护：

```ts
interface CircuitBreakerConfig {
  enabled: boolean;
  maxConsecutiveNoProgress: number;
  maxConsecutiveFailures: number;
  tokenVelocityMultiplier: number;
}
```

触发条件：

- N 次运行无任务/状态/评论进展
- 连续 N 次失败
- token 峰值相对于滚动平均值

### D 部分：重构心跳服务

将当前编排拆分为模块：

- 唤醒检测器
- checkout/锁管理器
- 适配器运行器
- 会话管理器
- 成本记录器
- 断路器评估器
- 事件流式传输器

### E 部分：回归套件

以下必须有自动化证明：

- 入门/认证矩阵
- 80/100 预算行为
- 无跨公司认证泄漏
- 空闲时无虚假唤醒行为
- 活跃运行恢复/中断
- 远程运行时烟雾测试

### 验收标准

- 没有新工作的空闲组织不从心跳扫描产生模型调用。
- 80% 仅显示警告。
- 100% 暂停智能体并阻止继续执行直到覆盖。
- 断路器暂停在审计/活动中可见。
- 运行时模块有明确的合约且可独立测试。

### 非目标

- 完美的自主优化
- 在每个适配器/提供商中消除所有浪费的调用

## 9）项目工作区、预览和 PR 交接——而不是变成 GitHub

这是解决代码工作流争论的正确方式。仓库已有 worktree 本地实例、项目 `workspaceStrategy.provisionCommand` 和适配器级 git worktree 隔离的 RFC。这是正确的架构方向：**项目执行策略和工作区隔离**，而不是内置 PR 审查。([GitHub][7])

### 产品决策

Paperclip 应管理**任务 → 工作区 → 预览/PR → 审查交接**生命周期，但将 diff/审查/合并留给外部工具。

### 提议的配置

优先使用仓库本地项目配置：

```yaml
# .paperclip/project.yml
execution:
  workspaceStrategy: shared | worktree | ephemeral_container
  deliveryMode: artifact | preview | pull_request
  provisionCommand: "pnpm install"
  teardownCommand: "pnpm clean"
  preview:
    command: "pnpm dev --port $PAPERCLIP_PREVIEW_PORT"
    healthPath: "/"
    ttlMinutes: 120
  vcs:
    provider: github
    repo: owner/repo
    prPerIssue: true
    baseBranch: main
```

### 规则

- 对于非代码项目：`deliveryMode=artifact`
- 对于 UI/应用工作：`deliveryMode=preview`
- 对于 git 支持的工程项目：`deliveryMode=pull_request`
- 对于 `prPerIssue=true` 的 git 项目，一个任务映射到一个隔离的分支/worktree

### UX

任务页面显示：

- 工作区链接/状态
- 预览 URL（如可用）
- PR URL（如已创建）
- "重新打开预览"按钮（带 TTL）
- 生命周期：

  - `todo`
  - `in_progress`
  - `in_review`
  - `done`

### 我们想要的

- 在一个仓库上安全的并行智能体工作
- 可预览的输出
- 外部 PR 审查
- 项目定义的钩子，而不是硬编码的假设

### 我们不想要的

- 内置的 diff 查看器
- 合并队列
- Jira 克隆
- 非代码工作的强制 PR

### 验收标准

- 多个工程智能体可以并发工作而不污染工作区。
- 当项目处于 PR 模式时，任务包含分支/worktree/预览/PR 元数据。
- 预览可以按需重新打开直到 TTL 过期。

### 非目标

- 替代 GitHub/GitLab
- 第一天就为每个框架提供通用预览托管

---

## 10）插件系统作为逃生口

路线图已包含插件，GitHub 讨论围绕它很活跃，还有一个开放的问题提议 SSE 桥接，专门用于启用流式插件 UI 如聊天、日志和监视器。这正是可选界面的正确位置。([GitHub][1])

### 产品决策

保持控制平面核心精简；将可选的高变化率体验放入插件。

### 第一方插件目标

- 聊天
- 知识库 / RAG
- 日志尾随 / 实时构建输出
- 自定义追踪或队列
- 文档编辑器 / 提案构建器

### 插件清单

```ts
interface PluginManifest {
  id: string;
  version: string;
  requestedPermissions: (
    | "read_company"
    | "read_issue"
    | "write_issue_comment"
    | "create_issue"
    | "stream_ui"
  )[];
  surfaces: ("company_home" | "issue_panel" | "agent_panel" | "sidebar")[];
  workerEntry: string;
  uiEntry: string;
}
```

### 平台要求

- 宿主 ↔ worker 操作桥接
- SSE/UI 流式传输
- 公司范围的认证
- 权限声明
- UI 中的界面插槽

### 验收标准

- 插件可以实时向 UI 流式传输事件。
- 聊天插件可以对话而不需要聊天成为 Paperclip 的核心产品。
- 插件权限是公司范围的且可审计。

### 非目标

- 插件直接变更核心 schema
- 没有显式权限的任意特权代码执行

## 我建议的优先顺序

根据仓库状态和文字记录，我会这样排序：

**P0**

1. 成本安全 + 心跳加固
2. 引导式入门 + 首次任务魔法
3. 共享/云部署基础
4. 工件第一阶段：非图片附件 + 交付物展示

**P1** 5. Board 命令界面 6. 可见性/可解释性层 7. 自动模式 + 中断/恢复 8. 最小多用户协作

**P2** 9. 项目工作区 / 预览 / PR 生命周期 10. 插件系统 + 可选聊天插件 11. 面向初创/代理/内部团队入门的模板/预设扩展

为什么这个顺序：当前仓库已经在入门失败、认证/入门验证、预算执行和浪费的 token 消耗上承受压力。如果这些不稳固，其他一切感觉令人印象深刻但不安全。([GitHub][3])

## 底线

最佳综合是：

- **保持** Paperclip 作为 board 级控制平面。
- **不要**让聊天、代码审查或工作流构建成为核心身份。
- **确实**让产品感觉具有对话性、可见性、产出导向和共享性。
- **确实**通过工作区/预览/PR 链接让编码工作流成为集成界面。
- **使用插件**来提供更丰富的边缘功能如聊天和知识。

这保持了仓库当前产品方向不变，同时解决了文字记录中几乎所有痛点。
