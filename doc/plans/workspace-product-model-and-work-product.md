# 工作区产品模型、工作产出与 PR 流程

## 背景

Paperclip 需要支持两种非常不同但同样有效的工作方式：

- 一个独立开发者直接在 `master` 上工作，或者在一个甚至不是 git 仓库的文件夹中工作
- 一个更大的工程工作流，具有隔离分支、预览、Pull Request 和清理自动化

今天，Paperclip 已经有了这个模型的雏形：

- `projects` 可以携带执行工作区策略
- `project_workspaces` 已作为持久的项目范围对象存在
- 任务可以携带执行工作区设置
- 运行时服务可以附加到工作区或任务

缺少的是一个清晰的产品模型和 UI，使这些能力可理解和可操作。

主要的产品风险是一个概念过载地承担太多：

- 让子任务做分支或 PR 的工作
- 使项目过于基础设施化
- 使工作区如此隐蔽以至于用户无法形成心智模型
- 使 Paperclip 感觉像一个代码审查工具而不是控制面

## 目标

1. 保持 `project` 足够轻量以继续作为规划容器。
2. 使工作区行为对 git 和非 git 项目都可理解。
3. 支持三种真实工作流而不强制其中之一：
   - 共享工作区/直接编辑工作流
   - 隔离任务工作区工作流
   - 长期分支或操作者集成工作流
4. 提供一等位置查看工作输出：
   - 预览
   - PR
   - 分支
   - 提交
   - 文档和产物
5. 保持主导航和任务板简洁。
6. 将现有 Paperclip 用户无缝升级到新模型，无需强制破坏性重新配置。
7. 支持云托管的 Paperclip 部署，其中执行发生在远程或适配器管理的环境中，而非本地工作者。

## 非目标

- 将 Paperclip 变成完整的代码审查产品
- 要求每个任务都有自己的分支或 PR
- 要求每个项目都配置代码/工作区自动化
- 在 V1 中使工作区成为顶级全局导航原语
- 要求本地文件系统路径或本地 git 检出才能使用工作区感知执行

## 核心产品决策

### 1. 项目保持为规划对象

`project` 仍然是围绕可交付成果或倡议分组工作的东西。

它可能有：

- 完全没有代码
- 一个默认的代码库/工作区
- 多个代码库/工作区

项目不需要变得重量级。

### 2. 项目工作区是一等对象，但限定在项目下

`项目工作区` 是项目的持久代码库或根环境。

示例：

- 磁盘上的本地文件夹
- git 仓库检出
- monorepo 包根目录
- 非 git 的设计/文档文件夹
- 远程适配器管理的代码库引用

这是操作者一次性配置的稳定锚点。

它不应成为主应用中的顶级侧边栏项。它应存在于项目体验下。

### 3. 执行工作区是一等运行时对象

`执行工作区` 是特定运行或任务实际执行的地方。

示例：

- 共享的项目工作区本身
- 隔离的 git worktree
- 长期的操作者分支检出
- 适配器管理的远程沙箱
- 云智能体提供商的隔离分支/会话环境

此对象必须显式记录，以便 Paperclip 可以：

- 显示工作发生在哪里
- 附加预览和运行时服务
- 链接 PR 和分支
- 决定清理行为
- 支持跨多个相关任务的复用

### 4. PR 是工作产出，而非核心任务模型

PR 是工作的输出，而非规划单元。

Paperclip 应将 PR 视为一种工作产出类型，链接回：

- 任务
- 执行工作区
- 可选地链接到项目工作区

Git 特定的自动化应存在于工作区策略下，而非核心任务抽象下。

### 5. 现有用户必须自动升级

Paperclip 已有用户和现有项目/任务数据。任何新模型都必须保持连续性。

产品应将现有安装默认为合理的兼容模式：

- 没有工作区配置的现有项目继续正常工作
- 现有的 `project_workspaces` 成为持久的 `项目工作区` 对象
- 现有的项目执行工作区策略被向前映射而非丢弃
- 没有显式工作区字段的任务继续继承当前行为

此迁移应感觉像是增量添加，而非强制性的重新引导流程。

### 6. 云托管的 Paperclip 必须是一等部署模式

Paperclip 不能假设它运行在与代码相同的机器上。

在云部署中，Paperclip 可能：

- 在 Vercel 或另一个无服务器主机上运行
- 没有长期运行的本地工作者进程
- 将执行委托给远程编码智能体或提供商管理的沙箱
- 从该远程环境接收回分支、PR、预览 URL 或产物

因此模型必须是可移植的：

- `项目工作区` 可能是远程管理的，而非本地的
- `执行工作区` 可能没有本地 `cwd`
- `运行时服务` 可能通过提供商引用和 URL 跟踪，而非主机进程
- 工作产出收集必须处理外部拥有的预览和 PR

### 7. 子任务保持为规划和所有权结构

子任务用于分解和并行所有权。

它们不等同于：

- 一个分支
- 一个 worktree
- 一个 PR
- 一个预览

它们可能与这些相关联，但不应被过载为它们的含义。

## 术语

在产品文案中一致使用以下术语：

- `项目`：规划容器
- `项目工作区`：持久配置的代码库/根目录
- `执行工作区`：用于任务执行的实际运行时工作区
- `隔离任务工作区`：面向用户的术语，指任务特定的派生工作区
- `工作产出`：预览、PR、分支、提交、产物、文档
- `运行时服务`：Paperclip 为工作区拥有或跟踪的进程或服务

在迁移和部署消息中一致使用以下术语：

- `兼容模式`：保留的现有行为，无新的工作区自动化
- `适配器管理的工作区`：由远程或云执行提供商实现的工作区

避免教用户"工作区"总是意味着"我机器上的 git worktree"。

## 产品对象模型

## 1. 项目

现有对象。角色无根本变化。

### 必需行为

- 可以在没有代码/工作区配置的情况下存在
- 可以有零个或多个项目工作区
- 可以定义新任务继承的执行默认值

### 建议字段

- `id`
- `companyId`
- `name`
- `description`
- `status`
- `goalIds`
- `leadAgentId`
- `targetDate`
- `executionWorkspacePolicy`
- `workspaces[]`
- `primaryWorkspace`

## 2. 项目工作区

持久的、已配置的、项目范围的代码库/根目录对象。

这应从当前的 `project_workspaces` 表演进为更显式的产品对象。

### 动机

这分离了：

- "这个项目使用什么代码库/根目录？"

与：

- "这个任务在什么临时执行环境中运行？"

这使模型对独立用户保持简单，同时仍支持高级自动化。
它还让云托管的 Paperclip 部署可以指向代码库和远程仓库，而不必假装 Paperclip 主机有直接的文件系统访问。

### 建议字段

- `id`
- `companyId`
- `projectId`
- `name`
- `sourceType`
  - `local_path`
  - `git_repo`
  - `remote_managed`
  - `non_git_path`
- `cwd`
- `repoUrl`
- `defaultRef`
- `isPrimary`
- `visibility`
  - `default`
  - `advanced`
- `setupCommand`
- `cleanupCommand`
- `metadata`
- `createdAt`
- `updatedAt`

### 说明

- `sourceType=non_git_path` 很重要，这样非 git 项目是一等的。
- 即使不使用隔离执行，`setupCommand` 和 `cleanupCommand` 也应在此允许用于工作区根引导。
- 对于 monorepo，多个项目工作区可能指向一个仓库下的不同根目录或包。
- `sourceType=remote_managed` 对于云部署很重要，其中持久代码库由提供商/仓库元数据定义，而非本地检出路径。

## 3. 项目执行工作区策略

项目级别的任务执行方式默认值。

这是主要的面向操作者的配置界面。

### 动机

这让 Paperclip 支持：

- 在共享工作区中直接编辑
- 用于任务并行的隔离工作区
- 长期集成分支工作流
- 返回分支或 PR 的远程云智能体执行

而不强制每个任务或智能体公开低级运行时配置。

### 建议字段

- `enabled: boolean`
- `defaultMode`
  - `shared_workspace`
  - `isolated_workspace`
  - `operator_branch`
  - `adapter_default`
- `allowIssueOverride: boolean`
- `defaultProjectWorkspaceId: uuid | null`
- `workspaceStrategy`
  - `type`
    - `project_primary`
    - `git_worktree`
    - `adapter_managed`
  - `baseRef`
  - `branchTemplate`
  - `worktreeParentDir`
  - `provisionCommand`
  - `teardownCommand`
- `branchPolicy`
  - `namingTemplate`
  - `allowReuseExisting`
  - `preferredOperatorBranch`
- `pullRequestPolicy`
  - `mode`
    - `disabled`
    - `manual`
    - `agent_may_open_draft`
    - `approval_required_to_open`
    - `approval_required_to_mark_ready`
  - `baseBranch`
  - `titleTemplate`
  - `bodyTemplate`
- `runtimePolicy`
  - `allowWorkspaceServices`
  - `defaultServicesProfile`
  - `autoHarvestOwnedUrls`
- `cleanupPolicy`
  - `mode`
    - `manual`
    - `when_issue_terminal`
    - `when_pr_closed`
    - `retention_window`
  - `retentionHours`
  - `keepWhilePreviewHealthy`
  - `keepWhileOpenPrExists`

## 4. 任务工作区绑定

任务级别的执行行为选择。

在正常情况下应保持轻量，仅在相关时展示更丰富的控制。

### 动机

不是代码项目中的每个任务都应创建新的派生工作区。

示例：

- 小修复可以在共享工作区中运行
- 三个相关任务可能有意共享一个集成分支
- 独立操作者可能直接在 `master` 上工作

### 建议的 `issues` 字段

- `projectWorkspaceId: uuid | null`
- `executionWorkspacePreference`
  - `inherit`
  - `shared_workspace`
  - `isolated_workspace`
  - `operator_branch`
  - `reuse_existing`
- `preferredExecutionWorkspaceId: uuid | null`
- `executionWorkspaceSettings`
  - 在此保留高级的按任务覆盖字段

### 规则

- 如果项目没有工作区自动化，这些字段可以全部为 null
- 如果项目有一个主工作区，任务创建应默认静默使用它
- `reuse_existing` 仅为高级选项，应针对活跃的执行工作区，而非整个工作区范围
- 没有这些字段的现有任务在迁移时应表现为 `inherit`

## 5. 执行工作区

共享或派生运行时工作区的持久记录。

这是使清理、预览、PR 和分支复用变得可处理的缺失对象。

### 动机

没有显式的 `执行工作区` 记录，Paperclip 就没有稳定的地方来附加：

- 派生的分支/worktree 身份
- 活跃预览所有权
- PR 链接
- 清理状态
- "复用此现有集成分支"行为
- 远程提供商会话身份

### 建议的新对象

`execution_workspaces`

### 建议字段

- `id`
- `companyId`
- `projectId`
- `projectWorkspaceId`
- `sourceIssueId`
- `mode`
  - `shared_workspace`
  - `isolated_workspace`
  - `operator_branch`
  - `adapter_managed`
- `strategyType`
  - `project_primary`
  - `git_worktree`
  - `adapter_managed`
- `name`
- `status`
  - `active`
  - `idle`
  - `in_review`
  - `archived`
  - `cleanup_failed`
- `cwd`
- `repoUrl`
- `baseRef`
- `branchName`
- `providerRef`
- `providerType`
  - `local_fs`
  - `git_worktree`
  - `adapter_managed`
  - `cloud_sandbox`
- `derivedFromExecutionWorkspaceId`
- `lastUsedAt`
- `openedAt`
- `closedAt`
- `cleanupEligibleAt`
- `cleanupReason`
- `metadata`
- `createdAt`
- `updatedAt`

### 说明

- `sourceIssueId` 是最初导致工作区被创建的任务，不一定是后续链接到它的唯一任务。
- 在长期分支工作流中，多个任务可能链接到同一个执行工作区。
- 对于远程执行工作区，`cwd` 可能为 null；提供商身份和工作产出链接仍使对象有用。

## 6. 任务到执行工作区链接

一个任务可能需要随时间链接到一个或多个执行工作区。

示例：

- 一个任务在共享工作区中开始，后来移到隔离工作区
- 一次失败尝试被归档，创建新的工作区
- 多个任务有意共享一个操作者分支工作区

### 建议对象

`issue_execution_workspaces`

### 建议字段

- `issueId`
- `executionWorkspaceId`
- `relationType`
  - `current`
  - `historical`
  - `preferred`
- `createdAt`
- `updatedAt`

### UI 简化

大多数任务在主 UI 中应只显示一个当前工作区。历史链接属于高级/历史视图。

## 7. 工作产出

面向用户的工作输出伞形概念。

### 动机

Paperclip 需要一个单一位置来展示：

- "这是预览"
- "这是 PR"
- "这是分支"
- "这是提交"
- "这是产物/报告/文档"

而不将任务变成适配器细节的原始倾倒。

  - `active`

  - `merged`
  - `closed`
  - `failed`
  - `archived`
- `reviewState`
  - `none`
  - `needs_board_review`
  - `approved`
  - `changes_requested`
- `isPrimary`

### 行为

- PR 作为 `type=pull_request` 存储在此
- 预览作为 `type=preview_url` 或 `runtime_service` 存储在此
- Paperclip 拥有的进程应自动更新健康/状态

- 云智能体应能在 Paperclip 不拥有执行主机的情况下创建工作产出记录

## 页面和 UI 模型

## 1. 全局导航

V1 中不要添加 `工作区` 作为顶级侧边栏项。

### 动机

这会使整个产品感觉基础设施化，即使对于不使用代码自动化的公司也是如此。

### 全局导航保持

- 仪表盘
- 收件箱
- 公司
- 智能体
- 目标
- 项目
- 任务
- 审批

工作区和工作产出应通过项目和任务详情视图展示。

## 2. 项目详情

添加项目子导航，保持规划在前、代码在后。

### 标签页

- `概览`
- `任务`
- `代码`
- `活动`

可选未来：

- `产出`

### `概览` 标签页

规划优先的摘要：

- 项目状态
- 目标
- 负责人
- 任务计数
- 顶层进度
- 最新主要工作产出摘要

### `任务` 标签页

- 默认只显示顶级任务
- 显示父任务汇总：
  - 子任务数
  - `x/y` 完成
  - 活跃的预览/PR 徽章
- 可选切换：`显示子任务`

### `代码` 标签页

这是主要的工作区配置和可见性界面。

#### 区域：`项目工作区`

列出项目的持久项目工作区。

卡片/列表列：

- 工作区名称
- 来源类型
- 路径或仓库
- 默认引用
- 主/默认徽章
- 活跃执行工作区数
- 活跃任务数
- 活跃预览数
- 远程管理时的托管类型/提供商

操作：

- `添加工作区`
- `编辑`
- `设为默认`
- `归档`

#### 区域：`执行默认值`

字段：

- `启用工作区自动化`
- `默认任务执行模式`
  - `共享工作区`
  - `隔离工作区`
  - `操作者分支`
  - `适配器默认`
- `默认代码库`
- `允许任务覆盖`

#### 区域：`预配置`

字段：

- `设置命令`
- `清理命令`
- `实现`
  - `共享工作区`
  - `Git worktree`
  - `适配器管理`
- `基础引用`
- `分支命名模板`
- `派生工作区父目录`

当选定的工作区不是 git 支持时隐藏 git 特定字段。
当选定的工作区是远程管理时隐藏本地路径特定字段。

#### 区域：`Pull Request`

字段：

- `PR 工作流`
  - `禁用`
  - `手动`
  - `智能体可以打开草稿 PR`
  - `打开 PR 需要审批`
  - `标记就绪需要审批`
- `默认基础分支`
- `PR 标题模板`
- `PR 正文模板`

#### 区域：`预览和运行时`

字段：

- `允许工作区运行时服务`
- `默认服务配置`
- `收集拥有的预览 URL`
- `跟踪外部预览 URL`

#### 区域：`清理`

字段：

- `清理模式`
  - `手动`
  - `当任务终态时`
  - `当 PR 关闭时`
  - `保留窗口后`
- `保留窗口`
- `预览活跃时保留`
- `PR 打开时保留`

## 3. 添加项目工作区流程

入口：`项目 > 代码 > 添加工作区`

### 表单字段

- `名称`
- `来源类型`
  - `本地文件夹`
  - `Git 仓库`
  - `非 git 文件夹`
  - `远程管理`
- `本地路径`
- `仓库 URL`
- `远程提供商`
- `远程工作区引用`
- `默认引用`
- `设为默认工作区`
- `设置命令`
- `清理命令`

### 行为

- 如果来源类型是非 git，隐藏分支/PR 特定设置
- 如果来源类型是 git，显示引用和可选的高级分支字段
- 如果来源类型是远程管理，显示提供商/引用字段并隐藏仅本地路径的配置
- 对于简单的独立用户，这可以是一个路径字段和一个保存按钮

## 4. 创建任务流程

任务创建默认应保持简单。

### 默认行为

如果选定的项目：

- 没有工作区自动化：不显示工作区 UI
- 有一个默认项目工作区和默认执行模式：静默继承

### 仅在相关时显示 `工作区` 部分

#### 基础字段

- `代码库`
  - 默认选择的项目工作区
- `执行模式`
  - `项目默认`
  - `共享工作区`
  - `隔离工作区`
  - `操作者分支`

#### 仅高级字段

- `复用现有执行工作区`

此下拉应仅显示选定项目工作区的活跃执行工作区，标签如：

- `dotta/integration-branch`
- `PAP-447-add-worktree-support`
- `shared primary workspace`

### 重要规则

不要默认显示包含每个可能的工作区对象的选择器。

正常流程应感觉像：

- 选择项目
- 可选地选择代码库
- 可选地选择执行模式

而不是：

- 从根目录、派生 worktree、预览和分支名称的长混合列表中选择

### 迁移规则

对于现有用户，任务创建应继续看起来相同，直到项目显式启用更丰富的工作区行为。

## 5. 任务详情

任务详情应清晰展示工作区和工作产出，但不要变成代码托管 UI。

### 页头标签

在标题/状态区域附近显示紧凑的摘要标签：

- `代码库: Web App`
- `工作区: 共享`
- `工作区: PAP-447-add-worktree-support`
- `PR: 打开`
- `预览: 健康`

### 标签页

- `评论`
- `子任务`
- `工作产出`
- `活动`

### `工作产出` 标签页

部分：

- `当前工作区`
- `预览`
- `Pull Request`
- `分支和提交`
- `产物和文档`

#### 当前工作区面板

字段：

- 工作区名称
- 模式
- 分支
- 基础引用
- 最后使用时间
- 链接任务数
- 清理状态

操作：

- `打开工作区详情`
- `标记为审查中`
- `请求清理`

#### Pull Request 卡片

字段：

- 标题
- 提供商
- 状态
- 审查状态
- 链接的分支
- 打开/就绪/合并时间戳

操作：

- `打开 PR`
- `刷新状态`
- `请求看板审查`

#### 预览卡片

字段：

- 标题
- URL
- 提供商
- 健康状态
- 所有权
- 更新时间

操作：

- `打开预览`
- `刷新`
- `归档`

## 6. 执行工作区详情

可以从项目代码标签页或任务工作产出标签页到达。

不需要在主侧边栏中。

### 部分

- 身份标识
- 源任务
- 链接的任务
- 分支/引用
- 提供商/会话身份
- 活跃运行时服务
- 预览
- PR
- 清理状态
- 事件/活动历史

### 动机

这是高级用户需要检查机制时去的地方。大多数用户在正常流程中不应需要它。

## 7. 收件箱行为

收件箱应展示可操作的工作产出事件，而非每个实现细节。

### 为以下显示收件箱项

- 任务被分配或更新
- PR 需要看板审查
- PR 被打开或标记为就绪
- 预览不健康
- 工作区清理失败
- 运行时服务失败
- 远程云智能体运行返回了需要审查的 PR 或预览

### 默认不显示

- 每个工作区心跳
- 每个分支更新
- 每个派生工作区创建

### 显示样式

如果收件箱项是关于预览或 PR 的，显示带任务上下文：

- 任务标识符和标题
- 如果是子任务则显示父任务
- 如果相关则显示工作区名称

## 8. 任务列表和看板

保持列表和看板以规划为先。

### 默认行为

- 默认显示顶级任务
- 显示子任务的父汇总
- 不要将每个子级执行细节扁平化到主看板中

### 行/卡片装饰

对于有链接工作产出的任务，显示紧凑徽章：

- `1 PR`
- `2 预览`
- `共享工作区`
- `隔离工作区`

### 可选高级过滤器

- `有 PR`
- `有预览`
- `工作区模式`
- `代码库`

## 升级和迁移计划

## 1. 产品级迁移立场

迁移必须默认静默且保持兼容。

现有用户不应被强制：

- 在能继续工作前手动创建新的工作区对象
- 重新标记旧任务
- 在基本任务流程继续运作前学习新的工作区概念

## 2. 现有项目迁移

升级时：

- 现有 `project_workspaces` 记录被保留并显示为 `项目工作区`
- 当前主工作区保持为默认代码库
- 现有项目执行工作区策略被映射到新的 `项目执行工作区策略` 界面
- 没有执行工作区策略的项目保持兼容/共享模式

## 3. 现有任务迁移

升级时：

- 现有任务默认为 `executionWorkspacePreference=inherit`
- 如果任务已有执行工作区设置，直接向前映射
- 如果任务没有显式工作区数据，保留现有行为且不强制用户可见的选择

## 4. 现有运行/运行时迁移

升级时：

- 活跃或近期的运行时服务在可行时可以回填到执行工作区历史中
- 缺失的历史不应阻断上线；向前正确性比完美的历史重建更重要

## 5. 上线用户体验

在 UI 中使用增量语言：

- `代码`
- `工作区自动化`
- `可选`
- `高级`

避免暗示用户之前"错误地"使用产品的迁移文案。

## 云部署要求

## 1. Paperclip 主机和执行主机必须解耦

Paperclip 可能运行在：

- 本地，有直接文件系统访问
- 云应用主机如 Vercel
- 混合设置，有外部作业运行器

工作区模型必须在所有三种情况下工作。

## 2. 远程执行必须支持一等工作产出报告

云智能体应能：

- 解析项目工作区
- 远程实现适配器管理的执行工作区
- 产生分支
- 打开或更新 PR
- 发出预览 URL
- 注册产物

而 Paperclip 主机本身不运行本地 git 或本地预览进程。

## 3. 仅本地假设必须是可选的

以下必须是可选的，而非必需的：

- 本地 `cwd`
- 本地 git CLI
- 主机管理的 worktree 目录
- 主机拥有的长期预览进程

## 4. 相同产品界面，不同提供商行为

UI 不应分裂为"本地模式"和"云模式"产品。

相反：

- 本地项目显示路径/git 实现细节
- 云项目显示提供商/引用细节
- 两者展示相同的高层对象：
  - 项目工作区
  - 执行工作区
  - 工作产出
  - 运行时服务或预览

## 从 Worktrunk 学到的模式

Worktrunk 是一个有用的参考点，因为它毫不掩饰地专注于基于 git-worktree 的开发者工作流。

Paperclip 不应整体复制其产品框架，但有几个值得应用的好模式。

参考：

- `https://worktrunk.dev/tips-patterns/`
- `https://github.com/max-sixty/worktrunk`

## 1. 确定性的按工作区资源

Worktrunk 将派生工作区视为可以确定性拥有的东西：

- 端口
- 本地 URL
- 数据库
- 运行时进程身份

这是 Paperclip 的一个强模式。

### 建议

执行工作区应能确定性地派生和公开：

- 预览 URL
- 端口分配
- 数据库/schema 名称
- 运行时服务复用键

这使预览和本地运行时服务在跨多个并行工作区时更可预测和更易管理。

## 2. 生命周期钩子应保持简单和显式

Worktrunk 使用实用的生命周期钩子，如面向创建/启动/移除/合并的命令。

主要教训不是构建庞大的工作流引擎。教训是给用户几个定义良好的生命周期时刻来附加自动化。

### 建议

Paperclip 应保持工作区自动化以一小组钩子为中心：

- `setup`
- `cleanup`
- 可选 `before_review`
- 可选 `after_merge` 或 `after_close`

这些应保持为项目/工作区策略关注点，而非智能体提示词约定。

## 3. 工作区状态可见性是真正的产品功能

Worktrunk 的列表/状态体验在做重要的产品工作：

- 哪些工作区存在
- 它们在什么分支上
- 它们拥有什么服务或 URL
- 它们是活跃的还是过时的

### 建议

Paperclip 应在项目 `代码` 界面中提供等价的可见性：

- 活跃的执行工作区
- 链接的任务
- 链接的 PR
- 链接的预览/运行时服务
- 清理资格

这加强了为什么 `执行工作区` 需要成为一等记录对象。

## 4. 执行工作区是运行时孤岛，不仅仅是检出

Worktrunk 最强的隐含想法之一是 worktree 不仅仅是代码。它通常拥有整个本地运行时环境。

### 建议

Paperclip 应将执行工作区视为以下的自然归属：

- 开发服务器
- 预览进程
- 沙箱凭证或提供商引用
- 分支/引用身份
- 本地或远程环境引导

这支持上面提出的 `工作产出` 模型和预览/运行时服务模型。

## 5. 机器可读的工作区状态很重要

Worktrunk 公开了可被工具和自动化消费的结构化状态。

### 建议

Paperclip 应确保执行工作区和工作产出有干净的结构化 API 界面，而非仅有 UI 表示。

这对以下很重要：

- 智能体
- CLI
- 仪表盘
- 未来的自动化和清理工具

## 6. 清理应是一等的，而非事后想法

Worktrunk 使创建/移除/合并清理成为工作流的一部分。

### 建议

Paperclip 应继续将清理策略视为核心工作区模型的一部分：

- 何时允许清理
- 什么阻断清理
- 什么被归档 vs 销毁
- 清理失败时发生什么

这验证了本计划前面提出的显式清理策略。

## 7. 不应复制的

类比也有重要的限制。

Paperclip 不应将这些 Worktrunk 假设作为通用产品规则采纳：

- 每个执行工作区都是本地 git worktree
- Paperclip 主机有直接的 shell 和文件系统访问
- 每个工作流都以合并为中心
- 每个用户都想在主导航中看到开发者工具级的工作区细节

### 产品含义

Paperclip 应借用 Worktrunk 的好执行模式，同时保持更广泛的 Paperclip 模型：

- 项目规划工作
- 工作区定义工作发生的地方
- 工作产出定义产出了什么
- git worktree 仍是一种实现策略，而非产品本身

## 行为规则

## 1. 清理不得依赖智能体记住 `in_review`

智能体仍可使用 `in_review`，但清理行为必须由策略和观察到的状态管控。

### 在以下任一条件为真时保持执行工作区存活

- 一个链接的任务是非终态的
- 一个链接的 PR 是打开的
- 一个链接的预览/运行时服务是活跃的
- 工作区仍在保留窗口内

### 先隐藏再激进删除

归档或空闲的工作区应在硬清理前从默认列表中隐藏。

## 2. 多个任务可以有意共享一个执行工作区

这是 Paperclip 如何支持：

- 独立开发者在共享分支上
- 操作者集成分支
- 相关功能批量进入一个 PR

这是不强制 1 任务 = 1 工作区 = 1 PR 的关键原因。

## 3. 隔离任务工作区保持可选

即使在 git 密集的项目中，隔离工作区也应是可选的。

共享模式有效的示例：

- 小 bug 修复
- 无分支的原型制作
- 非 git 项目
- 单用户本地工作流

## 4. PR 策略属于 git 支持的工作区策略

PR 自动化决策应在项目/工作区策略层做出。

任务应只：

- 展示结果 PR
- 路由审批/审查请求
- 显示状态和审查状态

## 5. 工作产出是面向用户的统一者

预览、PR、提交和产物都应通过一个一致的任务级功能可发现。

这使 Paperclip 专注于协调和可见性，而非将输出分散在许多隐藏子系统中。

## 推荐的实现顺序

## 阶段 1：在 UI 中澄清当前对象

1. 展示 `项目 > 代码` 标签页
2. 在那里显示现有的项目工作区
3. 用修订后的文案重新启用项目级执行工作区策略
4. 保持任务创建简单，使用继承的默认值

## 阶段 2：添加显式的执行工作区记录

1. 添加 `execution_workspaces`
2. 将运行、任务、预览和 PR 链接到它
3. 添加简单的执行工作区详情页面
4. 使 `cwd` 可选，确保从第一天起支持提供商管理的远程工作区

## 阶段 3：添加工作产出模型

1. 添加 `issue_work_products`
2. 摄入 PR、预览、分支、提交
3. 添加任务 `工作产出` 标签页
4. 为可操作的工作产出状态变化添加收件箱项
5. 支持远程智能体创建的 PR/预览报告而无需本地所有权

## 阶段 4：添加高级复用和清理工作流

1. 添加 `复用现有执行工作区`
2. 添加清理生命周期 UI
3. 添加操作者分支工作流快捷方式
4. 添加更丰富的外部预览收集
5. 添加迁移工具/回填以改善现有用户的连续性

## 为什么这个模型是正确的

这个模型保持了产品的平衡：

- 对独立用户足够简单
- 对真正的工程团队足够强大
- 对非 git 项目足够灵活
- 足够显式以管控 PR 和预览

最重要的是，它保持了抽象的干净：

- 项目规划工作
- 项目工作区定义持久的代码库
- 执行工作区定义工作在哪里运行
- 工作产出定义工作产出了什么
- PR 保持为输出，而非核心任务模型

它还使上线实际可行：

- 现有用户可以升级而无工作流破坏
- 本地优先的安装保持简单
- 云托管的 Paperclip 部署保持一等

这比任何极端都更适合 Paperclip：

- 隐藏工作区行为直到没人理解它
- 或者让整个应用围绕代码托管机制转
