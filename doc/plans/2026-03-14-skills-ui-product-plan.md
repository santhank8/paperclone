状态：已提议

## 1. 目的

本文档定义了 Paperclip 中技能管理的产品和 UI 计划。

目标是让技能在网站中可理解和可管理，同时不假装所有适配器的行为方式相同。

本计划假设：

- `SKILL.md` 保持 Agent Skills 兼容
- `skills.sh` 兼容性是 V1 需求
- Paperclip 公司导入/导出可以将技能作为包内容包含
- 适配器可能支持持久技能同步、临时技能挂载、只读技能发现，或者完全没有技能集成

## 2. 当前状态

在 `AgentDetail` 上已有一个初版的智能体级技能同步 UI。

目前支持：

- 加载适配器技能同步状态
- 清晰显示不支持的适配器
- 以复选框形式显示已管理的技能
- 单独显示外部技能
- 为实现了新 API 的适配器同步期望的技能

当前限制：

1. 没有公司级技能库 UI。
2. 网站中没有技能的包导入流程。
3. 没有区分技能包管理和按智能体的技能关联。
4. 没有多智能体的期望 vs 实际视图。
5. 当前 UI 是面向适配器同步的，而非面向包的。
6. 不支持的适配器可以安全降级，但不够优雅。

## 2.1 V1 决策

对于 V1，本计划假设以下产品决策已经做出：

1. `skills.sh` 兼容性是必需的。
2. `AGENTS.md` 中的智能体到技能关联使用短名或 slug。
3. 公司技能和智能体技能关联是不同的概念。
4. 智能体技能应移到自己的标签页，而非放在配置内部。
5. 公司导入/导出最终应能往返传递技能包和智能体技能关联。

## 3. 产品原则

1. 技能首先是公司资产，其次才是智能体关联。
2. 包管理和适配器同步是不同的关注点，不应在同一屏幕中混淆。
3. UI 必须始终如实告知 Paperclip 所知的情况：
   - Paperclip 中的期望状态
   - 适配器报告的实际状态
   - 适配器是否能调和两者
4. Agent Skills 兼容性必须在产品模型中保持可见。
5. 智能体到技能的关联应尽可能使用人类可读的短名。
6. 不支持的适配器仍应有有用的 UI，而非死胡同。

## 4. 用户模型

Paperclip 应在两个范围处理技能：

### 4.1 公司技能

这些是公司已知的可复用技能。

示例：

- 从 GitHub 仓库导入
- 从本地文件夹添加
- 从 `skills.sh` 兼容的仓库安装
- 后续在 Paperclip 中本地创建

这些应具有：

- 名称
- 描述
- slug 或包标识
- 来源/出处
- 信任级别
- 兼容性状态

### 4.2 智能体技能

这些是特定智能体的技能关联。

每个关联应具有：

- 短名
- Paperclip 中的期望状态
- 可读时适配器中的实际状态
- 同步状态
- 来源

智能体关联通常应通过短名或 slug 引用技能，例如：

- `review`
- `react-best-practices`

而非嘈杂的相对文件路径。

## 4.3 主要用户任务

UI 应干净地支持这些任务：

1. "展示这个公司有哪些技能。"
2. "从 GitHub 或本地文件夹导入一个技能。"
3. "查看一个技能是否安全、兼容，以及谁在使用它。"
4. "将技能关联到一个智能体。"
5. "查看适配器是否实际拥有那些技能。"
6. "调和期望 vs 实际技能状态。"
7. "了解 Paperclip 知道什么 vs 适配器知道什么。"

## 5. 核心 UI 界面

产品应有两个主要的技能界面。

### 5.1 公司技能页面

添加公司级页面，可能是：

- `/companies/:companyId/skills`

目的：

- 管理公司技能库
- 导入和检查技能包
- 了解出处和信任
- 查看哪些智能体使用哪些技能

#### 路由

- `/companies/:companyId/skills`

#### 主要操作

- 导入技能
- 检查技能
- 关联到智能体
- 从智能体取消关联
- 后续导出选定的技能

#### 空状态

当公司没有已管理的技能时：

- 解释什么是技能
- 解释 `skills.sh` / Agent Skills 兼容性
- 提供 `从 GitHub 导入` 和 `从文件夹导入`
- 可选地将适配器发现的技能显示为次要的"尚未管理"部分

#### A. 技能库列表

每个技能行应显示：

- 名称
- 简短描述
- 来源徽章
- 信任徽章
- 兼容性徽章
- 关联的智能体数量

建议的来源状态：

- local
- github
- imported package
- external reference
- adapter-discovered only

建议的兼容性状态：

- compatible
- paperclip-extension
- unknown
- invalid

建议的信任状态：

- markdown-only
- assets
- scripts/executables

建议的列表功能：

- 按名称或 slug 搜索
- 按来源过滤
- 按信任级别过滤
- 按使用情况过滤
- 按名称、最近导入、使用次数排序

#### B. 导入操作

允许：

- 从本地文件夹导入
- 从 GitHub URL 导入
- 从直接 URL 导入

未来：

- 从 `companies.sh` 安装
- 从 `skills.sh` 安装

V1 要求：

- 从 `skills.sh` 兼容源导入应无需 Paperclip 特定的包布局即可工作

#### C. 技能详情抽屉或页面

每个技能应有详情视图，显示：

- 渲染的 `SKILL.md`
- 包来源和固定版本
- 包含的文件
- 信任和许可警告
- 谁在使用它
- 适配器兼容性说明

推荐路由：

- `/companies/:companyId/skills/:skillId`

推荐部分：

- 概览
- 内容
- 使用情况
- 来源
- 信任/许可

#### D. 使用情况视图

每个公司技能应显示哪些智能体使用它。

建议列：

- 智能体
- 期望状态
- 实际状态
- 适配器
- 同步模式
- 最后同步状态

### 5.2 智能体技能标签页

保留并演进现有的 `AgentDetail` 技能同步 UI，但将其移出配置。

目的：

- 将公司技能关联/取消关联到一个智能体
- 检查该智能体的适配器实际状态
- 调和期望 vs 实际状态
- 保持关联格式可读并与 `AGENTS.md` 对齐

#### 路由

- `/agents/:agentId/skills`

#### 智能体标签页

预期的智能体级标签页模型变为：

- `dashboard`
- `configuration`
- `skills`
- `runs`

这优于将技能隐藏在配置内部，因为：

- 技能不仅仅是适配器配置
- 技能需要自己的同步/状态语言
- 技能是可复用的公司资产，不仅仅是一个智能体字段
- 屏幕需要为期望 vs 实际状态、警告和外部技能采纳留出空间

#### 标签页布局

`技能` 标签页应有三个堆叠部分：

1. 摘要
2. 已管理的技能
3. 外部/发现的技能

摘要应显示：

- 适配器同步支持
- 同步模式
- 已管理技能数量
- 外部技能数量
- 漂移或警告计数

#### A. 期望的技能

显示关联到智能体的公司管理技能。

每行应显示：

- 技能名称
- 短名
- 同步状态
- 来源
- 如可用则显示最后的适配器观察

每行应支持：

- 启用/禁用
- 打开技能详情
- 查看来源徽章
- 查看同步徽章

#### B. 外部或发现的技能

显示适配器报告的但非公司管理的技能。

这很重要，因为 Codex 和类似适配器可能已有 Paperclip 未安装的本地技能。

这些应清楚标记为：

- 外部
- 非 Paperclip 管理

每个外部行应支持：

- 检查
- 后续采纳到公司库
- 如适当后续作为已管理技能关联

#### C. 同步控制

支持：

- 同步
- 重置草稿
- 取消关联

未来：

- 将外部技能导入公司库
- 将临时本地技能提升为已管理的公司技能

推荐的底部操作：

- `同步技能`
- `重置`
- `刷新适配器状态`

## 6. UI 中的技能状态模型

每个技能关联应有一个面向用户的状态。

建议的状态：

- `in_sync`
- `desired_only`
- `external`
- `drifted`
- `unmanaged`
- `unknown`

定义：

- `in_sync`：期望和实际匹配
- `desired_only`：Paperclip 想要它，适配器还未显示它
- `external`：适配器有它但 Paperclip 不管理它
- `drifted`：适配器有冲突或意外的版本/位置
- `unmanaged`：适配器不支持同步，Paperclip 仅跟踪期望状态
- `unknown`：适配器读取失败或状态不可信

建议的徽章文案：

- `已同步`
- `需要同步`
- `外部`
- `已漂移`
- `未管理`
- `未知`

## 7. 适配器展示规则

UI 不应以相同方式描述所有适配器。

### 7.1 持久化适配器

示例：

- Codex 本地

措辞：

- 已安装
- 已同步到适配器主目录
- 检测到外部技能

### 7.2 临时适配器

示例：

- Claude 本地

措辞：

- 将在下次运行时挂载
- 有效运行时技能
- 未全局安装

### 7.3 不支持的适配器

措辞：

- 此适配器尚未实现技能同步
- Paperclip 仍可跟踪期望的技能
- 适配器实际状态不可用

此状态仍应允许：

- 将公司技能作为期望状态关联到智能体
- 导出/导入这些期望的关联

## 7.4 只读适配器

某些适配器可能能列出技能但不能修改它们。

措辞：

- Paperclip 可以查看适配器技能
- 此适配器不支持应用更改
- 可以跟踪期望状态，但调和是手动的

## 8. 信息架构

推荐的导航：

- 公司导航添加 `技能`
- 智能体详情添加 `技能` 作为独立标签页
- 公司技能详情在公司库上线时获得自己的路由

推荐的分离：

- 公司技能页面回答："我们有哪些技能？"
- 智能体技能标签页回答："这个智能体使用什么，是否已同步？"

## 8.1 建议的路由映射

- `/companies/:companyId/skills`
- `/companies/:companyId/skills/:skillId`
- `/agents/:agentId/skills`

## 8.2 导航和发现

推荐的入口点：

- 公司侧边栏：`技能`
- 智能体页面标签页：`技能`
- 公司导入预览：后续将导入的技能链接到公司技能页面
- 智能体技能行：链接到公司技能详情

## 9. 导入/导出集成

技能 UI 和包可移植性应在公司技能库中交汇。

导入行为：

- 导入包含 `SKILL.md` 内容的公司包应创建或更新公司技能
- 智能体关联应主要来自 `AGENTS.md` 短名关联
- `.paperclip.yaml` 可以添加 Paperclip 特定的保真度，但不应替代基础短名关联模型
- 引用的第三方技能应保持出处可见

导出行为：

- 导出公司时应在选定时包含公司管理的技能
- `AGENTS.md` 应按短名或 slug 发出技能关联
- `.paperclip.yaml` 可以在需要时后续添加 Paperclip 特定的技能保真度，但普通的智能体到技能关联不应需要它
- 仅适配器的外部技能不应被静默导出为公司管理的技能

## 9.1 导入工作流

V1 工作流应支持：

1. 从本地文件夹导入一个或多个技能
2. 从 GitHub 仓库导入一个或多个技能
3. 导入包含技能的公司包
4. 将导入的技能关联到一个或多个智能体

技能的导入预览应显示：

- 发现的技能
- 来源和固定版本
- 信任级别
- 许可警告
- 现有公司技能将被创建、更新还是跳过

## 9.2 导出工作流

V1 应支持：

1. 导出公司时在选定时包含已管理的技能
2. 导出智能体，其 `AGENTS.md` 包含短名技能关联
3. 为每个 `SKILL.md` 保持 Agent Skills 兼容性

V1 范围外：

- 自动将仅适配器的外部技能导出为已管理的包

## 10. 数据和 API 形态

本计划意味着后端概念的清晰拆分。

### 10.1 公司技能记录

Paperclip 应有公司范围的技能模型或已管理包模型，表示：

- 身份标识
- 来源
- 文件
- 出处
- 信任和许可元数据

### 10.2 智能体技能关联

Paperclip 应单独存储：

- 智能体 ID
- 技能标识
- 期望的启用状态
- 后续可选的排序或元数据

### 10.3 适配器同步快照

适配器读取应返回：

- 支持标志
- 同步模式
- 条目
- 警告
- 期望的技能

这已以粗略形式存在，应作为 UI 的基础。

### 10.4 UI 所需的 API

完整 UI 意味着这些 API 界面：

- 列出公司管理的技能
- 从路径/URL/GitHub 导入公司技能
- 获取一个公司技能详情
- 列出使用给定技能的智能体
- 为智能体关联/取消关联公司技能
- 列出智能体的适配器同步快照
- 为智能体应用期望的技能

现有的智能体级技能同步 API 可以保留作为智能体标签页的基础。
公司级库 API 仍需设计和实现。

## 13. 权限和治理

建议的 V1 策略：

- 看板用户可以管理公司技能
- 看板用户可以将技能关联到智能体
- 智能体本身默认不修改公司技能库
- 后续，某些智能体可能获得技能关联或同步的范围权限

## 14. UI 阶段

### 阶段 A：稳定当前智能体技能同步 UI

目标：

- 将技能移到 `AgentDetail` 标签页
- 改进状态措辞
- 在不支持的适配器上也支持仅期望状态
- 优化持久化 vs 临时适配器的文案

### 阶段 B：添加公司技能页面

目标：

- 公司级技能库
- 从 GitHub/本地文件夹导入
- 基础详情视图
- 按智能体的使用计数
- `skills.sh` 兼容的导入路径

### 阶段 C：将技能连接到可移植性

目标：

- 导入公司包创建公司技能
- 导出选定技能干净地工作
- 智能体关联主要通过 `AGENTS.md` 短名往返传递

### 阶段 D：外部技能采纳流程

目标：

- 检测适配器外部技能
- 在可能时允许将它们导入到公司管理状态
- 使出处明确

### 阶段 E：高级同步和漂移 UX

目标：

- 期望 vs 实际差异比较
- 漂移解决操作
- 多智能体技能使用和同步报表

## 15. 设计风险

1. 在智能体页面上过载包管理会使功能令人困惑。
2. 将不支持的适配器视为损坏而非未管理会使产品感觉不一致。
3. 在没有清晰标签的情况下混合适配器发现的外部技能和公司管理的技能会侵蚀信任。
4. 如果公司技能记录不存在，导入/导出和 UI 将保持松散耦合，往返保真度将保持薄弱。
5. 如果智能体技能关联是基于路径而非基于短名的，格式将感觉过于技术化和过于 Paperclip 特定。

## 16. 建议

下一个产品步骤应是：

1. 将技能从智能体配置中移出，放入专用的 `技能` 标签页
2. 添加专用的公司级 `技能` 页面作为库和包管理界面
3. 使公司导入/导出面向该公司技能库，而非直接面向智能体页面
4. 通过清晰分离以下内容在 UI 中保持适配器感知的真实性：
   - 期望的
   - 实际的
   - 外部的
   - 未管理的
5. 在 `AGENTS.md` 中保持智能体到技能的关联基于短名

这给 Paperclip 一个连贯的技能故事，而非将包管理、适配器同步和智能体配置强制塞入同一屏幕。
