# Paperclip — 产品定义

## 它是什么

Paperclip 是自主 AI 公司的控制平面 (control plane)。一个 Paperclip 实例可以运行多家公司。**公司 (company)** 是一等对象 (first-order object)。

## 核心概念

### 公司 (Company)

每个公司拥有：

- 一个 **目标 (goal)** —— 它存在的理由（例如：“在 3 个月内创建一个做到 100 万美元 MRR 的第一大 AI 笔记应用”）
- **员工 (Employees)** —— 每一名员工都是一个 AI 代理 (Agent)
- **组织架构 (Org structure)** —— 谁向谁汇报（汇报线）
- **收入与支出 (Revenue & expenses)** —— 在公司级别进行跟踪
- **任务层级 (Task hierarchy)** —— 所有的工作最终都能追溯到公司的目标

### 员工与代理 (Employees & Agents)

每一次提到的员工就是代理。当你创建一家公司时，你首先定义 CEO，然后以此为基础构建整个组织。

每个员工（代理）拥有：

- **适配器类型与配置 (Adapter type + config)** —— 该代理如何运行以及什么定义了它的身份/行为。这是特定于适配器的（例如，一个 OpenClaw 代理可能会使用 `SOUL.md` 和 `HEARTBEAT.md` 文件；一个 Claude Code 代理可能会使用 `CLAUDE.md`；一个基础脚本可能只使用命令行参数）。Paperclip 不规定格式——这是由适配器决定的。
- **角色与汇报关系 (Role & reporting)** —— 他们的头衔、他们向谁汇报、谁向他们汇报。
- **能力描述 (Capabilities description)** —— 简短描述该代理擅长什么以及什么时候该找他们（这有助于其他代理发现谁可以帮忙解决特定问题）。

示例：CEO 代理的适配器配置告诉它在每次心跳 (heartbeat) 时，“审查你的高管正在做什么，检查公司指标，如有必要进行重新确定优先级，并分配新的战略举措”。工程师的配置告诉它“检查分配的任务，选择优先级最高的任务开始工作”。

然后你定义谁向 CEO 汇报：例如管理程序员的 CTO，管理营销团队的 CMO，等等。树形结构中的每个代理都有自己独立的适配器配置。

### 代理执行 (Agent Execution)

运行代理心跳由两种基本模式：

1. **运行命令 (Run a command)** —— Paperclip 启动一个进程（Shell 命令、Python 脚本等）并对其进行跟踪。这里的心跳含义是“执行此命令并监控它”。
2. **触发并忘记的请求 (Fire and forget a request)** —— Paperclip 发出一个 Webhook/API 调用给外部运行的代理。这里的心跳含义是“通知这个代理醒来”。（OpenClaw 的钩子就是这种工作方式）。

我们提供合理的默认值——例如默认代理能够根据你的配置调用 Claude Code 或 Codex，记住会话 ID，或是运行基本脚本。但是你可以插入任何你想要的东西。

### 任务管理 (Task Management)

任务管理是分层的。在任何时刻，每一项工作都必须通过父任务链追溯到公司的顶层目标：

```text
我正在研究 Granola 使用的 Facebook 广告（当前任务）
  因为 → 我需要为我们的软件创建 Facebook 广告（父任务）
    因为 → 我需要通过新注册增加 100 个用户（父任务）
      因为 → 我需要在这周将收入提高到 $2,000（父任务）
        因为 → ...
          因为 → 我们正构建在 3 个月内达到 100 万美元 MRR 的第一 AI 笔记应用
```

任务具有从属关系（parentage）。每个任务都是为服务其父任务而存在的，一路向上直至公司总目标。这就是保持自主代理步调一致的方法——他们总是能回答“我为什么要做这件事？”

## 原则

1. **对你如何运行代理不持偏见。** 你的代理可以是 OpenClaw 机器人、Python 脚本、Node 脚本、Claude Code 会话或者 Codex 实例——我们不在乎。Paperclip 定义了通信的控制平面并提供了处理心跳的基础设施工具。它不强制要求任何特定的代理运行时 (runtime)。

2. **公司是组织的单位。** 所有事物都存在于一个公司之下。一个 Paperclip 实例，多个公司。

3. **适配器配置定义代理。** 每个代理都有一个适配器类型和配置，这控制了其身份和行为。最基本的契约仅仅是“它是可调用的 (be callable)”。

4. **所有的工作都追溯到目标。** 具有层级关系的任务管理意味着没有什么是孤立存在的。如果你无法解释一项任务对公司目标为什么重要，它就不应该存在。

5. **控制平面，而非执行平面。** Paperclip 负责编排 orchestrates。代理在它们能运行的任何地方运行并“打电话回家”。

## 用户流程（理想场景）

1. 打开 Paperclip，创建一个新公司
2. 定义公司目标：“创建第一大的 AI 笔记应用，在 3 个月内达到 100 万美元 MRR”
3. 创建 CEO
   - 选择一个适配器（例如用于 Claude Code 的 process 适配器，用于 OpenClaw 的 HTTP 适配器）
   - 配置该适配器（代理身份定义、循环行为、执行设置）
   - CEO 提出战略分解 → 董事会（用户）批准
4. 定义向 CEO 汇报的人员：CTO、CMO、CFO 等。
   - 每个人都获得独立的适配器配置和角色定义
5. 定义向他们汇报的人员：CTO 手下的工程师，CMO 手下的营销人员等。
6. 设定预算，定义最初的战略任务
7. 点击开始 —— 代理们开始他们的心跳，公司正式运转

## 指南

Paperclip 必须支持两种运行模式：

- `local_trusted` (默认): 无登录摩擦的单用户本地受信任部署
- `authenticated`: 需要登录的模式，支持私有网络和公共部署暴露策略

规范化的模式设计以及命令期望位于 `doc/DEPLOYMENT-MODES.md` 文档中。
