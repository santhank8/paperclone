# 任务管理数据模型

有关 Paperclip 中任务跟踪如何工作的参考。描述实体及其
关系以及管理任务生命周期的规则。编写为目标模型
——其中一些已经实施，一些是期望的。

---

## 实体层次结构

```
Workspace
  Initiatives          (roadmap-level objectives, span quarters)
    Projects           (time-bound deliverables, can span teams)
      Milestones       (stages within a project)
        Issues         (units of work, the core entity)
          Sub-issues   (broken-down work under a parent issue)
```

一切都流了下来。倡议包含项目；一个项目包含
里程碑和问题；一个问题可以有子问题。每个级别都会增加
粒度。

---

## 工作流程状态问题状态**不是**一个平面枚举。这是一组特定于团队的命名状态，
每个都属于这些固定**类别**之一：

|类别 |目的|示例状态 |
| ------------- | ---------------------------- | ------------------------------- |
| **分类** |传入，需要审核 |分流 |
| **积压** |已接受，尚未准备好工作 |积压，冰箱|
| **未启动** |准备好但尚未开始 |待办事项，准备就绪 |
| **开始** |积极工作 |进行中、审查中、质量检查中 |
| **已完成** |完成 |完成，已发货 |
| **取消** |被拒绝或放弃|已取消、无法修复、重复 |

### 规则

- 每个团队在这些类别中定义自己的工作流程状态
- 团队每个类别必须至少有一个州（分类是可选的）
- 可以在任何类别中添加自定义状态（例如“已开始”下的“审核中”）
- 类别是固定且有序的——您可以在类别内对状态重新排序
  但不是类别本身
- 新问题默认为团队的第一个 Backlog 状态
- 将问题移至已启动状态自动设置 `startedAt`；已完成的套装
  `completedAt`；取消套 `cancelledAt`
- 将问题标记为重复会自动将其移至“已取消”状态

### 工作流程状态字段

|领域 |类型 |笔记|
| ------------- | ------- | ----------------------------------------------------------------------------- |
| `id` | uuid|                                                                               |
| `name` |字符串|显示名称，例如“回顾中”|
| `type` |枚举 |其中之一：`triage`、`backlog`、`unstarted`、`started`、`completed`、`cancelled` |
| `color` |字符串|十六进制颜色 |
| `description` |字符串|可选指导文本 |
| `position` |浮动|类别内订购 |
| `teamId` | uuid FK | UUID每个州属于一支球队 |

## 优先级

固定的、不可定制的数字刻度：

|价值|标签|笔记|
| ----- | ----------- | -------------------------------------- |
| 0 |无优先权 |默认。在优先视图中排在最后。 |
| 1 |紧急|可以立即触发通知 |
| 2 |高|                                        |
| 3 |中等|                                        |
| 4 |低|                                        |规模故意较小且固定。使用标签进行附加
分类而不是添加更多优先级。

## 团队

团队是主要的组织单位。几乎所有的事情都限定在一个范围内
团队。

|领域 |类型 |笔记|
| ------------- | ------ | -------------------------------------------------------------- |
| `id` | uuid|                                                                |
| `name` |字符串|例如“工程”|
| `key` |字符串|短大写前缀，例如“英语”。用于问题标识符。 |
| `description` |字符串|                                                                |

### 团队范围界定

- 每个问题都属于一个团队
- 工作流程状态是每个团队的
- 标签可以是团队范围或工作空间范围的
- 项目可以跨越多个团队

在我们的背景下（人工智能公司），团队映射到职能领域。各智能体报告
基于角色的团队。

## 项目

项目将问题分组，以实现特定的、有时限的交付成果。他们可以跨越
多个团队。

|领域 |类型 |笔记|
| ------------- | --------- | ------------------------------------------------------------- |
| `id` | uuid|                                                               |
| `name` |字符串|                                                               |
| `description` |文字|                                                               |
| `summary` |字符串|简短简介|
| `status` |枚举 | `backlog`、`planned`、`in_progress`、`completed`、`cancelled` |
| `leadId` | uuid FK | UUID单一所有者问责制 |
| `startDate` |日期 |                                                               |
| `targetDate` |日期 |                                                               |
| `createdAt` |时间戳|                                                               |
| `updatedAt` |时间戳|                                                               |

### 规则

- 一个问题最多属于一个项目
- 项目状态**手动**更新（不是从问题状态自动得出）
- 项目可以包含文档（规格、简介）作为链接实体

## 标签/标签

标签提供分类标记。它们存在于两个范围：

- **工作区标签** -- 所有团队均可使用
- **团队标签** -- 仅限于特定团队

|领域 |类型 |笔记|
| ------------- | -------------- | ------------------------------- |
| `id` | uuid|                                 |
| `name` |字符串|                                 |
| `color` |字符串|十六进制颜色 |
| `description` |字符串|情境指导 |
| `teamId` | uuid FK | UUID工作区级别标签为 Null |
| `groupId` | uuid FK（自身）|用于分组的父标签 |

### 标签组

标签可以组织为一层嵌套（组 -> 标签）：

- 一组内的标签在某个问题上**互斥**（只能有一个）
  各组申请）
- 组不能包含其他组（仅限单嵌套级别）
- 示例：组“Type”包含标签“Bug”、“Feature”、“Chore”——一个问题
  最多获得一个

### 问题标签连接

通过 `issue_labels` 联接表进行多对多：

|领域 |类型 |
| --------- | ------- |
| `issueId` | uuid FK | UUID
| `labelId` | uuid FK | UUID

## 问题关系/依赖关系

问题之间的四种关系类型：

|类型 |意义|行为 |
| ------------ | -------------------------------- | --------------------------------------------- |
| `related` |一般连接 |信息链接|
| `blocks` |这个问题阻止了另一个问题 |带有标志的已阻止问题 |
| `blocked_by` |此问题已被另一个问题阻止 |块的逆 |
| `duplicate` |这个问题与另一个问题重复|自动将重复项移至“已取消”状态 |

### IssueRelation 字段|领域 |类型 |笔记|
| ---------------- | ------- | ---------------------------------------------- |
| `id` | uuid|                                                |
| `type` |枚举 | `related`、`blocks`、`blocked_by`、`duplicate` |
| `issueId` | uuid FK | UUID来源问题 |
| `relatedIssueId` | uuid FK | UUID目标问题|

### 规则

- 当阻塞问题得到解决后，关系就变成信息性的（标志
  变绿）
- 重复是单向的（您标记重复，而不是规范）
- 阻塞在系统级别**不可传递**（A 阻塞 B，B 阻塞 C
  不会自动阻止 A->C)

## 受让人

**单受让人模型**设计而成。

- 每个问题一次最多有一名受让人
- 这是经过深思熟虑的：明确的所有权可以防止责任分散
- 对于涉及多人的协作工作，请使用**子问题**
  不同的受让人

在我们的上下文中，智能体人是受让人。关于`assigneeId` FK的问题
指向 `agents` 表。

---

## 子问题（父/子）

问题支持父/子嵌套。

- 在问题上设置 `parentId` 使其成为子问题
- 子问题本身可以有子问题（多层嵌套）
- 子问题在创建时从其父问题继承**项目**
  时间（不追溯），但不包括团队、标签或受让人

### 自动关闭

- **子问题自动关闭**：当父问题完成时，剩余的子问题
  自动完成

### 转换

- 可以重新调整现有问题的父级（添加或删除 `parentId`）
- 具有许多子问题的父问题可以“升级”到项目

## 标识符

问题使用人类可读的标识符：`{TEAM_KEY}-{NUMBER}`

- 团队密钥：每个团队设置的短大写字符串（例如“ENG”、“DES”）
- Number：每队自动递增的整数
- 示例：`ENG-123`、`DES-45`、`OPS-7`
- 如果问题在团队之间转移，它会获得一个新的标识符，而旧的标识符将被保留
  保存在`previousIdentifiers`

这比 UUID 对于人类交流来说要好得多。人们说“抓住 ENG-42”
不是“抓住 7f3a...”。

## 实体关系

```
Team (1) ----< (many) Issue
Team (1) ----< (many) WorkflowState
Team (1) ----< (many) Label (team-scoped)

Issue (many) >---- (1) WorkflowState
Issue (many) >---- (0..1) Assignee (Agent)
Issue (many) >---- (0..1) Project
Issue (many) >---- (0..1) Milestone
Issue (many) >---- (0..1) Parent Issue
Issue (1) ----< (many) Sub-issues
Issue (many) >---< (many) Labels         (via issue_labels)
Issue (many) >---< (many) Issue Relations (via issue_relations)
Issue (1) ----< (many) Comments

Project (many) >---- (0..1) Lead (Agent)
Project (1) ----< (many) Milestones
Project (1) ----< (many) Issues

Initiative (many) >---< (many) Projects  (via initiative_projects)
Initiative (many) >---- (1) Owner (Agent)
```

## 实施优先级

推荐的构建顺序，最高值优先：

### 高价值

1. **队伍** -- `teams` 桌 + `teamId` FK 问题。基金会
   人类可读的标识符 (`ENG-123`) 和每个团队的工作流程状态。大多数
   其他功能取决于团队范围，因此首先构建此功能。
2. **工作流程状态** -- `workflow_states` 表 + `stateId` FK 问题。
   每个团队的自定义工作流程以及基于类别的状态转换。
3. **标签** -- `labels` + `issue_labels` 表。分类
   （错误/功能/琐事、区域标签等）而不污染状态字段。
4. **问题关系** -- `issue_relations` 表。阻塞/被阻塞是
   对于智能体协调至关重要（智能体 A 在智能体 B 完成之前无法启动）。
5. **子问题** -- `parentId` 在 `issues` 上自 FK。让特工崩溃
   大任务。
6. **注释** -- `comments` 表。智能体需要就问题进行沟通
   而不覆盖描述。

### 中等价值

7. **转换时间戳** -- `startedAt`、`completedAt`、`cancelledAt` 上
   问题，根据工作流状态变化自动设置。启用速度跟踪和 SLA
   测量。

### 较低优先级（稍后）

8. **里程碑**——一旦项目变得复杂到需要阶段时就很有用。
9. **倡议**——一旦我们有多个服务于共同目标的项目就很有用
   战略目标。
10. **估计**——当我们想要测量吞吐量并预测时很有用
    容量。
