# Paperclip 规格

Paperclip 控制平面的目标规范。动态文档——在规格面试期间逐步更新。

---

## 1. 公司模式[草案]

公司是一阶对象。一个 Paperclip 实例运行多个公司。公司没有独立的“目标”字段 - 它的方向是由其一组计划定义的（请参阅任务层次结构映射）。

### 字段（草稿）

|领域 |类型 |笔记|
| ----------- | ------------- | --------------------------------- |
| `id` | uuid|主键 |
| `name` |字符串|公司名称 |
| `createdAt` |时间戳|                                   |
| `updatedAt` |时间戳|                                   |

### 董事会治理[草案]

每家公司都有一个**董事会**来管理高影响力的决策。董事会是人类监督层。

**V1：单人板。** 一名操作员。

#### 董事会批准门 (V1)

- 雇用新特工（创建新特工）
- CEO 的初始战略分解（CEO 提议，董事会在执行开始前批准）
- [待定：其他治理控制行动——目标改变、解雇特工？]

#### 董事会权力（始终可用）

董事会始终可以**不受限制地访问**整个系统：

- **设置和修改公司预算** — 董事会设置顶级代币/LLM 成本预算
- **暂停/恢复任何智能体** — 立即停止智能体的心跳
- **暂停/恢复任何工作项目** — 暂停任务、项目、子任务树、里程碑。暂停的项目不会被智能体拾取。
- **完整的项目管理访问** — 通过 UI 创建、编辑、评论、修改、删除、重新分配任何任务/项目/里程碑
- **推翻任何智能体决定** — 重新分配任务、更改优先级、修改描述
- **手动更改任何级别的任何预算**

董事会不仅仅是一个批准门——它还是一个实时控制界面。人类可以随时在任何级别进行干预。

#### 预算授权

董事会制定公司级预算。 CEO 可以为其下级智能体设置预算，每个经理智能体都可以为其报告执行相同的操作。这种级联预算授权在实践中如何运作尚待确定，但权限结构支持它。董事会可以手动覆盖任何级别的任何预算。

**未来的治理模型**（不是 V1）：

- 招聘预算（每月 X 美元以内自动批准招聘）
- 多人董事会
- 授权（首席执行官可以在限制范围内聘用）

### 开放问题

- 外部收入/支出跟踪——未来的插件。 Token/LLM 成本预算是核心。
- 公司级别的设置和配置？
- 公司生命周期（暂停、存档、删除）？
- 除了招聘和首席执行官战略批准之外，还存在哪些治理控制措施？

---## 2. 智能体模型[草案]

每个员工都是智能体人。智能体人是劳动力。

### 智能体身份（适配器级）

SOUL.md（身份/任务）和 HEARTBEAT.md（循环定义）等概念**不是 Paperclip 协议的一部分**。它们是特定于适配器的配置。例如，OpenClaw 适配器可能使用 SOUL.md 和 HEARTBEAT.md 文件。 Claude Code 适配器可能使用 CLAUDE.md。裸 Python 脚本可能使用命令行参数。

Paperclip 没有规定智能体如何定义其身份或行为。它提供控制平面；适配器定义了智能体的内部工作方式。

### 智能体配置[草案]

每个智能体都有一个 **适配器类型** 和一个 **特定于适配器的配置 blob**。适配器定义存在哪些配置字段。

#### Paperclip 协议（Paperclip 所知道的）

在协议级别，Paperclip 跟踪：

- 智能体身份（ID、姓名、角色、头衔）
- 组织职位（他们向谁报告，谁向他们报告）
- 适配器类型+适配器配置
- 状态（活动、暂停、终止）
- 成本跟踪数据（如果智能体报告）

#### 适配器配置（特定于智能体）

每个适配器类型都定义其自己的配置架构。示例：

- **OpenClaw 适配器**：SOUL.md 内容、HEARTBEAT.md 内容、OpenClaw 特定设置
- **进程适配器**：运行命令、环境变量、工作目录
- **HTTP 适配器**：端点 URL、身份验证标头、负载模板

#### 可导出的组织配置

一个关键目标：**整个组织的智能体配置都是可导出的。**您可以将公司的完整智能体设置（每个智能体、其适配器配置、组织结构）导出为便携式工件。这使得：

- 共享公司模板（“这是一个预先构建的营销机构组织”）
- 版本控制您的公司配置
- 复制/分叉公司

#### 上下文传递

每个智能体均可配置。频谱的两端：

- **胖负载** - Paperclip 将相关上下文（当前任务、消息、公司状态、指标）捆绑到心跳调用中。适用于无法回拨 Paperclip 的简单/无状态智能体。
- **薄 ping** — 心跳只是一个唤醒信号。智能体调用 Paperclip 的 API 来获取它需要的任何上下文。适合管理自己状态的复杂智能体。

#### 最低合同

成为 Paperclip 智能体的最低要求：**可调用。** 就是这样。 Paperclip 可以通过命令或 webhook 调用您。无需返回报告 - Paperclip 尽可能从进程活跃度推断基本状态。

#### 集成级别

除了最低要求之外，Paperclip 还提供了逐渐丰富的集成：1. **可调用**（最低）- Paperclip 可以启动您。这是唯一的合同。
2. **状态报告** — 智能体在执行后报告成功/失败/进行中。
3. **完全检测** — 智能体报告状态、成本/令牌使用情况、任务更新和日志。与控制平面的双向集成。

Paperclip 附带**默认智能体**，展示了完全集成：进度跟踪、成本仪表和用于任务管理的 **Paperclip 技能**（用于与 Paperclip API 交互的 Claude Code 技能）。这些对于适配器作者来说既是有用的默认值又是参考实现。

#### 导出格式

两种导出模式：

1. **模板导出**（默认）- 仅结构：智能体定义、组织架构图、适配器配置、角色描述。可以选择包含一些种子任务来帮助入门。这是组建新公司的蓝图。
2. **快照导出** — 完整状态：结构+当前任务、进度、座席状态。您可以恢复或分叉完整的图片。

通常的工作流程：导出模板，从中创建一个新公司，添加几个初始任务，然后开始。

## 3. 组织结构[草案]

分层报告结构。首席执行官在上，报告向下。

### 智能体可见性

**整个组织的完全可见性。** 每个智能体都可以看到整个组织架构图、所有任务、所有智能体。组织结构定义**报告和授权路线**，而不是访问控制。

每个智能体都会发布其职责和能力的简短描述——几乎就像技能一样（“当我相关时”）。这可以让其他智能体发现谁可以提供帮助。

### 跨团队工作

座席可以创建任务并将其分配给报告线之外的座席。这就是跨团队协作的机制。这些规则主要编码在 Paperclip SKILL.md 中，建议所有智能体使用。 Paperclip 该应用程序强制执行工具和一些轻型治理，但下面的跨团队规则主要由智能体决策来实施。

#### 任务接受规则

当智能体收到来自其团队外部的任务时：

1. **同意合适+可以做** → 直接完成
2. **同意这是合适的 + 不能这样做** → 标记为已阻止
3. **询问是否值得做** → **不能自己取消。** 必须重新分配给自己的经理，解释情况。经理决定是否接受、重新分配或升级。

#### 经理升级协议

任何经理都有责任了解下属被阻止的原因并解决它：

0. **决定**——作为管理者，这项工作值得做吗？
1. **向下授权**——请他们手下的人帮忙解除封锁
2. **向上升级**——向上级经理寻求帮助

#### 请求深度跟踪当任务源自跨团队请求时，以整数形式跟踪**深度** - 来自原始请求者的委托跃点数。这可以让您了解工作在组织中的级联程度。

#### 帐单代码

任务带有**计费代码**，以便执行期间的令牌支出可以归因于请求任务/智能体的上游。当智能体 A 要求智能体 B 工作时，B 的工作成本将根据 A 的请求进行跟踪。这使得整个组织的成本归因成为可能。

### 开放问题

- 这是一棵严格的树还是智能体可以向多个经理报告？
- 组织结构可以在运行时改变吗？ （智能体人重新分配，团队重组）
- 智能体是否继承其经理的任何配置？
- 帐单代码格式 - 简单字符串？等级制度？

## 4.心跳系统[草案]

心跳是一个协议，而不是运行时。 Paperclip 定义如何启动智能体周期。智能体在这个周期中做什么——运行多长时间，无论是任务范围的还是连续的——完全取决于智能体。

### 执行适配器

智能体配置包括一个**适配器**，它定义 Paperclip 如何调用智能体。初始适配器：

|适配器|机制|示例|
| --------- | ----------------------- | --------------------------------------------- |
| `process` |执行子进程 | `python run_agent.py --agent-id {id}` |
| `http` |发送 HTTP 请求 | `POST https://openclaw.example.com/hook/{id}` |

`process` 和 `http` 适配器默认发货。可以通过插件系统添加其他适配器（请参阅插件/扩展架构）。

### 适配器接口

每个适配器都实现三种方法：

```
invoke(agentConfig, context?) → void     // Start the agent's cycle
status(agentConfig) → AgentStatus        // Is it running? finished? errored?
cancel(agentConfig) → void               // Graceful stop signal (for pause/resume)
```

这是完整的适配器合同。 `invoke` 启动智能体，`status` 让 Paperclip 检查它，`cancel` 启用板的暂停功能。其他一切（成本报告、任务更新）都是可选的，并通过 Paperclip REST API 流动。

### Paperclip 控制什么

- **何时**触发心跳（时间表/频率，每个智能体）
- **如何**启动它（适配器选择+配置）
- **要包含什么上下文**（瘦 ping 与胖有效负载，每个智能体）

### Paperclip 不控制的内容

- 智能体运行多长时间
- 智能体在其周期内做什么
- 智能体是任务范围的、时间窗口的还是连续的

### 暂停行为

当董事会（或系统）暂停客服人员时：

1. **向当前执行发出信号** — 向正在运行的进程/会话发送优雅终止信号
2. **宽限期** — 给智能体时间来总结、保存状态、报告最终状态
3. **超时后强制终止** — 如果智能体在宽限期内没有停止，则终止
4. **停止未来的心跳** - 在智能体恢复之前不会触发新的心跳周期这就是“优雅的信号+停止未来的心跳”。当前的运行有机会干净着陆。

### 开放问题

- 心跳频率——谁控制它？固定的？每个智能体？类似克朗？
- 当心跳调用失败时会发生什么？ （进程崩溃，HTTP 500）
- 健康监测——Paperclip 如何区分“卡住”和“处理长时间任务”？
- 智能体可以自行触发下一次心跳吗？ （“我完成了，5 分钟后再次叫醒我”）
- 宽限期持续时间——固定吗？每个智能体都可以配置吗？

## 5. 智能体间通信 [草案]

所有智能体通信都通过**任务系统**进行。

### 模型：任务 + 评论

- **委派** = 创建任务并将其分配给另一个智能体
- **协调** = 对任务进行评论
- **状态更新** = 更新任务状态和字段

没有单独的消息或聊天系统。任务是沟通的渠道。这使得所有上下文都与其相关的工作相关联，并创建自然的审计跟踪。

### 影响

- 客服人员的“收件箱”是：分配给他们的任务+对他们参与的任务的评论
- CEO 通过创建分配给 CTO 的任务来进行委派
- CTO 将这些分解为分配给工程师的子任务
- 讨论发生在任务评论中，而不是旁路渠道
- 如果智能体需要升级，他们会对父任务发表评论或重新分配

### 任务层次结构映射

完整的层次结构：**倡议**（公司目标）→ 项目→ 里程碑→ 问题→ 子问题。一切都可以追溯到一个倡议，而“公司目标”只是第一个/主要倡议。

## 6. 成本跟踪 [草案]

Token/LLM成本预算是Paperclip的核心部分。外部收入和支出跟踪是未来的插件。

### 成本报告

配备齐全的智能体将令牌/API 使用情况报告回 Paperclip。各个级别的成本都被跟踪：

- **每个智能体** — 该员工的费用是多少？
- **每个任务** — 这个工作单元的成本是多少？
- **每个项目** — 此可交付成果的成本是多少？
- **每家公司** — 总烧钱率

成本应以**代币和美元**计价。

任务的计费代码（请参阅组织结构）支持跨团队的成本归因 - 当智能体 A 向智能体 B 请求工作时，B 的成本将累计到 A 的请求。

### 预算控制

三层：

1. **可见性** — 显示各个级别支出（智能体、任务、项目、公司）的控制台
2. **软警报** — 可配置的阈值（例如，在预算的 80% 时发出警告）
3. **硬上限** — 当预算达到时自动暂停智能体。董事会已通知。董事会可以推翻/提高限制。

预算可以设置为**无限制**（无上限）。

### 开放问题- 成本报告 API — 智能体报告成本的模式是什么？
- 控制台设计——每个级别上哪些指标最重要？
- 预算周期——每天？每周？每月？滚动？

## 7. 默认智能体和引导流程 [草案]

### 自举序列

公司如何从“创建”到“运营”：

1. Human 创建一家公司及其最初的举措
2. 人类定义初始顶级任务
3. 人工创建CEO Agent（使用默认CEO模板或自定义）
4. CEO的第一次心跳：审查计划和任务，提出战略分解（组织结构、子任务、招聘计划）
5. **董事会批准**首席执行官的战略计划
6. CEO 开始执行——创建任务、提议聘用（董事会批准）、委派

### 默认智能体

Paperclip 附带默认智能体模板：

- **默认智能体** — 基本 Claude Code 或 Codex 循环。了解 **Paperclip 技能** (SKILL.md)，因此可以与任务系统交互、读取公司上下文、报告状态。
- **默认首席执行官** — 将默认智能体扩展到首席执行官特定的行为：战略规划、报告授权、进度审查、董事会沟通。

这些是起点。用户可以自定义或完全替换它们。

### 默认智能体行为

默认智能体的循环是**配置驱动的**。适配器配置包含定义智能体在每个心跳周期执行的操作的指令。没有硬编码的标准循环 - 每个智能体的配置决定其行为。

这意味着默认的 CEO 配置告诉 CEO 审查策略、检查报告等。默认的工程师配置告诉工程师检查分配的任务、选择最高优先级并执行它。但这些是配置选择，而不是协议要求。

### Paperclip 技能 (SKILL.md)

教导智能体如何与 Paperclip 交互的技能定义。提供：

- 任务CRUD（创建、读取、更新、完成任务）
- 状态报告（签到、报告进度）
- 公司背景（阅读目标、组织架构图、当前状态）
- 成本报告（记录代币/API 使用情况）
- 智能体间通信规则

该技能与适配器无关 - 它可以加载到 Claude Code 中、注入到提示中或用作自定义智能体的 API 文档。

## 8. 架构和部署 [草案]

### 部署模型

**单租户，可自行托管。**不是 SaaS。一个实例 = 一个运营商的公司。

#### 开发路径（渐进式部署）1. **本地开发** — 一个安装和运行命令。嵌入式 Postgres。一切都在你的机器上。智能体在本地运行。
2. **托管** — 部署到 Vercel/Supabase/AWS/任何地方。远程智能体通过共享数据库连接到您的服务器。用户界面可通过网络访问。
3. **开放公司** - 可以选择公开部分内容（例如，开放公司的工作委员会对公众可见）。

关键约束：从“我在我的机器上尝试这个”到“我的智能体在远程服务器上运行并与我的 Paperclip 实例对话”必须是微不足道的。

#### 智能体身份验证

当用户创建智能体时，Paperclip 会生成一个**连接字符串**，其中包含：服务器 URL、API 密钥以及如何进行身份验证的说明。假设智能体能够弄清楚如何使用其令牌/密钥来调用 API。

流程：

1. Human在UI中创建Agent
2. Paperclip 生成连接字符串（URL+key+指令）
3. 人类将此字符串提供给智能体（例如在其适配器配置、环境等中）
4. Agent使用密钥验证API对控制平面的调用

### 技术堆栈

|层|技术 |
| -------- | ------------------------------------------------------------ |
|前端 | React + Vite |
|后端| TypeScript + Hono（REST API，不是 tRPC — 需要非 TS 客户端）|
|数据库| PostgreSQL（有关详细信息，请参阅 [doc/DATABASE.md](./doc/DATABASE.md) — 用于开发的嵌入式 PGlite、用于生产的 Docker 或托管 Supabase）|
|授权 | [更好的认证](https://www.better-auth.com/) |

### 并发模型：原子任务结账

任务使用**单一分配**（每个任务一个智能体）和**原子结账**：

1. Agent尝试将任务设置为`in_progress`（认领）
2. API/数据库以原子方式强制执行此操作 - 如果另一个智能体已声明它，则请求将失败并显示错误，以确定哪个智能体拥有它
3. 如果任务已从之前的会话分配给请求智能体，他们可以恢复

不需要乐观锁定或 CRDT。单分配模型+原子校验在设计层面避免了冲突。

### 人在循环中

智能体可以创建分配给人类的任务。董事会成员（或任何有权访问的人员）可以通过 UI 完成这些任务。

当人类完成任务时，如果请求智能体的适配器支持 **pingbacks**（例如 OpenClaw 挂钩），Paperclip 会发送通知以唤醒该智能体。这使得人类在工作流程中成为罕见但可能的参与者。

不鼓励智能体将任务分配给 Paperclip 技能中的人类，但有时这是不可避免的。

### API 设计**单个统一的 REST API。** 相同的 API 服务于前端 UI 和智能体。身份验证决定权限 - 董事会身份验证具有完全访问权限，智能体 API 密钥具有范围访问权限（他们自己的任务、成本报告、公司背景）。

没有单独的“智能体 API”与“董事会 API”。相同的端点，不同的授权级别。

### 工作工件

Paperclip **不**管理工作工件（代码存储库、文件系统、部署、文档）。这完全是智能体人的职责范围。 Paperclip 跟踪任务和成本。工作在哪里以及如何完成超出了范围。

### 开放问题

- UI 实时更新 — WebSocket？上交所？轮询？
- 智能体 API 密钥范围 - 智能体到底可以访问什么？只有自己的任务吗？他们队的？整个公司？

### 崩溃恢复：手动，非自动

当智能体在任务中崩溃或消失时，Paperclip 不会**自动重新分配或自动释放任务。相反：

- Paperclip 通过控制台和报告显示过时的任务（`in_progress` 中没有最近活动的任务）
- Paperclip 不会默默地失败——审计和可见性工具使问题变得显而易见
- 恢复由人类或紧急流程处理（例如项目经理智能体，其工作是监视陈旧工作并将其浮出水面）

**原理：Paperclip 报告问题，但不会默默修复它们。** 自动恢复隐藏故障。良好的可见性可以让正确的实体（人或智能体）决定要做什么。

### 插件/扩展架构

核心Paperclip系统必须是可扩展的。知识库、外部收入跟踪和新智能体适配器等功能应该可以作为**插件**添加，而无需修改核心。这意味着：

- 插件可以挂钩的明确定义的 API 边界
- 用于对任务/智能体生命周期事件做出反应的事件系统或挂钩
- **智能体适配器插件** — 新的适配器类型可以通过插件系统注册
- 可插件注册的 UI 组件（未来）

这不是 V1 可交付成果（我们没有预先构建插件框架），但该架构不应该让我们陷入困境。保持边界清晰，以便扩展成为可能。

## 9. 前端/UI [草案]

### 主要观点

每个都是不同的页面/路线：1. **组织架构图** — 每个智能体具有实时状态指示器（运行/空闲/暂停/错误）的组织树。智能体正在做什么的实时活动源。
2. **任务板** — 任务管理。看板和列表视图。按团队、智能体、项目、状态过滤。
3. **控制台** — 高级指标：智能体数量、活动任务、成本、目标进度、消耗率。来自 GOAL.md 的“概览”视图。
4. **智能体详细信息** — 深入了解单个智能体：他们的任务、活动、成本、配置、状态历史记录。
5. **项目/倡议视图** — 根据里程碑和目标进行进度跟踪。
6. **成本控制台** — 每个级别（智能体、任务、项目、公司）的支出可视化。

### 主板控制（随处可用）

- 暂停/恢复智能体（任何视图）
- 暂停/恢复任务/项目（任何视图）
- 批准/拒绝待决行动（招聘、战略提案）
- 直接任务创建、编辑、评论

## 11. 知识库**核心的反目标。** 知识库不是 Paperclip 核心的一部分 - 它将是一个插件。任务系统+评论+智能体描述提供了足够的共享上下文。

该架构必须支持稍后添加知识库插件（干净的 API 边界、可挂钩的生命周期事件），但核心系统明确不包含该插件。

---

## 12. 反需求

Paperclip 明确**不**做的事情：

- **不是智能体运行时** — Paperclip 编排，智能体在其他地方运行
- **不是知识库** — 核心没有 wiki/docs/vector-DB（插件领域）
- **不是 SaaS** — 单租户、自托管
- **对智能体实现没有固执己见** - 任何语言、任何框架、任何运行时
- **不会自动自我修复** — 暴露问题，但不会默默修复它们
- **不管理工作工件** - 没有存储库管理，没有部署，没有文件系统
- **不自动重新分配工作** - 过时的任务会浮出水面，不会默默地重新分配
- **不跟踪外部收入/支出** - 这是未来的插件。 Token/LLM 成本预算是核心。

## 13. 原则（综合）

1. **对如何运行智能体不持任何意见。** 任何语言、任何框架、任何运行时。 Paperclip 是控制平面，而不是执行平面。
2. **公司是组织单位。**一切都在公司之下。
3. **任务是沟通渠道。** 所有Agent的沟通都是通过任务+评论来进行的。无侧通道。
4. **所有工作都追踪目标。** 分层任务管理——没有什么是孤立存在的。
5. **董事会进行管理。** 人类通过董事会保留控制权。保守的默认值（需要人工批准）。
6. **表面问题，不要隐藏它们。** 良好的审核和可见性。没有静默自动恢复。
7. **原子所有权。** 每个任务只有一个受让人。原子结账可防止冲突。
8. **渐进式部署。** 在本地启动很简单，可以直接扩展到托管。
9. **可扩展核心。** 清晰的边界，因此插件可以添加功能（适配器、知识库、收入跟踪）而无需修改核心。
