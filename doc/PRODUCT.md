# Paperclip — 产品定义

## 它是什么

Paperclip 是自主 AI 公司的控制面板。一个 Paperclip 实例可以运行多家公司。**公司**是一等对象。

## 核心概念

### 公司

一家公司具有：

- **目标** — 它存在的理由（"创建排名第一的 AI 笔记应用，3 个月内达到 100 万美元 MRR"）
- **员工** — 每个员工都是 AI 智能体
- **组织结构** — 谁向谁汇报
- **收入与支出** — 在公司层面追踪
- **任务层级** — 所有工作都追溯到公司目标

### 员工与智能体

每个员工都是一个智能体。创建公司时，你从定义 CEO 开始，然后从那里扩展。

每个员工具有：

- **适配器类型 + 配置** — 该智能体如何运行以及什么定义其身份/行为。这是适配器特定的（例如 OpenClaw 智能体可能使用 SOUL.md 和 HEARTBEAT.md 文件；Claude Code 智能体可能使用 CLAUDE.md；裸脚本可能使用 CLI 参数）。Paperclip 不规定格式 — 适配器规定。
- **角色与汇报** — 他们的职称、向谁汇报、谁向他们汇报
- **能力描述** — 关于该智能体做什么以及何时相关的简短段落（帮助其他智能体发现谁能帮助什么）

示例：CEO 智能体的适配器配置告诉它每次心跳时"检查你的高管在做什么，查看公司指标，如需要则重新排序优先级，分配新的战略举措"。工程师的配置告诉它"检查分配的任务，选择最高优先级的，开始工作"。

然后你定义谁向 CEO 汇报：管理程序员的 CTO、管理营销团队的 CMO，等等。树中的每个智能体都有自己的适配器配置。

### 智能体执行

运行智能体心跳有两种基本模式：

1. **运行命令** — Paperclip 启动一个进程（shell 命令、Python 脚本等）并追踪它。心跳就是"执行这个并监控它"。
2. **发后不管请求** — Paperclip 向外部运行的智能体发送 webhook/API 调用。心跳就是"通知这个智能体醒来"。（OpenClaw hooks 就是这样工作的。）

我们提供合理的默认值 — 一个默认智能体，使用你的配置调用 Claude Code 或 Codex，记住会话 ID，运行基本脚本。但你可以接入任何东西。

### 任务管理

任务管理是分层的。在任何时刻，每项工作必须通过一连串的父任务追溯到公司的顶层目标：

```
I am researching the Facebook ads Granola uses (current task)
  because → I need to create Facebook ads for our software (parent)
    because → I need to grow new signups by 100 users (parent)
      because → I need to get revenue to $2,000 this week (parent)
        because → ...
          because → We're building the #1 AI note-taking app to $1M MRR in 3 months
```

任务有父级关系。每个任务都为服务于一个父任务而存在，一直追溯到公司目标。这就是让自主智能体保持对齐的机制 — 它们总是能回答"我为什么要做这件事？"

更详细的任务结构待定。

## 原则

1. **对你如何运行智能体不做预设。**你的智能体可以是 OpenClaw 机器人、Python 脚本、Node 脚本、Claude Code 会话、Codex 实例 — 我们不关心。Paperclip 定义了通信的控制面板，并为心跳提供基础设施。它不强制要求智能体运行时。

2. **公司是组织单位。**一切都在公司之下。一个 Paperclip 实例，多家公司。

3. **适配器配置定义智能体。**每个智能体有一个适配器类型和配置来控制其身份和行为。最低合约就是"可被调用"。

4. **所有工作追溯到目标。**分层任务管理意味着没有任何事物孤立存在。如果你无法解释一个任务对公司目标的意义，它就不应该存在。

5. **控制面板，不是执行面板。**Paperclip 协调。智能体在它们运行的地方运行，然后回电汇报。

## 用户流程（理想场景）

1. 打开 Paperclip，创建一家新公司
2. 定义公司目标："创建排名第一的 AI 笔记应用，3 个月内达到 100 万美元 MRR"
3. 创建 CEO
   - 选择适配器（例如 Claude Code 的进程适配器、OpenClaw 的 HTTP 适配器）
   - 配置适配器（智能体身份、循环行为、执行设置）
   - CEO 提出战略分解 → 董事会审批
4. 定义 CEO 的直接下属：CTO、CMO、CFO 等
   - 每人都有自己的适配器配置和角色定义
5. 定义他们的下属：CTO 下的工程师、CMO 下的营销人员等
6. 设置预算，定义初始战略任务
7. 启动 — 智能体开始心跳，公司运转起来

## 指导方针

Paperclip 必须支持两种运行时模式：

- `local_trusted`（默认）：单用户本地信任部署，无登录摩擦
- `authenticated`：需要登录的模式，支持私有网络和公共部署暴露策略

规范模式设计和命令预期见 `doc/DEPLOYMENT-MODES.md`。

Paperclip 的核心身份是**自主 AI 公司的控制面板**，以**公司、组织架构图、目标、任务/评论、心跳、预算、审批和董事会治理**为中心。公开文档也明确了当前的边界：**任务/评论是内置的通信模型**，Paperclip **不是聊天机器人**，也**不是代码审查工具**。路线图已经指向了**更简单的入门、云端智能体、更简单的智能体配置、插件、更好的文档，以及 ClipMart/ClipHub 风格的可复用公司/模板**。

## Paperclip 应该做什么与不应该做什么

**应该做**

- 保持**董事会级和公司级**。用户应管理目标、组织、预算、审批和产出。
- 让前五分钟感觉很神奇：安装、回答几个问题、看到 CEO 做了真正的事情。
- 保持工作锚定在**任务/评论/项目/目标**上，即使表面感觉像对话。
- 将**代理机构/内部团队/创业公司**视为具有不同模板和标签的相同底层抽象。
- 让产出成为一等公民：文件、文档、报告、预览、链接、截图。
- 提供**工程工作流钩子**：worktree、预览服务器、PR 链接、外部审查工具。
- 使用**插件**处理边缘情况如富聊天、知识库、文档编辑器、自定义追踪。

**不应该做**

- 不要让核心产品成为通用聊天应用。当前产品定义明确是以任务/评论为中心的"不是聊天机器人"，这个边界很有价值。
- 不要构建完整的 Jira/GitHub 替代品。仓库/文档已经将 Paperclip 定位为组织编排，而非聚焦于 Pull Request 审查。
- 不要先构建企业级 RBAC。当前 V1 规范仍将多董事会治理和细粒度人类权限视为范围外，所以第一个多用户版本应该是粗粒度且公司范围的。
- 不要以原始 bash 日志和脚本作为主导。默认视图应该是人类可读的意图/进度，原始细节在下面。
- 不要强迫用户理解提供者/API 密钥配置，除非绝对必要。已经有活跃的入门/认证问题；这里的摩擦显然是真实的。

## 具体设计目标

1. **首次成功时间不超过 5 分钟**
   新用户应该能在一次坐下来的时间内从安装到"我的 CEO 完成了第一个任务"。

2. **董事会级抽象始终优先**
   默认 UI 应回答：公司在做什么、谁在做、为什么重要、花了多少钱、什么需要我审批。

3. **对话保持附着在工作对象上**
   "与 CEO 聊天"应仍然解析为策略线程、决策、任务或审批。

4. **渐进式披露**
   顶层：人类可读的摘要。中间层：清单/步骤/制品。底层：原始日志/工具调用/脚本。

5. **产出优先**
   工作未完成直到用户能看到结果：文件、文档、预览链接、截图、计划或 PR。

6. **本地优先，云端就绪**
   心智模型不应在本地单人使用和共享/私有或公共/云端部署之间改变。

7. **安全的自主性**
   允许自动模式；不允许隐藏的 token 消耗。

8. **精简核心，丰富边缘**
   将可选的聊天、知识和特殊界面放入插件/扩展中，而非膨胀控制面板。
