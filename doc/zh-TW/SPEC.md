# Paperclip 規格

Paperclip 控制平面的目標規格。動態文件－在規格面試期間逐步更新。

---

## 1. 公司模式[草稿]

公司是一階對象。一個 Paperclip 實例運行多個公司。公司沒有獨立的「目標」欄位 - 它的方向是由其一組計劃定義的（請參閱任務層次結構映射）。

### 欄位（草稿）

|領域 |類型 |筆記|
| ----------- | ------------- | --------------------------------- |
| `id` | uuid|主鍵 |
| `name` |字串|公司名稱 |
| `createdAt` |時間戳記| |
| `updatedAt` |時間戳| |

### 董事會治理[草案]

每家公司都有一個**董事會**來管理高影響力的決策。董事會是人類監督層。

**V1：單人板。 ** 一名操作員。

#### 董事會批准門 (V1)

- 僱用新特務（創建新特務）
- CEO 的初始策略分解（CEO 提議，董事會在執行開始前批准）
- [待定：其他治理控制行動－目標改變、解僱特務？ ]

#### 董事會權力（總是可用）

董事會始終可以**不受限制地存取**整個系統：

- **設定與修改公司預算** — 董事會設定頂級代幣/LLM 成本預算
- **暫停/恢復任何智能體** — 立即停止智能體的心跳
- **暫停/恢復任何工作項目** — 暫停任務、專案、子任務樹、里程碑。暫停的項目不會被智能體拾取。
- **完整的專案管理存取** — 透過 UI 建立、編輯、評論、修改、刪除、重新分配任何任務/專案/里程碑
- **推翻任何智能體決定** — 重新分配任務、更改優先順序、修改描述
- **手動更改任何級別的任何預算**

董事會不僅僅是一個批准門——它還是一個即時控制介面。人類可以隨時在任何層面進行幹預。

#### 預算授權

董事會制定公司級預算。 CEO 可以為其下級智能體設定預算，每個經理智能體都可以為其報告執行相同的操作。這種級聯預算授權在實踐中如何運作尚待確定，但權限結構支援它。董事會可以手動覆蓋任何級別的任何預算。

**未來的治理模式**（不是 V1）：

- 招聘預算（每月 X 美元以內自動批准招聘）
- 多人董事會
- 授權（執行長可以在限制範圍內聘用）

### 開放問題

- 外部收入/支出追蹤－未來的插件。 Token/LLM 成本預算是核心。
- 公司層級的設定和配置？
- 公司生命週期（暫停、存檔、刪除）？
- 除了招募和執行長策略批准之外，還有哪些治理控制措施？

---## 2. 智能體模型[草稿]

每個員工都是智能體。智能體是勞動力。

### 智能體程式身分（適配器層級）

SOUL.md（身分/任務）和 HEARTBEAT.md（循環定義）等概念**不是 Paperclip 協定的一部分**。它們是特定於適配器的配置。例如，OpenClaw 適配器可能使用 SOUL.md 和 HEARTBEAT.md 檔案。 Claude Code 適配器可能使用 CLAUDE.md。裸 Python 腳本可能使用命令列參數。

Paperclip 沒有規定智能體如何定義其身分或行為。它提供控制平面；適配器定義了智能體的內部工作方式。

### 智能體程式配置[草稿]

每個智能體程式都有一個 **適配器類型** 和一個 **特定於適配器的配置 blob**。適配器定義存在哪些配置欄位。

#### Paperclip 協定（Paperclip 所知道的）

在協議級別，Paperclip 追蹤：

- 智能體身分（ID、姓名、角色、頭銜）
- 組織職位（他們向誰報告，誰向他們報告）
- 適配器類型+適配器配置
- 狀態（活動、暫停、終止）
- 成本追蹤數據（如果智能體報告）

#### 適配器配置（特定於智能體程式）

每個適配器類型都定義其自己的配置架構。範例：

- **OpenClaw 適配器**：SOUL.md 內容、HEARTBEAT.md 內容、OpenClaw 特定設置
- **進程適配器**：執行指令、環境變數、工作目錄
- **HTTP 適配器**：端點 URL、驗證標頭、負載模板

#### 可匯出的組織配置

一個關鍵目標：**整個組織的智能體配置都是可匯出的。 **您可以將公司的完整智能體設定（每個智能體程式、其適配器配置、組織結構）匯出為便攜式工件。這使得：

- 分享公司範本（「這是一個預先建立的行銷機構組織」）
- 版本控制您的公司配置
- 複製/分叉公司

#### 上下文傳遞

每個智能體均可配置。頻譜的兩端：

- **胖負載** - Paperclip 將相關上下文（目前任務、訊息、公司狀態、指標）捆綁到心跳呼叫中。適用於無法回撥 Paperclip 的簡單/無狀態智能體程式。
- **薄 ping** — 心跳只是一個喚醒訊號。智能體呼叫 Paperclip 的 API 來取得它需要的任何上下文。適合管理自己狀態的複雜智能體。

#### 最低合約

成為 Paperclip 智能體程式的最低要求：**可呼叫。 ** 就是這樣。 Paperclip 可以透過指令或 webhook 呼叫您。無需返回報告 - Paperclip 盡可能從進程活躍度推斷基本狀態。

#### 整合級別

除了最低要求之外，Paperclip 還提供了逐漸豐富的整合：1. **可呼叫**（最低）- Paperclip 可以啟動您。這是唯一的合約。
2. **狀態報告** — 智能體執行後報告成功/失敗/進行中。
3. **完全偵測** — 智能體程式報告狀態、成本/令牌使用、任務更新和日誌。與控制平面的雙向整合。

Paperclip 附帶**預設智能體**，展示了完全整合：進度追蹤、成本儀表和用於任務管理的 **Paperclip 技能**（用於與 Paperclip API 交互的 Z2QQ00ZZ）。這些對於適配器作者來說既是有用的預設值又是參考實作。

#### 匯出格式

兩種匯出模式：

1. **範本匯出**（預設）- 僅結構：智能體定義、組織架構圖、適配器配置、角色描述。可以選擇包含一些種子任務來幫助入門。這是組成新公司的藍圖。
2. **快照匯出** — 完整狀態：結構+目前任務、進度、座席狀態。您可以恢復或分叉完整的圖片。

通常的工作流程：匯出模板，從中建立一個新公司，新增幾個初始任務，然後開始。

## 3. 組織架構[草稿]

分層報告結構。執行長在上，報告向下。

### 智能體可見性

**整個組織的完全可見性。 ** 每個智能體程式都可以看到整個組織架構圖、所有任務、所有智能體程式。組織結構定義**報告和授權路線**，而不是存取控制。

每個智能體都會發布其職責和能力的簡短描述——幾乎就像技能一樣（「當我相關時」）。這可以讓其他智能體發現誰可以提供幫助。

### 跨團隊工作

座席可以建立任務並將其指派給報告線之外的座席。這就是跨團隊協作的機制。這些規則主要編碼在 Paperclip SKILL.md 中，建議所有智能體程式使用。 Paperclip 該應用程式強制執行工具和一些輕型治理，但以下的跨團隊規則主要由智能體決策來實施。

#### 任務接受規則

當智能體收到來自其團隊外部的任務時：

1. **同意適合+可以做** → 直接完成
2. **同意這是適當的 + 不能這樣做** → 標記為已阻止
3. **詢問是否值得做** → **不能自己取消。 ** 必須重新分配給自己的經理，解釋情況。經理決定是否接受、重新分配或升級。

#### 經理升級協議

任何經理都有責任了解下屬被阻止的原因並解決它：

0. **決定**－身為管理者，這項工作值得做嗎？
1. **向下授權**－請他們手下的人幫忙解除封鎖
2. **向上升級**－向上級經理尋求協助

#### 請求深度跟踪當任務源自於跨團隊請求時，以整數形式追蹤**深度** - 來自原始請求者的委託躍點數。這可以讓您了解工作在組織中的級聯程度。

#### 帳單代碼

任務帶有**計費代碼**，以便執行期間的令牌支出可以歸因於請求任務/智能體的上游。當智能體 A 要求智能體 B 工作時，B 的工作成本將根據 A 的請求進行追蹤。這使得整個組織的成本歸因成為可能。

### 開放問題

- 這是一棵嚴格的樹還是智能體可以向多個經理報告？
- 組織結構可以在運作時改變嗎？ （智能體重新分配，團隊重組）
- 智能體是否繼承其經理的任何配置？
- 帳單代碼格式 - 簡單字串？等級制度？

## 4.心跳系統[草稿]

心跳是一個協議，而不是運行時。 Paperclip 定義如何啟動智能體週期。智能體在這個週期中做什麼——運行多長時間，無論是任務範圍的還是連續的——完全取決於智能體。

### 執行適配器

智能體配置包括一個**適配器**，它定義 Paperclip 如何呼叫智能體。初始適配器：

|適配器|機制|範例|
| --------- | ----------------------- | --------------------------------------------- |
| `process` |執行子程序| `python run_agent.py --agent-id {id}` |
| `http` |發送 HTTP 請求 | `POST https://openclaw.example.com/hook/{id}` |

`process` 和 `http` 轉接器預設出貨。可以透過插件系統新增其他適配器（請參閱插件/擴充架構）。

### 適配器接口

每個適配器都實作三種方法：

```
invoke(agentConfig, context?) → void     // Start the agent's cycle
status(agentConfig) → AgentStatus        // Is it running? finished? errored?
cancel(agentConfig) → void               // Graceful stop signal (for pause/resume)
```

這是完整的適配器合約。 `invoke` 啟動智能體，`status` 讓 Paperclip 檢查它，`cancel` 啟用板的暫停功能。其他一切（成本報告、任務更新）都是可選的，並透過 Paperclip REST API 流動。

### Paperclip 控制什麼

- **何時**觸發心跳（時間表/頻率，每個智能體）
- **如何**啟動它（適配器選擇+配置）
- **要包含什麼上下文**（瘦 ping 與胖有效負載，每個智能體）

### Paperclip 不控制的內容

- 智能體運行多長時間
- 智能體在其周期內做什麼
- 智能體是任務範圍的、時間視窗的還是連續的

### 暫停行為

當董事會（或系統）暫停客服人員時：

1. **向目前執行發出信號** — 向正在運行的進程/會話發送優雅終止信號
2. **寬限期** — 給智能體時間來總結、保存狀態、報告最終狀態
3. **逾時後強制終止** — 若智能體在寬限期內沒有停止，則終止
4. **停止未來的心跳** - 在智能體恢復之前不會觸發新的心跳週期這就是「優雅的訊號+停止未來的心跳」。當前的運行有機會乾淨著陸。

### 開放問題

- 心跳頻率－誰控制它？固定的？每個智能體？類似克朗？
- 當心跳呼叫失敗時會發生什麼事？ （進程崩潰，HTTP 500）
- 健康監測—Paperclip 如何區分「卡住」和「處理長時間任務」？
- 智能體可以自行觸發下一次心跳嗎？ （「我完成了，5 分鐘後再叫醒我」）
- 寬限期持續時間－固定嗎？每個智能體都可以配置嗎？

## 5. 智能體間通信 [草案]

所有智能體通訊都透過**任務系統**進行。

### 模型：任務 + 評論

- **委派** = 建立任務並將其指派給另一個智能體
- **協調** = 對任務進行評論
- **狀態更新** = 更新任務狀態與字段

沒有單獨的訊息或聊天系統。任務是溝通的管道。這使得所有上下文都與其相關的工作相關聯，並創建自然的審計追蹤。

### 影響

- 客服人員的「收件匣」是：分配給他們的任務+對他們參與的任務的評論
- CEO 透過建立分配給 CTO 的任務來進行委派
- CTO 將這些分解為分配給工程師的子任務
- 討論發生在任務評論中，而不是旁路渠道
- 如果智能體需要升級，他們會對父任務發表評論或重新分配

### 任務層次結構映射

完整的層級結構：**倡議**（公司目標）→ 專案→ 里程碑→ 問題→ 子問題。一切都可以追溯到一個倡議，而「公司目標」只是第一個/主要倡議。

## 6. 成本追蹤 [草案]

Token/LLM成本預算是Paperclip的核心部分。外部收入和支出追蹤是未來的插件。

### 成本報告

配備齊全的智能體將令牌/API 使用情況回報回 Paperclip。各個層級的成本都被追蹤：

- **每位智能體** — 該員工的費用是多少？
- **每個任務** — 這個工作單元的成本是多少？
- **每個專案** — 此可交付成果的成本是多少？
- **每家公司** — 總燒錢率

成本應以**代幣和美元**計價。

任務的計費代碼（請參閱組織結構）支援跨團隊的成本歸因 - 當智能體 A 向智能體 B 請求工作時，B 的成本將累積到 A 的請求。

### 預算控制

三層：

1. **可見性** — 顯示各個層級支出（智能體、任務、專案、公司）的控制台
2. **軟警報** — 可設定的閾值（例如，在預算的 80% 時發出警告）
3. **硬上限** — 預算達到時自動暫停智能體。董事會已通知。董事會可以推翻/提高限制。

預算可以設定為**無限制**（無上限）。

### 開放問題- 成本報告 API — 智能體報告成本的模式是什麼？
- 控制台設計－每個等級上哪些指標最重要？
- 預算週期－每天？每週？每月？滾動？

## 7. 預設智能體與引導流程 [草稿]

### 自舉序列

公司如何從「創建」到「營運」：

1. Human 創建一家公司及其最初的舉措
2. 人類定義初始頂級任務
3. 人工創建CEO Agent（使用預設CEO範本或自訂）
4. CEO的第一次心跳：檢視計畫與任務，提出策略分解（組織架構、子任務、招募計畫）
5. **董事會批准**執行長的策略計劃
6. CEO 開始執行－創造任務、提議聘用（董事會批准）、委派

### 預設智能體

Paperclip 隨附預設智能體模板：

- **預設智能體** — 基本 Claude Code 或 Codex 循環。了解 **Paperclip 技能** (SKILL.md)，因此可以與任務系統互動、讀取公司上下文、報告狀態。
- **預設執行長** — 將預設智能體擴展到執行長特定的行為：策略規劃、報告授權、進度審查、董事會溝通。

這些是起點。用戶可以自訂或完全替換它們。

### 預設智能體行為

預設智能體的循環是**配置驅動的**。適配器配置包含定義智能體在每個心跳週期執行的操作的指令。沒有硬編碼的標準循環 - 每個智能體程式的配置決定其行為。

這意味著預設的 CEO 配置告訴 CEO 審查策略、檢查報告等。預設的工程師配置告訴工程師檢查分配的任務、選擇最高優先順序並執行它。但這些是配置選擇，而不是協定要求。

### Paperclip 技能 (SKILL.md)

教導智能體如何與 Paperclip 互動的技能定義。提供：

- 任務CRUD（建立、讀取、更新、完成任務）
- 狀態報告（簽到、報告進度）
- 公司背景（閱讀目標、組織架構圖、目前狀態）
- 成本報告（記錄代幣/API 使用情形）
- 智能體間通訊規則

此技能與適配器無關 - 它可以載入到 Claude Code 中、注入到提示中或用作自訂智能體的 API 文件。

## 8. 架構與部署 [草稿]

### 部署模型

**單一租戶，可自行託管。 **不是 SaaS。一個實例 = 一個運營商的公司。

#### 開發路徑（漸進式部署）1. **本地開發** — 一個安裝和運行命令。嵌入式 Postgres。一切都在你的機器上。智能體在本地運行。
2. **託管** — 部署到 Vercel/Supabase/AWS/任何地方。遠端智能體程式透過共享資料庫連接到您的伺服器。使用者介面可透過網路存取。
3. **開放公司** - 可以選擇公開部分內容（例如，開放公司的工作委員會對公眾可見）。

關鍵約束：從“我在我的機器上嘗試這個”到“我的智能體在遠端伺服器上運行並與我的 Paperclip 實例對話”必須是微不足道的。

#### 智能體身份驗證

當使用者建立智能體程式時，Paperclip 會產生一個**連接字串**，其中包含：伺服器 URL、API 金鑰以及如何進行驗證的說明。假設智能體能夠弄清楚如何使用其令牌/密鑰來呼叫 API。

流程：

1. Human在UI中創建Agent
2. Paperclip 產生連接字串（URL+key+指令）
3. 人類將此字串提供給智能體（例如在其適配器配置、環境等）
4. Agent使用金鑰驗證API對控制平面的調用

### 技術堆疊

|層|技術 |

|前端 | React + Vite |

### 並發模型：原子任務結帳

任務使用**單一分配**（每個任務一個智能體）和**原子結帳**：

1. Agent嘗試將任務設定為`in_progress`（認領）
2. API/資料庫以原子方式強制執行此操作 - 如果另一個智能體程式已聲明它，則請求將失敗並顯示錯誤，以確定哪個智能體擁有它
3. 如果任務已從先前的會話指派給請求智能體，他們可以恢復

不需要樂觀鎖定或 CRDT。單分配模型+原子校驗在設計層面避免了衝突。

### 人在循環中

智能體可以創建分配給人類的任務。董事會成員（或任何有權存取的人員）可以透過 UI 完成這些任務。

當人類完成任務時，如果請求智能體的適配器支援 **pingbacks**（例如 OpenClaw 掛鉤），Paperclip 會發送通知以喚醒該智能體。這使得人類在工作流程中成為罕見但可能的參與者。

不鼓勵智能體將任務分配給 Paperclip 技能中的人類，但有時這是不可避免的。

### API 設計**單一統一的 REST API。 ** 相同的 API 服務於前端 UI 和智能體程式。身分驗證決定權限 - 董事會身分驗證具有完全存取權限，智能體 API 金鑰具有範圍存取權限（他們自己的任務、成本報告、公司背景）。

沒有單獨的「智能體 API」與「董事會 API」。相同的端點，不同的授權等級。

### 工作工件

Paperclip **不**管理工作工件（程式碼儲存庫、檔案系統、部署、文件）。這完全是智能體的職責範圍。 Paperclip 追蹤任務和成本。工作在哪裡以及如何完成超出了範圍。

### 開放問題

- UI 即時更新 — WebSocket？上交所？輪詢？
- 智能體 API 金鑰範圍 - 智能體到底可以存取什麼？只有自己的任務嗎？他們隊的？整個公司？

### 崩潰復原：手動，非自動

當智能體在任務中崩潰或消失時，Paperclip 不會**自動重新分配或自動釋放任務。相反：

- Paperclip 透過控制台和報表顯示過時的任務（`in_progress` 中沒有最近活動的任務）
- Paperclip 不會默默地失敗——審計和可見性工具使問題變得顯而易見
- 恢復由人類或緊急流程處理（例如專案經理智能體，其工作是監視陳舊工作並將其浮出水面）

**原理：Paperclip 報告問題，但不會默默修復它們。 ** 自動恢復隱藏故障。良好的可見性可以讓正確的實體（人或智能體）決定要做什麼。

### 外掛程式/擴充架構

核心Paperclip系統必須是可擴充的。知識庫、外部收入追蹤和新智能體適配器等功能應該可以作為**插件**添加，而無需修改核心。這意味著：

- 插件可以掛鉤的明確定義的 API 邊界
- 用於對任務/智能體生命週期事件做出反應的事件系統或掛鉤
- **智能體適配器外掛程式** — 新的適配器類型可以透過插件系統註冊
- 可外掛程式註冊的 UI 元件（未來）

這不是 V1 可交付成果（我們沒有預先建立插件框架），但該架構不應該讓我們陷入困境。保持邊界清晰，以便擴展成為可能。

## 9. 前端/UI [草稿]

### 主要觀點

每個都是不同的頁面/路線：1. **組織架構圖** — 每個智能體程式具有即時狀態指示器（運行/空閒/暫停/錯誤）的組織樹。智能體正在做什麼的即時活動來源。
2. **任務板** — 任務管理。看板和清單視圖。依團隊、智能體、專案、狀態過濾。
3. **控制台** — 進階指標：智能體數量、活動任務、成本、目標進度、消耗率。來自 GOAL.md 的“概覽”視圖。
4. **智能體詳細資料** — 深入了解單一智能體：他們的任務、活動、成本、配置、狀態歷史記錄。
5. **專案/倡議視圖** — 根據里程碑和目標進行進度追蹤。
6. **成本控制台** — 每個等級（智能體、任務、專案、公司）的支出視覺化。

### 主機板控制（隨處可使用）

- 暫停/恢復智能體（任何視圖）
- 暫停/恢復任務/項目（任何視圖）
- 批准/拒絕待決行動（招募、策略提案）
- 直接任務建立、編輯、評論

## 12. 反需求

Paperclip 明確**不**做的事情：

- **不是智能體運行時** — Paperclip 編排，智能體在其他地方運行
- **不是知識庫** — 核心沒有 wiki/docs/vector-DB（外掛領域）
- **不是 SaaS** — 單一租用戶、自架
- **對智能體實現沒有固執己見** - 任何語言、任何框架、任何運行時
- **不會自動自我修復** — 暴露問題，但不會默默修復它們
- **不管理工作工件** - 沒有儲存庫管理，沒有部署，沒有檔案系統
- **不自動重新分配工作** - 過時的任務會浮出水面，不會默默地重新分配
- **不追蹤外部收入/支出** - 這是未來的插件。 Token/LLM 成本預算是核心。

## 13. 原則（綜合）

1. **對如何運行智能體不持任何意見。 ** 任何語言、任何框架、任何執行時期。 Paperclip 是控制平面，而不是執行平面。
2. **公司是組織單位。 **一切都在公司之下。
3. **任務是溝通管道。 ** 所有Agent的溝通都是透過任務+評論來進行的。無側通道。
4. **所有工作都追蹤目標。 ** 分層任務管理－沒有什麼是孤立存在的。
5. **董事會進行管理。 ** 人類透過董事會保留控制權。保守的預設值（需要人工批准）。
6. **表面問題，不要隱藏它們。 ** 良好的審核和可見性。沒有靜默自動恢復。
7. **原子所有權。 ** 每個任務只有一個受讓人。原子結帳可防止衝突。
8. **漸進式部署。 ** 在本機啟動很簡單，可以直接擴展到託管。
9. **可擴展核心。 ** 清晰的邊界，因此外掛程式可以添加功能（適配器、知識庫、收入追蹤）而無需修改核心。
