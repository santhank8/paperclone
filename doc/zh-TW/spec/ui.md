# Paperclip UI 規範

狀態：草案
日期：2026-02-17

## 1. 設計理念

Paperclip的UI是一個專業級的控制平面，而不是一個玩具控制台。它應該感覺就像你整天使用的那種工具——快速、鍵盤驅動、資訊密集而不混亂、預設為黑暗主題。每個像素都應該贏得自己的位置。

設計原則：

- **密集但可掃描。 ** 顯示最大訊息，無需點擊即可顯示。使用空格來分隔，而不是填充。
- **鍵盤優先。 ** 用於搜尋 (Cmd+K)、新問題 (C)、導航的全域快速鍵。高級用戶應該很少觸碰滑鼠。
- **上下文，非模態。 ** 透過對話方塊進行內嵌編輯。頁面導覽上的下拉式選單。使用者的心理環境不應該被不必要地破壞。
- **預設深色主題。 **中性灰色，而不是純黑色。強調顏色很少用於表示狀態和優先順序。文本是主要的視覺元素。

### 顏色系統

- **背景：** `hsl(220, 13%, 10%)`（深木炭，不是純黑色）
- **表面/卡：** `hsl(220, 13%, 13%)`
- **邊框：** `hsl(220, 10%, 18%)`
- **主要簡訊：** `hsl(220, 10%, 90%)`
- **簡訊輔助：** `hsl(220, 10%, 55%)`
- **強調（互動）：** `hsl(220, 80%, 60%)`（柔和的藍色）

狀態顏色（所有實體一致）：
- **積壓：** 灰色 `hsl(220, 10%, 45%)`
- **Todo：**灰藍色 `hsl(220, 20%, 55%)`
- **進行中：** 黃色 `hsl(45, 90%, 55%)`
- **評論中：** 紫羅蘭 `hsl(270, 60%, 60%)`
- **完成：** 綠色 `hsl(140, 60%, 50%)`
- **取消：** 灰色 `hsl(220, 10%, 40%)`
- **已封鎖：** 琥珀色 `hsl(25, 90%, 55%)`

優先指標：
- **關鍵：** 紅色圓圈，已填充
- **高：** 橘色圓圈，半填充
- **中：** 黃色圓圈，輪廓
- **低：** 灰色圓圈、輪廓、虛線

### 版式

- **字體：**系統字體堆疊（若載入則為Inter，否則為`-apple-system, BlinkMacSystemFont, 'Segoe UI'`）
- **正文：** 13px / 1.5 行高
- **標籤/元資料：** 11px / 大寫跟踪
- **標題：** 14-18px / 半粗體，切勿全部大寫

### 圖標

始終使用 `lucide-react`。每個側邊欄項目、每個狀態指示器、每個操作按鈕都應該有一個圖示。圖示在導覽中為 16 像素，在行中為 14 像素。

## 2. 應用程式外殼

該應用程式採用三區域佈局：

```
┌──────────┬────────────────────────────────────────────────┐
│          │  Breadcrumb bar                                │
│ Sidebar  ├──────────────────────────┬─────────────────────┤
│ (240px)  │  Main content            │  Properties panel   │
│          │  (flex-1)                │  (320px, optional)  │
│          │                          │                     │
└──────────┴──────────────────────────┴─────────────────────┘
```

- **側邊欄：** 固定在左側，240 像素。透過切換或鍵盤快速鍵可折疊為僅圖示（48px）。
- **麵包屑欄：** 跨越主+屬性上方的整個寬度。顯示導覽路徑、實體操作和視圖控制項。
- **主要內容：** 可捲動。包含主要視圖（清單、詳細資訊、圖表等）。
- **屬性面板：** 右側，320 像素。顯示在詳細視圖上（問題詳細資料、項目詳細資料、智能體程式詳細資料）。隱藏在清單檢視和控制台上。可調整大小。

當您按一下詳細視圖時，屬性面板會滑入；當您返回清單時，屬性面板會滑出。它不是側邊欄——它與所選實體相關。

## 3. 側邊欄側邊欄是主要導覽。它被分組為具有可折疊標題的邏輯部分。

### 3.1 公司標題

側邊欄頂部。始終可見。

```
┌─────────────────────────┐
│ [icon] Acme Corp      ▼ │  ← Company switcher dropdown
├─────────────────────────┤
│ [🔍]  [✏️]              │  ← Search + New Issue
└─────────────────────────┘
```

**公司切換器**是一個下拉按鈕，佔據側邊欄標題的整個寬度。它顯示：
- 公司圖示（帶有公司顏色的首字母頭像，或上傳的圖示）
- 公司名稱（如果過長則用省略號截斷）
- V 形向下圖標

點選開啟一個下拉式選單：
- 所有公司清單（有狀態點：綠色=活動，黃色=暫停，灰色=存檔）
- 下拉清單頂部的搜尋欄位（適用於擁有許多公司的使用者）
- 分音器
- `+ Create company` 底部操作

公司名稱下方，有一排圖示按鈕：
- **搜尋**（放大鏡圖示）- 開啟 Cmd+K 搜尋模式
- **新問題**（鉛筆/方筆圖示）- 在目前公司環境中開啟新問題模式

### 3.2 個人部分

沒有部分標題 - 這些部分始終位於頂部，公司標題下方。

```
  Inbox                    3
  My Issues
```

- **收件匣** — 需要董事會操作員注意的項目。徽章計數在右側。包括：待批准、預算警報、失敗的偵測訊號。該數字是未讀/未解決的總數。
- **我的問題** — 由董事會操作員創建或指派給董事會操作員的問題。

### 3.3 工作部分

節標題：**工作**（可折疊，帶有 V 形開關）

```
  Work                     ▼
    Issues
    Projects
    Goals
    Views
```

- **問題** — 所選公司的主要任務清單。這是主力視圖。
- **項目** — 項目清單。項目將問題分組並與目標連結。
- **目標** — 公司目標層次結構。
- **檢視** — 已儲存的篩選/排序配置（例如，「關鍵錯誤」、「未指派的任務」、「CEO 的任務」）。使用者可以在此處建立、命名和固定自訂視圖。

### 3.4 公司部分

節標題：**公司**（可折疊）

```
  Company                  ▼
    Dashboard
    Org Chart
    Agents
    Costs
    Activity
```

- **控制台** — 公司健康狀況概覽：智能體狀態、任務速度、成本消耗、待審核計數。
- **組織架構圖** — 智能體報告層次結構的互動式樹形視覺化。
- **智能體** — 所有智能體的簡單列表，包括狀態、角色、最後一次心跳、支出。
- **成本** — 成本控制台，依智能體、專案、模型、時間細分。
- **活動** — 所有系統事件的審核日誌。

注意：批准沒有頂級側邊欄條目。它們透過**收件匣**（主要互動點）、**控制台**（待處理計數指標）和**實體頁面上的內聯**（例如，智能體詳細資訊頁面顯示授權其僱用的批准）來顯示。 `/approvals` 路由仍然存在，並且可以透過收件匣和控制台中的「查看所有核准」連結訪問，但它不在側邊欄導覽中。

### 3.5 部分行為- 每個部分標題均可按一下以折疊/展開其子項。
- 折疊狀態保留在 localStorage 中。
- 活動導航項目以左邊框強調和背景色調突出顯示。
- 將滑鼠懸停在導航項目上會顯示微妙的背景突出顯示。
- 徽章計數右對齊，呈現為小藥丸（例如，圓角矩形中的 `3`）。
- 圖示為 16 像素，左對齊，與標籤文字有 8 像素間隙。

### 3.6 側邊欄圖標

每個導航項目都有一個獨特的圖示（lucide-react）：

|項目 |圖示|
|------|------|
|收件匣 | `Inbox` |
|我的問題 | `CircleUser` |
|問題 | `CircleDot` |
|專案 | `Hexagon` |
|目標| `Target` |
|意見 | `LayoutList` |
|控制台| `LayoutDashboard` |
|組織架構圖| `GitBranch` |
|智能體| `Bot` |
|成本| `DollarSign` |
|活動 | `History` |

## 4. 麵包屑導覽列

麵包屑欄位於主要內容和屬性面板上方。它既充當導航又充當上下文指示器。

### 4.1 結構

```
┌─────────────────────────────────────────────────────────────────────┐
│ Projects › Paperclip › Issues › CLIP-42  [⭐] [···]     [🔔] [⬜] │
└─────────────────────────────────────────────────────────────────────┘
```

**左側：**
- 麵包屑片段，以 `›` V 形分隔。
- 每個部分都可以單擊以導航到該級別。
- 目前片段是不可點擊的、稍微粗體的文字。
- 星形圖示可收藏/固定目前實體。
- 用於實體操作的三點選單（刪除、存檔、複製、複製連結等）

**右側：**
- 通知鈴聲（如果在詳細視圖中 - 訂閱該實體的變更）
- 面板切換（顯示/隱藏右側屬性面板）

### 4.2 特定於視圖的選項卡

在某些詳細資訊頁面上，麵包屑欄還包含麵包屑下方的選項卡行：

**項目詳情：**
```
  Overview    Updates    Issues    Settings
```

**智能體詳細資料：**
```
  Overview    Heartbeats    Issues    Costs
```

選項卡呈現為藥丸狀按鈕。活動標籤具有微妙的背景填充。

## 5. 問題（任務管理）

問題是核心工作單元。本節詳細介紹了完整的問題體驗。

### 5.1 問題清單視圖

當您按一下側邊欄中的“問題”時，問題清單是預設視圖。

**佈局：**
```
┌─────────────────────────────────────────────────────────────────┐
│ [All Issues] [Active] [Backlog]  [⚙ Settings]    [≡ Filter]  [Display ▼] │
├─────────────────────────────────────────────────────────────────┤
│ ▼ Todo                                                3    [+] │
│ ☐ --- CLIP-5  ○ Implement user auth          @CTO    Feb 16  │
│ ☐ --- CLIP-3  ○ Set up CI pipeline           @DevOps Feb 16  │
│ ☐ --- CLIP-8  ○ Write API documentation      @Writer Feb 17  │
│                                                                 │
│ ▼ In Progress                                         2    [+] │
│ ☐ !!! CLIP-1  ◐ Build landing page           @FE     Feb 15  │
│ ☐ --- CLIP-4  ◐ Database schema design       @CTO    Feb 14  │
│                                                                 │
│ ▼ Backlog                                             5    [+] │
│ ☐ --- CLIP-9  ◌ Research competitors                  Feb 17  │
│ ...                                                             │
└─────────────────────────────────────────────────────────────────┘
```

**頂部工具列：**
- **狀態標籤：** `All Issues`、`Active`（待辦事項 + 進行中 + 審核中 + 已封鎖）、`Backlog`。每個標籤顯示一個狀態圖示和計數。活動選項卡已填充，其他選項卡已概述。
- **設定齒輪：** 配置問題顯示預設值、自訂欄位。
- **過濾器按鈕：** 開啟工具列下方的過濾器列。
- **顯示下拉式選單：** 在分組模式（按狀態、按優先順序、按受讓人、按項目、無）和版面模式（清單、看板/看板）之間切換。

**分組：**
- 預設情況下，問題按狀態分組（與參考螢幕截圖相符）。
- 每個群組標題顯示：折疊 V 形、狀態圖示、狀態名稱、計數以及用於在該狀態下建立新問題的 `+` 按鈕。
- 組是可折疊的。折疊的群組僅顯示帶有計數的標題。**問題行：**
每行從左到右包含：
1. **複選框** — 用於批次選擇。預設隱藏，懸停時顯示（優先順序左側）。
2. **優先權指示器** — 代表嚴重/高/中/低的圖示（請參閱上方的顏色系統）。始終可見。
3. **問題密鑰** — 例如，`CLIP-5`。等寬、柔和的色彩。前綴源自於專案（如果沒有專案則源自公司）。
4. **狀態圈** — 可點選以開啟狀態變更下拉清單（與參考螢幕截圖相同）。圓圈的填滿/顏色反映目前狀態。
5. **標題** — 主要文本，如果太長則用省略號截斷。
6. **受讓人** — 頭像（智能體圖示）+ 智能體名稱，右對齊。如果未分配，則顯示虛線圓圈佔位符。
7. **日期** — 建立日期或目標日期，靜音文本，最右側。

**行交互：**
- 按一下行 → 導覽至問題詳細資料檢視。
- 點選狀態圓圈 → 開啟內嵌狀態下拉選單（待辦事項、待辦事項、進行中、審核中、完成、已取消），並使用鍵盤數字作為快速鍵 (1-6)。
- 點選複選框 → 選擇批次操作。選取任何複選框後，清單底部會出現一個批次操作欄。
- 懸停 → 顯示複選框，行會獲得微妙的背景突出顯示。
- 右鍵單擊 → 上下文選單（與三點選單相同的操作）。

**批次操作列：**
當選擇一個或多個問題時，底部會出現一個浮動欄：
```
┌─────────────────────────────────────────────────────────┐
│  3 selected    [Status ▼] [Priority ▼] [Assignee ▼] [Project ▼]  [🗑 Delete]  [✕ Cancel] │
└─────────────────────────────────────────────────────────┘
```

### 5.2 問題篩選欄

點擊“過濾器”會在工具列下方顯示一個過濾器列：

```
┌─────────────────────────────────────────────────────────┐
│ [+ Add filter]  Status is Todo, In Progress  [×]        │
│                 Priority is Critical, High    [×]        │
│                 Assignee is CTO-Agent         [×]        │
└─────────────────────────────────────────────────────────┘
```

- 每個過濾器都是一個顯示 `field operator value` 的晶片。
- 單擊一個晶片即可編輯。
- `×` 刪除過濾器。
- `+ Add filter` 開啟可用欄位的下拉清單：狀態、優先權、受讓人、項目、目標、建立日期、標籤、建立者。
- 過濾器由 AND 組成。
- 活動過濾器保留在 URL 查詢字串中，因此可以共用/可加書籤。

### 5.3 問題詳細資料檢視（三窗格）

點擊問題將開啟詳細資訊視圖。主要內容區域分為兩個區域，側邊欄在左側仍可見。

```

│          │                                │   Priority  High     │
│          │ ┌──────────────────────────┐   │   Assignee  CTO      │
│          │ │ Properties bar (inline)  │   │   Project   Auth     │
│          │ │ In Progress · High ·     │   │   Goal      Security │
│          │ │ CTO · Auth project · ... │   │   Labels    bug, auth│
│          │ └──────────────────────────┘   │   Start     Feb 15   │
│          │                                │   Target    Feb 20   │

│          │ ─────────────────              │                      │
```

#### 中間窗格（主要內容）

**標題區域：**
- 問題標題，大（18px，半粗體），點選可編輯（內嵌編輯）。
- 字幕：以靜音文字發出金鑰 `CLIP-42`。
- 標題下方：內嵌屬性列將關鍵屬性顯示為可點擊的晶片（與參考螢幕截圖相同的模式）：`[○ In Progress] [!!! High] [👤 CTO] [📅 Target date] [📁 Auth] [···]`。每個晶片都可以單擊來更改內聯屬性。

**描述：**
- Markdown 渲染的描述。
- 點選編輯 — 就地開啟 Markdown 編輯器。
- 支援標題、清單、程式碼區塊、連結、圖像。

**子任務（如果有）：**
- 下面列出了作為可折疊部分的描述。
- 每個子任務都是一個小問題行（狀態圈+標題+受讓人）。
- `+ Add subtask` 按鈕位於底部。**評論：**
- 按時間順序排列的評論清單。
- 每則評論顯示：作者頭像/圖示、作者姓名、時間戳記、正文（降價呈現）。
- 底部的評論輸入 — 一個支援 Markdown 的文字區域和一個「評論」按鈕。
- 智能體的評論顯示機器人圖示；董事會的評論顯示一個使用者圖示。

#### 右窗格（屬性面板）

**標題：**「屬性」標籤，帶有 `+` 按鈕，用於新增自訂欄位。

**屬性清單：** 每個屬性都是一行，左邊有標籤，右邊有可編輯值。

|物業 |控制|
|----------|---------|
|狀態 |有狀態選項+彩色點的下拉式選單|
|優先|帶有優先選項+圖示的下拉選單|
|受讓人|有搜尋功能的智能體選擇器下拉選單 |
|項目|項目選擇器下拉式選單 |
|目標|目標選擇器下拉式選單 |
|標籤|多重選擇標籤輸入 |
|鉛 |智能體選擇器 |
|會員 |多選智能體選擇器 |
|開始日期 |日期選擇器 |
|目標日期 |日期選擇器 |
|創建者 |唯讀文字 |
|建立 |唯讀時間戳 |
|帳單代碼 |文字輸入 |

下面的屬性，一個分隔符，然後：

## 11. 批准（根據上下文，而不是獨立的）

批准是治理之門－董事會必須做出的決定（聘請智能體、批准執行長策略）。它們不是工作項目。他們的資料模型與問題（不同的狀態機、副作用觸發器、非結構化有效負載）保持分離。但他們不需要自己的頂級導航條目。

### 11.1 批准的體現

**1.收件匣（主要）。 ** 待批准是最高優先順序的收件匣項目。董事會操作員可以透過內聯批准/拒絕操作看到它們的前面和中間（請參閱第 14 節）。

**2.控制台指標卡。 **「待批准」卡片顯示計數以及完整批准清單的連結。

**3.內嵌在實體頁面上。 ** 透過核准建立實體時，詳細資料頁面會顯示上下文橫幅：
- 智能體詳細資料頁面：`"Hired via approval — requested by CEO on Feb 15"`，附有審核記錄連結。
- 處於 `pending` 狀態（尚未建立）的智能體程式可能會顯示：`"Pending approval — requested by CEO"`，並內聯核准/拒絕操作。**4.活動日誌。 ** 批准事件（建立、批准、拒絕）與任何其他事件一樣顯示在活動時間軸中。

- **[查看詳細資訊 →]** 複雜批准 (approve_ceo_strategy) 的鏈接，需要在做出決定之前進行全面審查。
- 類別標題中的「查看所有核准 →」連結導覽至 `/approvals`。

**警報。 ** 智能體錯誤（心跳失敗、錯誤狀態）和預算警報（智能體或公司接近 80% 或 100% 限制）。每個連結到相關智能體或成本頁面。**過時的工作。 ** `in_progress` 或 `todo` 中沒有活動（沒有評論，沒有狀態變更）超出可設定閾值（預設值：24 小時）的任務。每個都顯示問題鍵、標題以及自上次活動以來的時間。點擊導航到問題。

### 14.3 收件匣行為

- 未讀項目的左側有一個實心藍點指示器。
- 點擊一個項目將其標記為已讀。
- 一旦批准/拒絕，批准將從收件匣中消失（它們轉為已解決狀態）。
- 當根本情況得到解決（智能體恢復、預算增加）時，警報就會消失。
- 側邊欄徽章計數反映了未解決的收件匣項目總數。
- 收件匣是根據即時資料（待批准查詢+警報條件）計算的，而不是單獨的通知表。這使得 V1 變得簡單。

## 15. 搜尋（Cmd+K 模態）

透過 `Cmd+K` 或側邊欄搜尋圖示可存取全域搜尋。

```
┌─────────────────────────────────────────────────────────┐
│ 🔍 Search issues, agents, projects...                   │
├─────────────────────────────────────────────────────────┤
│ Recent                                                  │
│   📋 CLIP-42 Fix user authentication bug                │
│   🤖 CEO                                                │
│   📁 Auth project                                       │
├─────────────────────────────────────────────────────────┤
│ Actions                                                 │
│   ✏️  Create new issue                         C        │
│   🤖 Create new agent                                   │
│   📁 Create new project                                 │
└─────────────────────────────────────────────────────────┘
```

- 跨所有實體類型（問題、智能體、專案、目標）的預先輸入搜尋。
- 結果按類型分組並帶有圖示。
- 輸入為空時顯示最近的項目。
- 底部的快速操作部分。
- 箭頭鍵進行導航，Enter 進行選擇，Esc 進行關閉。

## 17. 回應行為

- **>1400px：** 完整的三窗格佈局（側邊欄+主+屬性）。
- **1024-1400px：** 側邊欄折疊為圖示。透過切換可用的屬性面板。
- **<1024px：** 側邊欄隱藏（漢堡選單）。屬性面板隱藏（切換或選項卡）。

屬性面板始終是可關閉的——它永遠不應該阻擋主要內容。

## 18. 空狀態

每個清單視圖都應該有一個深思熟慮的空狀態：

- **沒有問題：**「還沒有問題。建立您的第一期以開始追蹤工作。」帶有 `[Create issue]` 按鈕。
- **無智能體：** 「本公司沒有智能體。建立一個智能體來開始組建您的團隊。」帶有 `[Create agent]` 按鈕。
- **未選擇公司：**「選擇一家公司開始。」使用公司切換器或 `[Create company]` 按鈕。

空狀態應該使用柔和的插圖（簡單的線條藝術，而不是卡通）和單一的號召性用語。

---

## 19. 載入與錯誤狀態- **載入：** 骨架佔位符與預期內容的佈局相符（不是旋轉器）。骷髏塊帶有微妙的微光。
- **錯誤：** 帶有重試按鈕的內聯錯誤訊息。除非應用程式本身損壞，否則永遠不會出現整頁錯誤。
- **衝突 (409)：** Toast 通知：「此問題已由其他使用者更新。刷新即可查看變更。」透過[刷新]操作。
- **樂觀更新：** 狀態變更和屬性編輯應立即在 UI 中更新，並在失敗時回滾。
