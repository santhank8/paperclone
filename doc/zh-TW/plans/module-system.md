# Paperclip 模組系統

## 概述

Paperclip 的模組系統可讓您透過新功能（收入追蹤、可觀察性、通知、控制台）擴展控制平面，而無需分叉核心。模組是獨立的包，用於註冊路由、UI 頁面、資料庫表和生命週期掛鉤。

另外，**公司範本**是無程式碼的資料包（智能體團隊、組織架構圖、目標層次結構），您可以匯入它們來引導新公司。

兩者都可以透過 **公司商店** 找到。

---

## 概念

|概念|它是什麼 |有程式碼嗎？ |
|---------|-----------|----------------|
| **模組** |擴充 Paperclip 的 API、UI 與資料模型的套件 |
| **公司範本** |資料快照 — 智能體、專案、目標、組織架構 |否（僅限 JSON）|
| **公司專賣店** |用於瀏覽/安裝模組和模板的註冊表| — |
| **掛鉤** |模組可以訂閱的核心中的命名事件 | — |
| **插槽** |一個專有類別，其中只有一個模組可以處於活動狀態（例如，`observability`）| — |

## 模組架構

### 檔案結構

```
modules/
  observability/
    paperclip.module.json     # manifest (required)
    src/
      index.ts                # entry point — exports register function
      routes.ts               # Express router
      hooks.ts                # hook handlers
      schema.ts               # Drizzle table definitions
      migrations/             # SQL migrations (generated by drizzle-kit)
      ui/                     # React components (lazy-loaded by the shell)
        index.ts              # exports page/widget definitions
        TokenDashboard.tsx
```

模組位於頂級 `modules/` 目錄中。每個模組都是一個 pnpm 工作區包。

### 清單 (`paperclip.module.json`)

```json
{
  "id": "observability",
  "name": "Observability",
  "description": "Token tracking, cost metrics, and agent performance instrumentation",
  "version": "0.1.0",
  "author": "paperclip",

  "slot": "observability",

  "hooks": [
    "agent:heartbeat",
    "agent:created",
    "issue:status_changed",
    "budget:threshold_crossed"
  ],

  "routes": {
    "prefix": "/observability",
    "entry": "./src/routes.ts"
  },

  "ui": {
    "pages": [
      {
        "path": "/observability",
        "label": "Observability",
        "entry": "./src/ui/index.ts"
      }
    ],
    "widgets": [
      {
        "id": "token-burn-rate",
        "label": "Token Burn Rate",
        "placement": "dashboard",
        "entry": "./src/ui/index.ts"
      }
    ]
  },

  "schema": "./src/schema.ts",

  "configSchema": {
    "type": "object",
    "properties": {
      "retentionDays": { "type": "number", "default": 30 },
      "enablePrometheus": { "type": "boolean", "default": false },
      "prometheusPort": { "type": "number", "default": 9090 }
    }
  },

  "requires": {
    "core": ">=0.1.0"
  }
}
```

關鍵領域：

- **`id`**：唯一標識符，用作npm包名後綴（`@paperclipai/mod-observability`）
- **`slot`**：可選的專屬類別。如果設置，則只有具有此插槽的一個模組可以處於活動狀態。對於可以自由共存的模組則省略。
- **`hooks`**：此模組訂閱了哪些核心事件。預先聲明，以便核心知道要發出什麼。
- **`routes.prefix`**：安裝在 `/api/modules/<prefix>` 下。此模組擁有該命名空間。
- **`ui.pages`**：將項目新增至側邊欄。延遲載入 React 元件。
- **`ui.widgets`**：將元件注入現有頁面（例如儀表闆卡）。
- **`schema`**：Drizzle 模組擁有的表格的表格定義。以 `mod_<id>_` 為前綴以避免衝突。
- **`configSchema`**：JSON 模組配置架構。在模組加載之前進行驗證。

### 入口點

模組的`src/index.ts`導出一個`register`函數，該函數接收模組API：

```typescript
import type { ModuleAPI } from "@paperclipai/core";
import { createRouter } from "./routes.js";
import { onHeartbeat, onBudgetThreshold } from "./hooks.js";

export default function register(api: ModuleAPI) {
  // Register route handler
  api.registerRoutes(createRouter(api.db, api.config));

  // Subscribe to hooks
  api.on("agent:heartbeat", onHeartbeat);
  api.on("budget:threshold_crossed", onBudgetThreshold);

  // Register a background service (optional)
  api.registerService({
    name: "metrics-aggregator",
    interval: 60_000, // run every 60s
    async run(ctx) {
      await aggregateMetrics(ctx.db);
    },
  });
}
```

### 模組 API 表面

```typescript
interface ModuleAPI {
  // Identity
  moduleId: string;
  config: Record<string, unknown>;  // validated against configSchema

  // Database
  db: Db;                           // shared Drizzle client

  // Routes
  registerRoutes(router: Router): void;

  // Hooks
  on(event: HookEvent, handler: HookHandler): void;

  // Background services
  registerService(service: ServiceDef): void;

  // Logging (scoped to module)
  logger: Logger;

  // Access core services (read-only helpers)
  core: {
    agents: AgentService;
    issues: IssueService;
    projects: ProjectService;
    goals: GoalService;
    activity: ActivityService;
  };
}
```

模組獲得範圍記錄器、共享資料庫的存取權限以及核心服務的讀取存取權限。他們註冊自己的路線和掛鉤處理程序。它們不會對核心進行猴子補丁——它們透過定義的介面進行擴展。

## 掛鉤系統

### 核心掛鉤點

Hooks 是主要的整合點。核心在明確定義的時刻發出事件。模組在其 `register` 函數中進行訂閱。|鉤|有效負載|當 |
|------|---------|------|
| `server:started` | `{ port }` | Express 伺服器開始監聽之後 |
| `agent:created` | `{ agent }` |插入新智能體程式後 |
| `agent:updated` | `{ agent, changes }` |智能體記錄修改後 |
| `agent:deleted` | `{ agent }` |移除智能體程式後 |
| `agent:heartbeat` | `{ agentId, timestamp, meta }` |當智能體簽入。 `meta` 攜帶 token_used、成本、延遲等 |
| `agent:status_changed` | `{ agent, from, to }` |當智能體狀態轉換時（空閒→活動、活動→錯誤等）|
| `issue:created` | `{ issue }` |插入新問題後 |
| `issue:status_changed` | `{ issue, from, to }` |當問題在狀態之間移動時 |
| `issue:assigned` | `{ issue, agent }` |當問題指派給智能體程式時 |
| `goal:created` | `{ goal }` |插入新目標後 |
| `goal:completed` | `{ goal }` |當目標狀態變成完成 |
| `budget:spend_recorded` | `{ agentId, amount, total }` |支出增加後 |
| `budget:threshold_crossed` | `{ agentId, budget, spent, percent }` |當智能體超過預算的 80%、90% 或 100% 時 |

### 鉤子執行模型

```typescript
// In the core — hook emitter
class HookBus {
  private handlers = new Map<string, HookHandler[]>();

  register(event: string, handler: HookHandler) {
    const list = this.handlers.get(event) ?? [];
    list.push(handler);
    this.handlers.set(event, list);
  }

  async emit(event: string, payload: unknown) {
    const handlers = this.handlers.get(event) ?? [];
    // Run all handlers concurrently. Failures are logged, never block core.
    await Promise.allSettled(
      handlers.map(h => h(payload))
    );
  }
}
```

設計規則：
- **鉤子是一勞永逸的。 **失敗的鉤子處理程序永遠不會崩潰或阻止核心操作。
- **掛鉤是併發的。 ** 事件的所有處理程序透過 `Promise.allSettled` 並行運作。
- **鉤子是提交後的。 **它們在資料庫寫入成功之後觸發，而不是之前。沒有否決權。
- **鉤子接收不可變的快照。 **處理程序取得資料的副本，而不是可變的引用。

這使核心保持快速和有彈性。如果您需要預先提交驗證（例如，「拒絕此預算變更」），這是一種不同的機制（中間件/攔截器），我們可以在需要時添加。

### 可觀察性掛鉤範例

```typescript
// modules/observability/src/hooks.ts
import type { Db } from "@paperclipai/db";
import { tokenMetrics } from "./schema.js";

export function createHeartbeatHandler(db: Db) {
  return async (payload: {
    agentId: string;
    timestamp: Date;
    meta: { tokensUsed?: number; costCents?: number; model?: string };
  }) => {
    const { agentId, timestamp, meta } = payload;

    if (meta.tokensUsed != null) {
      await db.insert(tokenMetrics).values({
        agentId,
        tokensUsed: meta.tokensUsed,
        costCents: meta.costCents ?? 0,
        model: meta.model ?? "unknown",
        recordedAt: timestamp,
      });
    }
  };
}
```

每次心跳，可觀察性模組都會將代幣使用情況記錄到自己的 `mod_observability_token_metrics` 表中。核心不知道也不關心這個表——它只是發出鉤子。

## 模組的資料庫策略

### 表格命名空間

模組表以 `mod_<moduleId>_` 為前綴，以避免與核心表和其他模組發生衝突：

```typescript
// modules/observability/src/schema.ts
import { pgTable, uuid, integer, text, timestamp } from "drizzle-orm/pg-core";

export const tokenMetrics = pgTable("mod_observability_token_metrics", {
  id: uuid("id").primaryKey().defaultRandom(),
  agentId: uuid("agent_id").notNull(),
  tokensUsed: integer("tokens_used").notNull(),
  costCents: integer("cost_cents").notNull().default(0),
  model: text("model").notNull(),
  recordedAt: timestamp("recorded_at", { withTimezone: true }).notNull().defaultNow(),
});

export const alertRules = pgTable("mod_observability_alert_rules", {
  id: uuid("id").primaryKey().defaultRandom(),
  agentId: uuid("agent_id"),
  metricName: text("metric_name").notNull(),
  threshold: integer("threshold").notNull(),
  enabled: boolean("enabled").notNull().default(true),
  createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
});
```

### 遷移策略

每個模組在 `src/migrations/` 中管理自己的遷移。核心遷移運行程式發現並應用它們：

1. 核心遷移首先運行（總是）
2. 模組遷移依依賴順序運行
3. 每個模組的遷移都在 `mod_migrations` 表中透過模組 ID 進行追蹤
4. `pnpm db:migrate` 運行一切。 `pnpm db:migrate --module observability` 運行一個。

模組可以透過外鍵（例如，`agent_id → agents.id`）引用核心表，但核心表永遠不會引用模組表。這是一種嚴格的單向依賴。

## 模組載入和生命週期

### 發現

伺服器啟動時：

```
1. Scan modules/ directory for paperclip.module.json manifests
2. Validate each manifest (JSON Schema check on configSchema, required fields)
3. Check slot conflicts (error if two active modules claim the same slot)
4. Topological sort by dependencies (if module A requires module B)
5. For each module in order:
   a. Validate module config against configSchema
   b. Run pending migrations
   c. Import entry point and call register(api)
   d. Mount routes at /api/modules/<prefix>
   e. Start background services
6. Emit server:started hook
```

### 配置

模組配置位於伺服器環境或設定檔：

```jsonc
// paperclip.config.json (or env vars)
{
  "modules": {
    "enabled": ["observability", "revenue", "notifications"],
    "config": {
      "observability": {
        "retentionDays": 90,
        "enablePrometheus": true
      },
      "revenue": {
        "stripeSecretKey": "$STRIPE_SECRET_KEY"
      }
    }
  }
}
````$ENV_VAR` 引用在載入時解析。秘密永遠不會直接進入設定檔。

### 停用模組

將模組的啟用狀態設定為 false：
1. 停止後台服務
2. 卸載其路由（返回404）
3. 取消訂閱其鉤子處理程序
4. 不刪除其資料庫表（資料被保留）

```
1. Parse and validate the template JSON
2. Check for ref uniqueness and dangling references
3. Insert agents (topological sort by reportsTo)

5. Insert projects
6. Insert issues (resolve projectRef, assigneeRef, goalRef to real IDs)
7. Log activity events for everything created
```

## 實施計劃

### 第一階段：核心基礎設施

加入`@paperclipai/server`：

1. **HookBus** — 使用 `register()` 和 `emit()` 的事件發射器，使用 `Promise.allSettled`
2. **模組載入器** — 掃描 `modules/`，驗證清單，呼叫 `register(api)`
3. **模組API物件** — `registerRoutes()`、`on()`、`registerService()`、記錄器、核心服務訪問
4. **模組配置** — `paperclip.config.json` 具有每個模組的配置、環境變數內插
5. **模組遷移運行程式** — 擴展 `db:migrate` 以發現和運行模組遷移
6. **從核心服務發出掛鉤** - 將 `hookBus.emit()` 呼叫新增至現有的 CRUD 操作

加入`@paperclipai/ui`：

7. **模組頁面載入器** - 讀取模組清單，產生惰性路由
8. **控制台小工具插槽** - 在控制台頁面上渲染模組提供的小工具
9. **側邊欄擴充** — 動態新增模組導覽項

新增包：

10. **`@paperclipai/module-sdk`** — `ModuleAPI`、`HookEvent`、`HookHandler` 的 TypeScript 類型，清單模式

### 第 2 階段：第一個模組（可觀察性）

11. 建構 `modules/observability` 作為參考實現

## 設計原則

1. **模組擴展，從不修補。 ** 模組新增新的路由、表格和掛鉤處理程序。他們從不修改核心表或覆蓋核心路由。

2. **掛鉤是提交後、即發即忘的。 ** 模組故障永遠不會中斷核心操作。

3. **單向依賴。 ** 模組依賴核心。核心從不依賴模組。模組表可以 FK 到核心表，反之則不然。

4. **聲明式清單，命令式註冊。 ** JSON 中的靜態元資料（無需執行程式碼即可驗證）。運行時行為透過 API 註冊。

5. **命名空間隔離。 ** 模組路由位於 `/api/modules/<id>/` 下。模組表的前綴為 `mod_<id>_`。模組配置的範圍僅限於其 ID。

6. **優雅降級。 ** 如果模組載入失敗，記錄錯誤並繼續。系統的其餘部分工作正常。7. **資料在停用後仍然存在。 ** 停用模組會停止其程式碼，但保留其資料。重新啟用將從中斷處繼續。
