# 任務管理資料模型

有關 Paperclip 中任務追蹤如何運作的參考。描述實體及其
關係以及管理任務生命週期的規則。編寫為目標模型
——其中一些已經實施，一些是期望的。

---

## 實體層次結構

```
Workspace
  Initiatives          (roadmap-level objectives, span quarters)
    Projects           (time-bound deliverables, can span teams)
      Milestones       (stages within a project)
        Issues         (units of work, the core entity)
          Sub-issues   (broken-down work under a parent issue)
```

一切都流了下來。倡議包含項目；一個項目包含
里程碑和問題；一個問題可以有子問題。每個等級都會增加
粒度。

## 工作流程狀態問題狀態**不是**一個平面枚舉。這是一組特定於團隊的命名狀態，
每個都屬於這些固定**類別**之一：

|類別 |目的|範例狀態 |
| ------------- | ---------------------------- | ------------------------------- |
| **分類** |傳入，需審核 |分流 |
| **積壓** |已接受，尚未準備好工作 |積壓，冰箱|
| **未啟動** |準備好但尚未開始 |待辦事項，準備就緒 |
| **開始** |積極工作 |進行中、審查中、品質檢查中 |
| **已完成** |完成 |完成，已發貨 |
| **取消** |被拒絕或放棄|已取消、無法修復、重複 |

### 規則

- 每個團隊在這些類別中定義自己的工作流程狀態
- 團隊每個類別必須至少有一個州（分類是可選的）
- 可在任何類別中新增自訂狀態（例如「已開始」下的「審核中」）
- 類別是固定且有序的－您可以在類別內對狀態重新排序
  但不是類別本身
- 新問題預設為團隊的第一個 Backlog 狀態
- 將問題移至已啟動狀態自動設定 `startedAt`；已完成的套裝
  `completedAt`;取消套 `cancelledAt`
- 將問題標記為重複會自動將其移至「已取消」狀態

### 工作流程狀態字段

|領域 |類型 |筆記|
| ------------- | ------- | ----------------------------------------------------------------------------- |
| `id` | uuid| |
| `name` |字串|顯示名稱，例如「回顧中」 |
| `type` |枚舉 |其中之一：`triage`、`backlog`、`unstarted`、ZXQQQ00034QQXZ、`completed`、ZQQX
| `color` |字串|十六進位顏色 |
| `description` |字串|可選指導文本 |
| `position` |浮動|類內訂購 |
| `teamId` | uuid FK | UUID每個州屬於一支球隊|

## 團隊

團隊是主要的組織單位。幾乎所有的事情都限定在一個範圍內
團隊。

|領域 |類型 |筆記|
| ------------- | ------ | -------------------------------------------------------------- |
| `id` | uuid| |
| `name` |字串|例如「工程」|
| `key` |字串|短大寫前綴，例如「英文」。用於問題標識符。 |
| `description` |字串| |

### 團隊範圍界定

- 每個問題都屬於一個團隊
- 工作流程狀態是每個團隊的
- 標籤可以是團隊範圍或工作空間範圍的
- 專案可以跨越多個團隊

在我們的背景下（人工智慧公司），團隊映射到職能領域。各智能體報告
基於角色的團隊。

| `startDate` |日期 | |
| `targetDate` |日期 | |

### 規則

- 一個問題最多屬於一個項目
- 專案狀態**手動**更新（不是從問題狀態自動得出）
- 項目可以包含文件（規格、簡介）作為連結實體

## 標籤/標籤

標籤提供分類標記。它們存在於兩個範圍：

- **工作區標籤** -- 所有團隊均可使用
- **團隊標籤** -- 僅限於特定團隊

|領域 |類型 |筆記|
| ------------- | -------------- | ------------------------------- |
| `id` | uuid| |
| `name` |字串| |
| `color` |字串|十六進位顏色 |
| `description` |字串|情境指引 |
| `teamId` | uuid FK | UUID工作區等級標籤為 Null |
| `groupId` | uuid FK（自身）|用於分組的父標籤 |

### 標籤組

標籤可以組織為一層嵌套（群組 -> 標籤）：

- 一組內的標籤在某個問題上**互斥**（只能有一個）
  各組申請）
- 群組不能包含其他群組（僅限單一嵌套層級）
- 範例：群組「Type」包含標籤「Bug」、「Feature」、「Chore」——一個問題
  最多獲得一個

### 問題標籤連接

透過 `issue_labels` 聯接表進行多對多：

|領域 |類型 |
| --------- | ------- |
| `issueId` | uuid FK | UUID
| `labelId` | uuid FK | UUID

## 問題關係/依賴關係

問題之間的四種關係類型：

|類型 |意義|行為 |
| ------------ | -------------------------------- | --------------------------------------------- |
| `related` |一般連結 |資訊連結|
| `blocks` |這個問題阻止了另一個問題 |帶有標誌的已阻止問題 |
| `blocked_by` |此問題已被另一個問題阻止 |區塊的逆 |
| `duplicate` |這個問題與另一個問題重複|自動將重複項移至「已取消」狀態 |

### IssueRelation 字段|領域 |類型 |筆記|
| ---------------- | ------- | ---------------------------------------------- |
| `id` | uuid| |
| `type` |枚舉 | `related`、`blocks`、`blocked_by`、`duplicate` |
| `issueId` | uuid FK | UUID來源問題 |
| `relatedIssueId` | uuid FK | UUID目標問題|

### 規則

- 當阻塞問題解決後，關係就變成資訊性的（標誌
  變綠）
- 重複是單向的（您標記重複，而不是規範）
- 阻塞在系統層級**不可傳遞**（A 阻塞 B，B 阻塞 C
  不會自動阻止 A->C)

## 受讓人

**單受讓人模型**設計而成。

- 每題一次最多有一名受讓人
- 這是經過深思熟慮的：明確的所有權可以防止責任分散
- 對於涉及多人的協作工作，請使用**子問題**
  不同的受讓人

在我們的上下文中，智能體是受讓人。關於`assigneeId` FK的問題
指向 `agents` 表。

---

## 子問題（父/子）

問題支援父/子嵌套。

- 在問題上設定 `parentId` 使其成為子問題
- 子問題本身可以有子問題（多層嵌套）
- 子問題在創建時從其父問題繼承**項目**
  時間（不追溯），但不包括團隊、標籤或受讓人

### 自動關閉

- **子問題自動關閉**：當父親問題完成時，剩餘的子問題
  自動完成

### 轉換

- 可以重新調整現有問題的父級（新增或刪除 `parentId`）
- 具有許多子問題的父問題可以「升級」到項目

## 標識符

問題使用人類可讀的識別碼：`{TEAM_KEY}-{NUMBER}`

- 團隊金鑰：每個團隊設定的短大寫字串（例如“ENG”、“DES”）
- Number：每隊自動遞增的整數
- 例：`ENG-123`、`DES-45`、`OPS-7`
- 如果問題在團隊之間轉移，它會獲得一個新的標識符，而舊的標識符將被保留
  保存在`previousIdentifiers`

這比 UUID 對於人類交流來說要好得多。人們說“抓住 ENG-42”
不是“抓住 7f3a...”。

## 實作優先權

建議的建置順序，最高值優先：

### 高價值

1. **隊伍** -- `teams` 桌 + `teamId` FK 問題。基金會
   人類可讀的識別碼 (`ENG-123`) 和每個團隊的工作流程狀態。大多數
   其他功能取決於團隊範圍，因此首先建立此功能。
2. **工作流程狀態** -- `workflow_states` 表 + `stateId` FK 問題。
   每個團隊的自訂工作流程以及基於類別的狀態轉換。
3. **標籤** -- `labels` + `issue_labels` 表。分類
   （錯誤/功能/瑣事、區域標籤等）而不污染狀態欄位。
4. **問題關係** -- `issue_relations` 表。阻塞/被阻塞是
   對於智能體協調至關重要（智能體 A 在智能體 B 完成之前無法啟動）。
5. **子問題** -- `parentId` 在 `issues` 上自 FK。讓特工崩潰
   大任務。
6. **註釋** -- `comments` 表。智能體需要就問題進行溝通
   而不覆蓋描述。

### 中價值

7. **轉換時間戳記** -- `startedAt`、`completedAt`、`cancelledAt` 上
   問題，根據工作流程狀態變化自動設定。啟用速度追蹤和 SLA
   測量。

### 較低優先權（稍後）

8. **里程碑**－一旦專案變得複雜到需要階段就很有用。
9. **倡議**—一旦我們有多個服務於共同目標的專案就很有用
   戰略目標。
10. **估計**－當我們想要測量吞吐量並預測時很有用
    容量。
