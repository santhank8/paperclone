## Story

**As a** usuário da LP2 e como membro do time técnico,
**I want** um footer completo e a página validada para SEO e performance,
**so that** a landing page esteja pronta para deploy com qualidade profissional e não canibalize a LP1 nos mecanismos de busca.

## Acceptance Criteria

1. `<footer>` implementado com: links de navegação, copyright, links legais (Privacidade, Termos)
2. SEO técnico completo: `<title>` único, `<meta description>` de 150-160 chars, canonical URL, Open Graph completo
3. Lighthouse score >= 90 em Performance, Accessibility e SEO
4. HTML válido: sem erros críticos no W3C Validator
5. Nenhum console error no browser no carregamento inicial
6. Todas as imagens com `alt` descritivo (não vazio, não "image")
7. Robots meta: `<meta name="robots" content="index, follow">` (não indexar de forma a competir com LP1 enquanto não definida URL final)
8. `<link rel="canonical">` apontando para URL definitiva da LP2 (placeholder se não definida)
9. Acessibilidade: todos links de navegação com foco visível, landmarks HTML5 corretos (`<main>`, `<nav>`, `<footer>`)

## Dev Notes

### Fonte: `docs/lp2/project-brief.md`

**SEO diferenciado** (do project brief): "Meta tags diferenciadas para não canibalizar LP1" — LP1 é `landing/index.html`. LP2 deve ter um angle de keyword diferente.

**Sugestão de meta description da LP2:**
```
"Paperclip × AIOX: seu time de engenharia de IA. Entregue 3x mais features sem aumentar headcount. Agentes especializados que trabalham como um time real. Comece grátis."
```

**URL da LP2** ainda é uma Open Question do project brief. Possibilidades: `/v2`, `/lp2`, subdomínio, novo domínio. Usar `[LP2-URL]` como placeholder até definição.

**Robots:** enquanto a URL definitiva não estiver definida, pode ser adequado usar `noindex` para evitar indexação prematura. Discutir com @pm.

**Performance targets do project brief:** LCP < 2.5s, CLS < 0.1, TBT < 200ms. O Lighthouse deve confirmar conformidade.

### Testing

- Rodar `npx lighthouse [url] --view` localmente para checar scores
- Validar HTML em validator.w3.org
- Abrir no browser e checar console (F12) por erros
- Verificar navegação via teclado (Tab através de todos os links)
- Testar em modo de alto contraste do OS para verificar acessibilidade
