# Phase 2 Design: Release Changelog & PR Report Integration

**Document Type**: Architecture & Design  
**Version**: 1.0  
**Created**: 2026-03-31  
**Status**: DRAFT (Ready for Implementation)  
**Purpose**: Define the complete Phase 2 architecture for automating release notes and PR reports from sprint artifacts

---

## Executive Summary

**Phase 2 extends Sprint Co from delivery automation into release artifact generation.** After the Delivery Engineer ships code to production (Phase 1), Phase 2 agents automatically generate:

1. **Release Changelog Entry** — Markdown section for `CHANGELOG.md` with feature summaries, issue links, and contributors
2. **PR Report** — Rich GitHub PR comment with feature matrix, QA scores, issue traceability, and sprint recommendations

This eliminates manual changelog writing (5-10 min per sprint) and provides traceability from sprint planning all the way to released features, closing the feedback loop to Paperclip's task coordination.

**Impact**: Automates release documentation, provides sprint→issue mapping for auditing, and signals when work is genuinely shipped.

## 1. Overview

### What Phase 2 Delivers

#### 1.1 Release Changelog Generation

After successful deployment, `sprint-release-generator` reads sprint artifacts and produces:

```
Input artifacts:
  - sprint-report.md (from Delivery Engineer)
  - eval-*.md files (from QA Engineer)
  - handoff-*.md files (from each engineer)
  - task-breakdown.md (from Product Planner)
  
Processing:
  - Extract shipped features from eval reports (PASS status)
  - Link to Paperclip issue IDs embedded in artifacts
  - Query GitHub for commit messages + PR links
  - Summarize dropped features (QA FAIL or v2)
  - Collect contributor names from git history
  
Output:
  - CHANGELOG.md entry (calver-versioned section)
  - Structured JSON for external systems
```

### Why It Matters

| Problem | Phase 1 Impact | Phase 2 Solution |
|---------|---------------|-----------------|
| **Manual release notes** | 5–10 min per sprint writing prose | Auto-generated from structured artifacts |
| **Lost issue traceability** | PRs shipped but no link back to original Paperclip issues | Embed Paperclip issue ID → GitHub PR → CHANGELOG |
| **QA data buried** | Eval scores exist but not visible in releases | QA matrix in PR comment + CHANGELOG |
| **Dropped features unclear** | V2/dropped features not documented | Explicit "Features Dropped" section with reason + Paperclip ID |
| **No sprint closure signal** | Deployment happens but Paperclip issue not marked shipped | Auto-update Paperclip: mark issues as released, set version metadata |
| **Contributor attribution lost** | Engineers ship code but not mentioned in release notes | Auto-extract GitHub usernames, populate Contributors section |

### Success Criteria

**Phase 2 is successful when:**

1. ✅ Release Changelog generated in <2 minutes post-deployment
2. ✅ CHANGELOG.md entry matches human-written quality (no verbose bloat)
3. ✅ Every PASS feature has Paperclip issue ID + GitHub PR link
4. ✅ PR comment includes QA matrix + feature status table
5. ✅ Issue ID mapping has ≥95% accuracy (automated + 1 manual review pass)
6. ✅ Dropped features clearly documented with reason
7. ✅ Paperclip issues auto-marked as shipped with version metadata
8. ✅ End-to-end flow tested with 3 real sprints before production release

## 3. Skill Changes Required

### 3.1 Recommendation: Create New Skill `sprint-release-generator`

**NOT extending sprint-delivery** because:

| Aspect | Extend delivery? | Create new skill? |
|--------|------------------|-------------------|
| **Scope creep** | sprint-delivery already covers: pre-deploy checklist, deployment, smoke tests, git tagging, sprint report. Adding release generation makes it 3 separate concerns | One concern per skill: release generation is its own job |
| **Sequencing** | Delivery publishes code, then release-gen runs. Separate phases. | Cleaner separation of concerns |
| **Reusability** | Release-gen could run for non-sprint contexts (hotfixes, emergency patches) | Sprint-delivery is sprint-specific; release-gen is general |
| **Testing** | Easier to test release-gen independently of deployment | 10+ test cases can run without Cloudflare/smoke tests |
| **Error isolation** | If release-gen fails, deployment already succeeded (good). If integrated, one failure blocks both. | Deployment isolated from release notes |

### 3.2 New Skill: `sprint-release-generator`

**Location**: `/Volumes/JS-DEV/paperclip/skills/sprint-release-generator/SKILL.md`

**Responsibility**:
1. Parse sprint artifacts (eval, handoff, task-breakdown, sprint-report)
2. Enrich with git history + GitHub PR data
3. Generate CHANGELOG.md entry
4. Post GitHub PR comment with release report
5. Update Paperclip issues (mark as shipped, set version)

**Invocation**: After Delivery Engineer signals successful deployment

### 4.5 Version Numbering Scheme

Use **Paperclip's Calendar Versioning (CalVer)**:

```
Stable release: vYYYY.DDD.P
  YYYY = 4-digit year (2026)
  DDD = 3-digit day-of-year (001–366)
  P = patch slot (0, 1, 2, ...)

Examples:
  v2026.090.0 = Jan 1, 2026, patch 0
  v2026.090.2 = Jan 1, 2026, patch 2 (third release that day)
  v2026.325.0 = Nov 21, 2026, patch 0
```

**Derive version from**:
- Deployment date (from `sprint-report.md` timestamp)
- Count existing releases on that date
- Increment patch slot: `version = f"{year}.{day:03d}.{next_patch}"`

**Python helper**:
```python
from datetime import datetime

def get_version_number(deploy_date: str) -> str:
    dt = datetime.fromisoformat(deploy_date)
    day_of_year = dt.timetuple().tm_yday
    
    # Check how many releases exist for this date
    import glob
    releases = glob.glob(f"releases/v{dt.year}.{day_of_year:03d}.*.md")
    next_patch = len(releases)
    
    return f"v{dt.year}.{day_of_year:03d}.{next_patch}"
```

### 5.3 Signal Timing: When Does Release-Generator Wake?

#### Option A: After Delivery Engineer Posts Success Comment
```
Delivery Engineer @ T+2:50
  |
  git push + git tag
  |
  POST comment: "@sprint-release-generator Ready to generate release"
  |
  [Release-Generator starts immediately]
  |
  Parse artifacts, generate CHANGELOG, update Paperclip
  @ T+2:55 (release artifacts ready)
```

**Pros**: Explicit signal, clear trigger, no polling  
**Cons**: Requires Delivery Engineer to signal

#### Option B: Automatic on Paperclip Sprint Status Change
```
Delivery Engineer
  |
  PATCH sprint issue: status = "done"
  |
  Paperclip event: issue.status.changed → "done"
  |
  Webhook → Sprint Release Generator
  |
  Generate release artifacts
  @ T+2:55
```

**Pros**: Fully automatic, no manual signaling  
**Cons**: Requires Paperclip webhooks configured

#### Recommendation: Option A (explicit signal) for Phase 2.1

- Safer (release-gen only runs when explicitly triggered)
- Easier to test (mock the signal)
- Easier to debug (clear intent in comment)

**Implement as fallback**: If Paperclip webhooks exist, use them. Otherwise, rely on Delivery Engineer to signal.

### 7.3 Version Numbering Scheme

Use Paperclip's CalVer:

```
vYYYY.DDD.P

where:
  YYYY = year (2026)
  DDD = day of year (001–366, zero-padded)
  P = patch number on that day (0, 1, 2, ...)

Examples:
  v2026.001.0 = Jan 1, 2026, first release
  v2026.001.1 = Jan 1, 2026, second release
  v2026.325.0 = Nov 21, 2026, first release
```

**Derivation logic**:
```
Deploy date: 2026-03-31
Day of year: 90 (march 31 = day 90)
Existing releases on 2026-03-31: 2 (v2026.090.0, v2026.090.1)
Next patch number: 2
Version: v2026.090.2
```

## 9. Risks & Mitigations

### Risk 1: Changelog Drift from Actual Releases

**Problem**: Release-generator produces CHANGELOG.md, but someone might manually edit it, causing drift between CHANGELOG and actual shipped code.

**Mitigation**:
- Make CHANGELOG.md generation atomic with Paperclip issue updates
- Add header comment to generated entries:
  ```markdown
  <!-- GENERATED BY SPRINT CO RELEASE GENERATOR -->
  <!-- DO NOT EDIT MANUALLY — see sprint artifacts for source of truth -->
  
  # v2026.090.2
  ...
  ```
- Implement validation script: `scripts/validate-changelog.ts`
  - Verifies each CHANGELOG entry has corresponding Paperclip issue
  - Checks version matches git tag
  - Flags manual edits with warnings (non-blocking)
  - Runs in CI/pre-push hook

### Risk 2: GitHub PR Comment Truncation for Large Releases

**Problem**: GitHub comment limit is 65,536 characters. Large releases might exceed this.

**Mitigation**:
- Implement splitting logic:
  1. Generate full comment
  2. If > 60KB, split into multiple comments:
     - Comment 1: Header + feature table
     - Comment 2: QA results + timeline
     - Comment 3: Notes + recommendations
  3. In comment 1, add reply references: "See replies below for full results."
- Test with worst-case: 10 shipped features + full QA matrix + detailed notes
- Fallback: If GitHub API fails, save comment to disk (`release-comment.md`) for manual posting

### Risk 3: Issue ID Mapping Ambiguity

**Problem**: Multiple sources of issue IDs (Paperclip PAP-XXXX vs GitHub #XXXX), could cause confusion or mismatches.

**Mitigation**:
- Define precedence order:
  1. **Primary**: Paperclip ID from task-breakdown.md (authoritative)
  2. **Secondary**: GitHub PR from git history / GitHub API
  3. **Tertiary**: Manual override in sprint-report.md
- Always display both: `[PAP-1234](https://paperclip.ai/issues/PAP-1234)` with `[#2847](https://github.com/...)`
- Add validation:
  ```typescript
  if (paperclipId && gitHubPR) {
    // Both exist: cross-reference in output
  } else if (paperclipId) {
    // Only Paperclip: note that GitHub PR link not found
  } else if (gitHubPR) {
    // Only GitHub: note that Paperclip ID not found (error!)
  }
  ```
- Require at least one ID per feature (fail if neither found)

### Risk 4: Missing Eval Reports

**Problem**: QA Engineer might forget to save eval report, release-generator wouldn't know feature failed.

**Mitigation**:
- Validation logic in release-generator:
  ```typescript
  for each feature in task-breakdown:
    if no eval-[TASK-ID].md exists:
      status = "MISSING_EVAL"
      error("Feature {TASK_ID} has no eval report")
  ```
- Fail loudly if any feature missing eval (don't ship incomplete data)
- Delivery Engineer must fix (ensure QA ran evaluation) before releasing
- This reinforces Phase 1 discipline (delivery engineer can't deploy without QA)

### Risk 5: Contributor Attribution Errors

**Problem**: Email → GitHub username mapping might be inaccurate (noreply emails, mismatches).

**Mitigation**:
- Implement multi-step lookup:
  1. Extract email from commit author
  2. If noreply: `author@users.noreply.github.com` → `author`
  3. Query GitHub API: `GET /users/{username}` (verify user exists)
  4. If lookup fails: Use email as fallback (don't break release)
- Manual review step: Include contributor list in release comment for human approval
- Document any attribution misses in release notes

### Risk 6: API Rate Limiting (GitHub, Paperclip)

**Problem**: Querying GitHub for PRs + Paperclip for issue updates might hit rate limits.

**Mitigation**:
- Implement exponential backoff:
  ```typescript
  async function callWithRetry(fn, maxRetries = 3) {
    for (let i = 0; i < maxRetries; i++) {
      try {
        return await fn();
      } catch (err) {
        if (err.statusCode === 429) {  // Rate limit
          const delay = Math.pow(2, i) * 1000;
          await sleep(delay);
        } else {
          throw err;
        }
      }
    }
  }
  ```
- Batch API calls where possible (list issues once, not per-feature)
- Cache results (save API responses for 1 hour)
- Graceful degradation: If API unavailable, generate changelog with partial data (note what's missing)

### Risk 7: Paperclip Integration Downtime

**Problem**: Paperclip API might be down when release-generator runs.

**Mitigation**:
- Make Paperclip updates optional (not blocking):
  ```
  Generate CHANGELOG.md (required)
  Post GitHub comment (required)
  Update Paperclip (optional, retry on next signal)
  ```
- Store pending updates in `.sprint-release-pending.json`:
  ```json
  {
    "sprintTaskId": "issue-12345",
    "pendingUpdates": [
      { "issueId": "...", "status": "done", "metadata": {...} }
    ]
  }
  ```
- Retry job: If Paperclip updates fail, queue a 5-min retry (up to 3 times)
- Alert: Log clear warning if Paperclip sync fails (ops can investigate)

### Risk 8: Incorrect Version Derivation

**Problem**: Multiple releases on same day; version number derivation could be off-by-one.

**Mitigation**:
- Implement reliable version derivation:
  ```typescript
  function deriveVersion(deployDate: Date): string {
    const year = deployDate.getFullYear();
    const dayOfYear = getDayOfYear(deployDate);
    
    // Check git tags for versions on this date
    const existingTags = execSync(
      `git tag -l 'v${year}.${dayOfYear:03d}.*' --sort=-version:refname`
    ).toString().trim().split('\n').filter(Boolean);
    
    // Existing tags are ["v2026.090.1", "v2026.090.0"]
    // Next patch is 2
    const nextPatch = existingTags.length;
    
    return `v${year}.${dayOfYear:03d}.${nextPatch}`;
  }
  ```
- Test with real git repo before first release
- Allow manual override: `FORCE_VERSION=v2026.090.2`
