# Skill Brief: Build a Memory System That Survives Context Compaction\n\n## Demand Signal\n\n- ClawHub "ontology" (structured agent memory) has 107k downloads — #2 skill on the platform\n- Combined memory-related skills (ontology + recall/persistence variants) exceed 200k downloads\n- GitHub #34556: User with 140+ hours of autonomous operation across 3,500+ sessions wrote a detailed breakdown of their custom 3-tier memory architecture — highest-signal issue on the topic\n- GitHub #34661: "Agent --resume ignores MEMORY.md: complete context loss on session resume" — 2 comments, direct evidence of the core pain\n- GitHub #34610: "/recap command for session context" — users asking for built-in cross-session state recovery; top response is a custom SessionStart + Stop hooks implementation\n- GitHub #34629: "Prompt cache regression in --print --resume" — 20x cost increase; users relying on session resume as a memory mechanism and getting burned\n- GitHub #34600: "Stop hook with exit code 2 used to block shutdown for pre-exit memory save" — users already building memory-save hooks on their own, confirming demand for the pattern\n- YouTube gap: no tutorial teaches file-based MEMORY.md + hook-based capture as a system — existing content is either "how to set CLAUDE.md" (one-off) or "install the ontology skill" (dependency)\n\n## Target Audience\n\nDevelopers who've built or are building an autonomous Claude Code workflow and keep losing state:\n\n- Their agent learns something in session 1 and forgets it in session 2\n- They have a CLAUDE.md but it's growing uncontrolled with stale notes\n- They've heard of MEMORY.md but don't know how to structure it or what to put in it\n- They restart a  session and watch the model lose the thread entirely\n- They're manually copying learnings from one session to start the next\n\nThey want persistence without a database, a framework, or a paid API. They want the agent to remember the way a good senior dev remembers — not verbatim, but the decisions, the gotchas, the patterns that saved them last time.\n\n## Core Thesis\n\nClaude Code has no built-in long-term memory — but you can build one from hooks and files that survives compactions, resumes, and fresh sessions. The pattern is: write structured memory files, index them in MEMORY.md, load them on SessionStart, and save new learnings in a Stop hook. Zero external dependencies.\n\n## Skill Scope\n\n### In Scope\n\n- Understanding why context compaction erases memory and what to do about it\n- The four memory types (user, feedback, project, reference) and when to use each\n- Writing memory files with frontmatter (name, description, type)\n- Maintaining MEMORY.md as an index (not a memory store)\n- SessionStart hook that loads MEMORY.md into context\n- Stop hook that triggers a memory-save review before exit\n- PostToolUse hook that captures key decisions/gotchas at the moment they happen\n- Checkpoint discipline: when to save vs. when to skip\n- Anti-patterns: what NOT to put in memory (code patterns derivable from source, git history)\n- Worked example: a session that captures a gotcha and avoids repeating it next session\n\n### Out of Scope\n\n- Recall MCP / vector databases (mentioned briefly as an optional upgrade, not taught)\n- Full autonomous agent setup (covered in skill 001)\n- CLAUDE.md behavioral rules (covered in skill 001)\n- Subagent memory passing (advanced — separate skill)\n- Cloud sync or multi-device memory\n\n## Sections\n\n1. **Why Your Agent Keeps Forgetting** — How context compaction works, what  actually does vs. what users think it does, why CLAUDE.md alone isn't enough. The mental model: in-context memory vs. file-based memory — one survives the session, the other doesn't.\n\n2. **The Memory File Format** — Frontmatter spec (name, description, type). The four memory types: user (who you're working with), feedback (corrections and preferences), project (decisions, deadlines, rationale), reference (where to find things). One fact per file, named by topic not date.\n\n3. **MEMORY.md: The Index** — What goes in it (file links + one-line descriptions), what doesn't (the memory content itself). Why it's an index: MEMORY.md is always in context, so it must stay small. The 200-line limit. Organizing by topic, not chronologically.\n\n4. **The SessionStart Hook** — How to write a hook that reads MEMORY.md and surfaces it as a system reminder at session open. Why timing matters: load memory before the first user message, not after. Working hook implementation.\n\n5. **The Stop Hook: Memory Before You Go** — Writing a Stop hook that prompts a memory review before exit. When to auto-save vs. prompt. The "memory or handoff" decision: short learnings go to memory files, longer context goes to a handoff.md for the next session.\n\n6. **Continuous Capture: The PostToolUse Pattern** — Capturing decisions and gotchas at the moment they happen (not at end-of-session when they've been forgotten). What qualifies for capture: decisions, gotchas, milestones, corrections, trade-offs. Threshold heuristic: "Would I be annoyed if I had to re-derive this next session?"\n\n7. **Anti-Patterns and the Decay Problem** — What NOT to save: code derivable from source, git history, ephemeral state, in-progress work. How to update stale memories before they become wrong. Deduplication: check MEMORY.md before writing a new file.\n\n8. **Complete Walkthrough: A Session That Remembers** — End-to-end: session opens (SessionStart hook loads index), work happens (PostToolUse captures a decision), session ends (Stop hook saves the gotcha), next session opens (index loads, prior learning visible). The full loop with working code.\n\n## Success Criteria\n\nAfter installing this skill, a developer should be able to:\n\n- [ ] Explain the difference between in-context memory and file-based persistent memory\n- [ ] Create a MEMORY.md index and at least two typed memory files\n- [ ] Write a SessionStart hook that loads memory context at session open\n- [ ] Write a Stop hook that triggers a memory review before exit\n- [ ] Know the four memory types and correctly categorize a new learning\n- [ ] Identify three things that should NOT go in memory (code, git history, ephemeral state)\n- [ ] Successfully recover a decision from a prior session via their memory index in a fresh session\n\n## Keywords\n\nclaude code memory, persistent memory, context compaction, CLAUDE.md memory, session memory, memory.md, claude code hooks, SessionStart hook, Stop hook, agent memory, cross-session memory, file-based memory, claude code learning, context survival\n\n## Competitive Positioning\n\n| Their Approach | Our Approach |\n|---------------|-------------|\n| Install "ontology" from ClawHub (107k downloads) | Build it yourself — same pattern, zero dependency |\n| Manual copy-paste from session notes | PostToolUse hook captures at the moment decisions happen |\n| CLAUDE.md grows unbounded with stale context | Typed memory files with index — structured and prunable |\n| --resume as memory (gets burned by cache regressions) | File-based memory survives compaction AND --resume failures |\n| Recall MCP / vector search (paid, setup required) | Works offline, in any project, with any model |\n\n## Estimated Complexity\n\nLow-Medium.
