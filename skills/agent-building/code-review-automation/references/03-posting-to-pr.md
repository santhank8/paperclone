# Posting Review Results to GitHub PRs

## Method 1: Simple PR Comment (Recommended Starting Point)

```bash
# Save review output to a file first
cat > /tmp/review-output.md << 'EOF'
## Code Review — claude-sonnet-4-6

| File:Line | Severity | Finding | Suggestion |
|-----------|----------|---------|------------|
| auth.ts:42 | CRITICAL | SQL injection via unsanitized `userId` | Use parameterized query: `db.query("SELECT * FROM users WHERE id = ?", [userId])` |
| api.ts:118 | HIGH | N+1 query: `getUser()` called inside loop | Batch with `getUsersByIds(ids)` before the loop |

**Summary:** 2 findings — 1 critical, 1 high

*Generated by Claude Code code-review-automation skill*
EOF

# Post it
gh pr comment 47 --body "$(cat /tmp/review-output.md)"
```

## Method 2: Edit Last Comment (For Iteration)

When you're running a second review on the same PR after fixes:

```bash
gh pr comment 47 --edit-last --body "$(cat /tmp/review-output.md)"
```

This updates the existing review comment rather than creating a new one — keeps the PR thread clean.

## Method 3: GitHub Review API (Line-Level Annotations)

For line-specific comments that show inline in the GitHub diff view:

```bash
# First, get the commit SHA
COMMIT=$(gh pr view 47 --json headRefOid -q .headRefOid)

# Create a review with line-level comments
gh api repos/{owner}/{repo}/pulls/47/reviews \
  -f commit_id="$COMMIT" \
  -f event="COMMENT" \
  -f body="Code review findings" \
  -f "comments[][path]=src/auth.ts" \
  -f "comments[][position]=42" \
  -f "comments[][body]=**CRITICAL:** SQL injection via unsanitized input. Use: \`db.query('SELECT * FROM users WHERE id = ?', [userId])\`"
```

Line-level comments are higher friction to set up but show directly in the diff — better for CRITICAL and HIGH findings.

## Comment Template

Use this as your standard review comment format:

```markdown
## 🔍 Code Review — [timestamp]

**Reviewers:** Security · Performance · Correctness · Style (parallel)
**Diff:** [PR #N / branch main...HEAD]
**Findings:** N total (X critical, Y high, Z medium, W low)

---

| File:Line | Severity | Reviewer | Finding | Suggestion |
|-----------|----------|----------|---------|------------|
| auth.ts:42 | 🔴 CRITICAL | Security | SQL injection... | Use parameterized query... |
| api.ts:118 | 🟠 HIGH | Performance | N+1 query... | Batch with getUsersByIds()... |
| utils.ts:67 | 🟡 MEDIUM | Correctness | Missing null check... | Add `if (!user) return null`... |

---

**Stack-specific checks:** ✅ Next.js/Convex checklist applied

*Generated by [code-review-automation](https://github.com/your-org/skills) · Claude Code*
```

## Severity Emoji Reference

| Severity | Emoji | Meaning |
|---|---|---|
| CRITICAL | 🔴 | Blocks merge — security hole, data loss, broken auth |
| HIGH | 🟠 | Fix before merge |
| MEDIUM | 🟡 | Fix before merge if possible |
| LOW | 🔵 | Consider fixing |

## When to Use Each Posting Method

| Situation | Method |
|---|---|
| First review on a PR | Simple comment (Method 1) |
| Re-reviewing after fixes | Edit last comment (Method 2) |
| CRITICAL finding that needs visibility in diff | Line-level annotation (Method 3) |
| Automated CI posting | Simple comment (most reliable) |

## Get the Right Repo for `gh api`

```bash
# Auto-detect from git remote
REMOTE=$(git remote get-url origin)
OWNER=$(echo "$REMOTE" | sed 's|.*github.com[:/]\([^/]*\)/.*|\1|')
REPO=$(echo "$REMOTE" | sed 's|.*github.com[:/][^/]*/\([^.]*\).*|\1|')

echo "Posting to: $OWNER/$REPO"
```
