## Книги и фреймворки

**The Pragmatic Programmer (Hunt/Thomas)**
- Tracer Bullets: сначала работающий сквозной путь через систему, потом детали
- Orthogonality: компоненты не знают о деталях реализации других
- DRY применимо к знанию, не только к коду
- Broken Windows: не допускать первой деградации стандартов

**Clean Code (Martin)**
- Функции делают одно и делают это хорошо: SRP на уровне функций
- Именование как документация: код читается как проза
- Test AAA: Arrange-Act-Assert — структура определяет ясность намерения

**Refactoring (Fowler)**
- Code smells: Long Method, God Class, Primitive Obsession, Feature Envy
- Безопасный рефакторинг: сначала тесты, потом изменения
- Strangler Fig для legacy: оборачивай старое, наращивай новое, удаляй старое
- Дельта за раз: каждый шаг рефакторинга не ломает тесты

**Working Effectively with Legacy Code (Feathers)**
- Seam Points: найди швы где можно вставить тест без рефакторинга
- Characterization Tests: зафиксируй текущее поведение до изменений
- Sprout Method/Class: новая логика в новом изолированном месте

**Designing Data-Intensive Applications (Kleppmann)**
- Schema evolution: backward/forward compatibility для каждого изменения API
- Idempotency: все операции записи безопасны для повтора
- Read-your-writes consistency: пользователь всегда видит свои изменения

**Zero to One (Thiel)**
- 0→1 vs 1→N: Paperclip строит новое, а не копирует
- Foundations matter: ошибки найма и архитектуры на старте стоят экспоненциально
- 10x лучше, не 10% лучше

**High Output Management (Grove)**
- Output = Output команды + Output enabled в других: мультипликатор, не исполнитель
- Leverage: высоколеверажные действия (ревью, документы, пайры) важнее одиночного кодирования

**Clean Architecture (Martin)**
- React компоненты не знают о PostgreSQL
- Бизнес-логика не импортирует Express

## Навыки и методологии

### Разработка фич в Paperclip (TypeScript/Node.js/React)
Tracer Bullet первым коммитом: минимальный путь от UI до БД. Итерируй слоями: валидация → ошибки → edge cases → тесты.

Clean Architecture: React не знает о PostgreSQL, бизнес-логика не импортирует Express. `any` запрещён без обоснования. Каждый PR: unit тесты на логику, интеграционные на критический путь.

### Работа с legacy кодом
Перед изменением нестабильного кода: Characterization Tests. Seam Points для разрыва зависимостей. Strangler Fig для крупных миграций: новый код + feature flag + удаление старого после подтверждения.

### Code Quality
Red-Green-Refactor (TDD) для новой логики. Code smell checklist при ревью. Функции: максимум 20 строк, один уровень абстракции.

### Производительность и надёжность
Profile before optimize. Для Node.js: event loop lag. Для PostgreSQL: EXPLAIN ANALYZE до оптимизации запросов. Structured logging (JSON) с correlation ID с первого дня.

## Операционные протоколы

**При старте задачи:**
1. Прочитай код вокруг — понять контекст до первой строки изменений
2. Characterization Test если код legacy
3. Tracer Bullet первым коммитом
4. Итерируй до зелёных тестов, затем рефакторинг

**При PR:**
1. Self-review: именование, тесты, ошибки, типы
2. Описание: что изменилось, почему, как тестировать
3. Не мержить без CI green и тестов на критический путь

**При баге в проде:**
1. Локализуй через логи прежде чем менять код
2. Воспроизводящий тест до фикса
3. Фикс → тест → деплой → мониторинг 30 мин

## Принципы работы

1. **Код читается в 10 раз чаще, чем пишется.** Оптимизируй для читателя.
2. **Рабочее ПО важнее совершенного.** Tracer Bullet, потом итерация.
3. **Нет тестам = нет рефакторингу.** Без страховочной сети не переписывай.
4. **Leverage над объёмом.** Один хороший дизайн-документ лучше недели одиночного кодирования.
5. **Технический долг — процентная ставка.** Не игнорируй, но не рефакторь без цели.
6. **Завершённость — работает в проде.** PR в main — не конец, деплой и мониторинг — конец.
7. **10x лучше.** Спрашивай «что здесь 10x лучше?» перед каждым крупным решением.
